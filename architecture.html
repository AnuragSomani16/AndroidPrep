Architecture — MVVM / MVI / Clean / MVC

Full examples in both Compose and XML. When to choose each. Repository pattern, multi-module, and the dependency rule.

unidirectional data flow single source of truth dependency inversion repository pattern state holders side effect channel reducer pattern pure domain layer multi-module enforced boundaries API vs implementation dependencies
MVVM = Model-View-ViewModel. ViewModel exposes UI state via observable streams; View observes; events flow back via method calls. MVI = Model-View-Intent. Strict unidirectional flow: View sends Intent → ViewModel reduces State → View renders new State. Side effects on a separate channel. Clean Architecture = a layering pattern (presentation / domain / data) that's orthogonal to MVVM/MVI — you compose them. MVC = the original GUI pattern; rarely a good fit on Android because Activity ends up playing both Controller and View.

1. The TL;DR you should be able to recite

Senior framing

MVC — Controller mediates Model and View; legacy on Android; rarely used today.
MVVM — ViewModel exposes state via StateFlow/LiveData; View observes; events flow back via methods. The default for most Android apps.
MVI — Strict unidirectional data flow: Intent → Reducer → State → View. Side effects on a separate channel. Best for state-heavy screens.
Clean Architecture — A layering concern (data / domain / presentation), orthogonal to MVVM/MVI. You compose them: MVVM-Clean or MVI-Clean.

2. MVC — for the trivia question

The original GUI pattern. Model = data; View = UI; Controller = handles input, updates model, picks views. On Android, the Activity often plays both Controller and View — leading to "fat activities." This is why MVVM emerged.

Don't recommend MVC for new Android apps

If asked "would you use MVC?" — say "I wouldn't on Android. The framework's lifecycle pulls Activity/Fragment toward Controller, and they end up owning view state too. MVVM/MVI separates these cleanly via the lifecycle-aware ViewModel."

ViewModel (shared between both UIs)

@HiltViewModel
class NotesVm @Inject constructor(
    private val getNotes: GetNotesUseCase
) : ViewModel() {

    private val _notes = MutableStateFlow<List<Note>>(emptyList())
    val notes: StateFlow<List<Note>> = _notes.asStateFlow()

    private val _loading = MutableStateFlow(false)
    val loading: StateFlow<Boolean> = _loading.asStateFlow()

    fun load() = viewModelScope.launch {
        _loading.value = true
        runCatching { getNotes() }
            .onSuccess { _notes.value = it }
            .onFailure { /* surface */ }
        _loading.value = false
    }
    fun delete(id: String) = viewModelScope.launch { /* ... */ }
}

Compose UI (MVVM)

@Composable
fun NotesScreen(vm: NotesVm = hiltViewModel()) {
    val notes by vm.notes.collectAsStateWithLifecycle()
    val loading by vm.loading.collectAsStateWithLifecycle()

    LaunchedEffect(Unit) { vm.load() }

    if (loading) Spinner()
    LazyColumn {
        items(notes, key = { it.id }) { note ->
            NoteRow(note, onDelete = { vm.delete(note.id) })
        }
    }
}

XML UI (MVVM)

@AndroidEntryPoint
class NotesFragment : Fragment(R.layout.frag_notes) {
    private val vm: NotesVm by viewModels()
    override fun onViewCreated(v: View, s: Bundle?) {
        val binding = FragNotesBinding.bind(v)
        val adapter = NotesAdapter { vm.delete(it.id) }
        binding.list.adapter = adapter
        viewLifecycleOwner.lifecycleScope.launch {
            repeatOnLifecycle(Lifecycle.State.STARTED) {
                launch { vm.notes.collect { adapter.submitList(it) } }
                launch { vm.loading.collect { binding.progress.isVisible = it } }
            }
        }
        vm.load()
    }
}

MVVM tradeoffs

  • Pros: low ceremony, easy to start with, native fit for StateFlow.
  • Cons: state spread across multiple flows; "what's the screen showing right now?" can be hard to answer; testing every state combination is awkward.

4. MVI — full example

The pattern

┌─────┐  Intent  ┌────────────────┐  State   ┌─────┐
│ UI  │ ────────▶│ ViewModel      │ ────────▶│ UI  │
└─────┘          │ - reducer       │          └─────┘
                 │ - effects ─────▶─── one-shot ──▶ snackbar/nav
                 └────────────────┘

State, Intent, Effect

// Single source of truth for the screen
data class NotesState(
    val isLoading: Boolean = false,
    val notes: List<Note> = emptyList(),
    val error: String? = null
)

// User actions
sealed interface NotesIntent {
    data object Load : NotesIntent
    data object Refresh : NotesIntent
    data class Delete(val id: String) : NotesIntent
    data class TogglePin(val id: String) : NotesIntent
}

// One-shot side effects
sealed interface NotesEffect {
    data class ShowSnackbar(val msg: String) : NotesEffect
    data class NavigateToDetail(val id: String) : NotesEffect
}

ViewModel

@HiltViewModel
class NotesVm @Inject constructor(
    private val getNotes: GetNotesUseCase,
    private val deleteNote: DeleteNoteUseCase
) : ViewModel() {

    private val _state = MutableStateFlow(NotesState())
    val state: StateFlow<NotesState> = _state.asStateFlow()

    private val _effects = MutableSharedFlow<NotesEffect>(replay = 0, extraBufferCapacity = 1)
    val effects: SharedFlow<NotesEffect> = _effects.asSharedFlow()

    fun onIntent(intent: NotesIntent) {
        when (intent) {
            NotesIntent.Load, NotesIntent.Refresh -> load()
            is NotesIntent.Delete -> delete(intent.id)
            is NotesIntent.TogglePin -> togglePin(intent.id)
        }
    }

    private fun load() = viewModelScope.launch {
        _state.update { it.copy(isLoading = true, error = null) }
        runCatching { getNotes() }
            .onSuccess { notes -> _state.update { it.copy(isLoading = false, notes = notes) } }
            .onFailure { e ->
                _state.update { it.copy(isLoading = false, error = e.message) }
                _effects.tryEmit(NotesEffect.ShowSnackbar("Couldn't load"))
            }
    }
    private fun delete(id: String) = viewModelScope.launch {
        runCatching { deleteNote(id) }
            .onSuccess { load() }
            .onFailure { _effects.tryEmit(NotesEffect.ShowSnackbar("Delete failed")) }
    }
    private fun togglePin(id: String) { /* ... */ }
}
Three pieces work together:
  1. NotesState = the entire screen's snapshot. Loading + data + error in one immutable object. The UI binds to this — never to scattered fields.
  2. NotesIntent = sealed interface of every action the user can take. The UI never mutates state directly; it dispatches an Intent.
  3. NotesEffect = one-shot side effects (snackbar, navigation). On a SharedFlow with replay=0 so a screen rotation doesn't replay yesterday's snackbars.
The onIntent function is the reducer — pure routing of Intent to state-update logic. _state.update { it.copy(...) } is atomic and respects equality dedupe. Tests can assert state-by-state by feeding intents and reading state.value.

UI consumption

Compose (MVI)

@Composable
fun NotesScreen(vm: NotesVm = hiltViewModel(), nav: NavController) {
    val state by vm.state.collectAsStateWithLifecycle()
    val snackbarHost = remember { SnackbarHostState() }

    LaunchedEffect(Unit) { vm.onIntent(NotesIntent.Load) }

    LaunchedEffect(Unit) {
        vm.effects.collect {
            when (it) {
                is NotesEffect.ShowSnackbar -> snackbarHost.showSnackbar(it.msg)
                is NotesEffect.NavigateToDetail -> nav.navigate("note/${it.id}")
            }
        }
    }

    Scaffold(snackbarHost = { SnackbarHost(snackbarHost) }) { padding ->
        when {
            state.isLoading -> Spinner()
            state.error != null -> ErrorView(state.error!!) { vm.onIntent(NotesIntent.Refresh) }
            else -> NoteList(
                notes = state.notes,
                onDelete = { vm.onIntent(NotesIntent.Delete(it)) },
                onPin = { vm.onIntent(NotesIntent.TogglePin(it)) },
                modifier = Modifier.padding(padding)
            )
        }
    }
}

XML (MVI)

@AndroidEntryPoint
class NotesFragment : Fragment(R.layout.frag_notes) {
    private val vm: NotesVm by viewModels()
    override fun onViewCreated(v: View, s: Bundle?) {
        val b = FragNotesBinding.bind(v)
        val adapter = NotesAdapter(
            onDelete = { vm.onIntent(NotesIntent.Delete(it.id)) },
            onPin    = { vm.onIntent(NotesIntent.TogglePin(it.id)) }
        )
        b.list.adapter = adapter

        viewLifecycleOwner.lifecycleScope.launch {
            repeatOnLifecycle(Lifecycle.State.STARTED) {
                launch {
                    vm.state.collect { st ->
                        b.progress.isVisible = st.isLoading
                        b.error.isVisible = st.error != null
                        b.error.text = st.error
                        adapter.submitList(st.notes)
                    }
                }
                launch {
                    vm.effects.collect { e ->
                        when (e) {
                            is NotesEffect.ShowSnackbar -> Snackbar.make(b.root, e.msg, Snackbar.LENGTH_SHORT).show()
                            is NotesEffect.NavigateToDetail -> findNavController().navigate(/* ... */)
                        }
                    }
                }
            }
        }
        vm.onIntent(NotesIntent.Load)
    }
}
Why MVI shines

"Single source of truth for screen state means I can reproduce any UI state in a preview/test by passing a NotesState. Side effects on a separate channel mean they fire once — no double-snackbars after rotation. The reducer is pure: (prev, intent) → next. Easy to unit test."

5. Clean Architecture — the dependency rule

┌──────────────────────────────────────────────┐
│ presentation (Activity/Fragment, Compose, VM)│
│   ↓ depends on                               │
├──────────────────────────────────────────────┤
│ domain  (UseCases, models, repo INTERFACES)  │  ← NO Android dependency
│   ↑ implementations sit OUTSIDE              │
├──────────────────────────────────────────────┤
│ data    (Repo impls, Room, Retrofit, mappers)│
└──────────────────────────────────────────────┘

Inner layers know nothing about outer layers. Domain has no Android dependency — pure Kotlin. Data layer implements interfaces declared in domain (dependency inversion).

Module structure

:app                 (entry point, Hilt graph)
:feature:notes
    :ui              (Compose screens, ViewModels)
    :domain          (use cases, models, repository interface)
    :data            (Room, Retrofit impl, repo impl)
:core
    :network         (OkHttp, Retrofit, base auth)
    :database        (Room base)
    :design-system   (theme, common composables)
    :common          (shared utilities)

Layer code

// :feature:notes:domain — pure Kotlin, no android deps
data class Note(val id: String, val title: String, val body: String)

interface NotesRepository {
    fun observeAll(): Flow<List<Note>>
    suspend fun add(note: Note)
    suspend fun delete(id: String)
    suspend fun refresh(): Result<Unit>
}

class GetNotesUseCase @Inject constructor(
    private val repo: NotesRepository,
    @IoDispatcher private val io: CoroutineDispatcher
) {
    suspend operator fun invoke(): List<Note> = withContext(io) {
        repo.observeAll().first().filter { it.title.isNotBlank() }
    }
}

// :feature:notes:data — Android deps allowed
class NotesRepositoryImpl @Inject constructor(
    private val dao: NotesDao,
    private val api: NotesApi
) : NotesRepository {
    override fun observeAll(): Flow<List<Note>> =
        dao.observeAll().map { entities -> entities.map { it.toDomain() } }

    override suspend fun add(note: Note) = dao.upsert(note.toEntity())
    override suspend fun delete(id: String) = dao.delete(id)
    override suspend fun refresh(): Result<Unit> = runCatching {
        val remote = api.fetchNotes()
        dao.replaceAll(remote.map { it.toEntity() })
    }
}

// :feature:notes:ui — depends on :domain (NOT :data)
@HiltViewModel
class NotesVm @Inject constructor(
    private val getNotes: GetNotesUseCase,
    private val repo: NotesRepository    // injected via interface
) : ViewModel() { /* ... */ }

When to use Use Cases vs skip them

  • Use them when business logic is composed from multiple repos, reused across screens, or has non-trivial logic worth unit-testing.
  • Skip them when the VM just calls a single repo method passthrough — adding a use case is bloat. Be pragmatic.

6. Repository pattern in detail

The repository is the single source of truth for a data type. It coordinates between data sources (network, DB, in-memory cache) and returns domain models, not entities/DTOs.

// Offline-first pattern
class NotesRepositoryImpl @Inject constructor(
    private val dao: NotesDao,
    private val api: NotesApi,
    @IoDispatcher private val io: CoroutineDispatcher
) : NotesRepository {

    // UI observes the DB. DB is the source of truth.
    override fun observeAll(): Flow<List<Note>> =
        dao.observeAll().map { it.map(NoteEntity::toDomain) }

    // Refresh: network -> DB. UI auto-updates via Flow.
    override suspend fun refresh(): Result<Unit> = withContext(io) {
        runCatching {
            val remote = api.fetchNotes()
            dao.replaceAll(remote.map(NoteDto::toEntity))
        }
    }

    override suspend fun add(note: Note) {
        dao.upsert(note.toEntity().copy(status = "PENDING_CREATE"))
        // sync worker will push to network later
    }
}
Senior framing

"For offline-first features I make the DB the source of truth. The UI observes dao.observeAll() as a Flow; the repository's refresh() writes through to the DB; the UI auto-updates. This guarantees no UI / cache drift."

7. Multi-module — practical guidance

Why multi-module

  • Build speed — modules compile in parallel; touching one feature doesn't recompile the world.
  • Enforced boundaries — :ui can't accidentally call into :data; the build won't let it.
  • Explicit API surface — what's api-exposed vs implementation-private.
  • Reusability — pull a feature into another app variant by changing build.gradle.

api vs implementation

// :feature:notes:ui
dependencies {
    implementation(project(":feature:notes:domain"))   // not exposed to consumers
    api(project(":core:design-system"))                // exposed — anyone using :ui sees design-system
}

Default to implementation. api only when the consumer of your module legitimately needs the type in their public API.

Avoiding circular dependencies

  • Define interfaces in inner layers (:domain), implementations outside.
  • A :core:common module for shared types if needed.
  • Feature modules depend only on :domain + :core:design-system + :common, never on each other.

8. The decision matrix — when to use what

ScenarioArchitectureWhy
Simple CRUD list/detail screensMVVMState is straightforward; MVI ceremony unnecessary
Multi-step form, video player, searchMVIMany state interactions; benefit from single state object
Codebase > 50K LoC, multiple teams+ Clean + multi-moduleEnforce boundaries; build speed
Single screen, prototypingMVVM, monolithDon't pre-architect
"Should we use MVI?""Per-screen choice"Mix — MVVM by default, lift to MVI when state grows
Don't say "MVI is always better"

That's a junior take. Senior take: "I default to MVVM, lift to MVI when state interactions get complex enough that 'what's the screen showing right now?' becomes hard to answer. Single source of truth is the win, not the boilerplate."

9. Anti-patterns to call out (and avoid in your interview)

  • Fat ViewModel with 800 lines, business logic + UI logic + side effects in one place. Extract use cases / mappers.
  • Repository returning DTOs. Map at the boundary; UI shouldn't know about UserDto.
  • SingleLiveEvent hacks for one-shot events. Use SharedFlow(replay=0) or Channel.
  • Bare collectAsState. Use collectAsStateWithLifecycle.
  • Activity holding the data. ViewModel survives config changes; activity doesn't.
  • Singleton state via object Repo. Use DI; lifecycle-managed scope.
  • Multiple sources of truth. Cache + DB + memory all updating independently → drift bugs.
