Publishing your project as an Android library

Turning a feature module into a reusable library — when, how, and the dozen things you must take care of (API stability, ProGuard rules, manifest merging, dependency exposure, signing, versioning).

Android Library = a packaged module other apps can depend on. Output is an AAR (Android Archive) — a zip containing compiled bytecode, resources (themes, drawables, layouts), AndroidManifest fragment, and assets. Distributed via Maven Central, JitPack, internal Artifactory, or copied as a file. Replaces "copy-paste this code into every app" with a versioned, single-source artifact.
AAR vs JAR com.android.library plugin api vs implementation exposure consumer ProGuard rules manifest merging resource prefix to avoid collisions minSdk compatibility SemVer for libraries Maven Publish + signing POM + Gradle Module Metadata

1. AAR vs JAR — when each

JARAAR
FormatPlain Java archive (Kotlin/Java bytecode + META-INF)Zip with classes.jar + res/ + AndroidManifest.xml + assets/ + jni/
Use forPure Kotlin/Java logic, no Android typesAnything Android — references Context, has resources, declares manifest entries, ships native libs
Pluginjava-library or kotlin("jvm")com.android.library
Build outputbuild/libs/foo.jarbuild/outputs/aar/foo-release.aar
Senior framing

"For pure logic that doesn't touch Android, I ship a JAR — easier to consume from non-Android JVM apps too. For anything that needs Context, ships drawables, registers a BroadcastReceiver, or has its own manifest entries, I ship an AAR."

2. Convert a feature module to a library

// build.gradle.kts (the module)
plugins {
    id("com.android.library")               // not "com.android.application"
    kotlin("android")
    id("com.google.devtools.ksp")
    id("maven-publish")                      // for publishing
}

android {
    namespace = "com.example.player"        // unique package — used to generate R class
    compileSdk = 34

    defaultConfig {
        minSdk = 24                          // be conservative; consumers must support this or higher
        consumerProguardFiles("consumer-rules.pro")
        // No applicationId — libraries don't have one
    }

    buildFeatures {
        buildConfig = true                    // enable BuildConfig if needed
        compose = true                        // if you ship composables
    }

    publishing {
        singleVariant("release") {
            withSourcesJar()                  // attach sources for IDE step-through
            withJavadocJar()
        }
    }
}

dependencies {
    // api: exposed to consumers — they see this transitively
    api("androidx.media3:media3-exoplayer:1.3.0")

    // implementation: hidden from consumers — they can't accidentally depend on it
    implementation("com.squareup.okhttp3:okhttp:4.12.0")
}
  1. com.android.library instead of application tells AGP to build an .aar instead of an APK.
  2. namespace must be globally unique (reverse-DNS). It defines where R, BuildConfig, generated Hilt code goes. Two libraries with the same namespace would collide on consumer classpath.
  3. consumerProguardFiles ships a ProGuard rules file with the AAR. Consumers automatically apply your rules when they enable R8. Critical if your library uses reflection (Gson, Moshi, retrofit interfaces).
  4. api vs implementation is the most important call. api means consumers see Media3 transitively — they can use ExoPlayer types in their code without declaring it themselves. implementation hides OkHttp — consumers don't accidentally couple to your network choice. Default to implementation.
  5. No applicationId — libraries don't ship as standalone apps. Everything is namespaced under your namespace.

3. The 12 things you MUST take care of

3.1 — API surface design

Once a class is public in version 1.0, you've signed a contract. Breaking it forces every consumer to update their code on upgrade.

Mark internals correctly

Make every class, function, and property internal by default. Only promote to public when consumers genuinely need it. Use @RestrictTo(LIBRARY_GROUP) for cross-module-but-not-public types.

// Library API surface
class PlayerController(/* ... */) {                    // public — consumers can construct
    fun play() { /* ... */ }
    fun pause() { /* ... */ }
    internal fun rawPlayer(): ExoPlayer = exo          // internal — only this module sees it
}

@RestrictTo(RestrictTo.Scope.LIBRARY_GROUP)
class PlayerInternals { /* ... */ }                     // public bytecode, but only your modules can call

3.2 — Hide your transitive dependencies

If you mark a dep as api, every consumer is locked to your version of it. Upgrading the dep across consumers becomes painful. Use implementation unless the consumer is meant to interact with that library's types directly.

3.3 — Consumer ProGuard rules

# consumer-rules.pro — ships INSIDE the AAR; consumers auto-apply when minifying
# Keep your public API
-keep public class com.example.player.PlayerController { public *; }

# Keep classes referenced from XML (custom views, attrs)
-keep public class com.example.player.ui.** extends android.view.View { public (...); }

# If you ship Retrofit interfaces or @Serializable types
-keep,allowobfuscation,allowshrinking interface com.example.player.api.**
-keepclassmembers @kotlinx.serialization.Serializable class * { *; }
Consumer rules vs project rules

proguardFiles applies only when YOUR module is built (rarely matters for libraries). consumerProguardFiles applies on the consuming app when it minifies. Always use the consumer flavor for library rules.

3.4 — Manifest merging

Your library's AndroidManifest.xml gets merged into the consumer's manifest. Anything you declare (permissions, services, providers, intent filters) shows up in their app.

<!-- library AndroidManifest.xml -->
<manifest xmlns:tools="http://schemas.android.com/tools">
    <!-- Required for your library -->
    <uses-permission android:name="android.permission.INTERNET" />

    <application>
        <service android:name=".PlaybackService"
            android:foregroundServiceType="mediaPlayback"
            tools:node="merge" />
    </application>
</manifest>
Merge gotchas

(1) Don't ask for permissions consumers don't need — surprises them at install time. (2) Avoid declaring an <application> block unless required. (3) For optional providers/services, document them; let consumers opt-in via their own manifest if possible. (4) Use tools:node="remove" in consumer manifest to strip unwanted entries.

3.5 — Resource prefix to avoid collisions

android {
    resourcePrefix = "player_"              // every res/ name must start with this prefix
}

Without a prefix, colors.xml defining primary would clobber the consumer's primary on resource merge. With player_primary, no collision possible.

3.6 — minSdk and version compatibility

Your library's minSdk sets a floor for every consumer. Don't bump it casually — pick the lowest you can support, document it, only raise on a major version. Use VERSION.SDK_INT guards inside the library to handle older devices gracefully.

3.7 — Versioning (SemVer)

BumpWhen
MAJOR (1.0.0 → 2.0.0)Breaking API change — removed/renamed public method, changed signature, raised minSdk
MINOR (1.0.0 → 1.1.0)New public API, backward-compatible
PATCH (1.0.0 → 1.0.1)Bug fix, no API change
Senior phrase

"SemVer is a contract with consumers — major bumps require a migration guide. We use binary compatibility validator (Kotlin's kotlinx-binary-compatibility-validator) in CI to catch accidental breaks at PR time, not at release time."

3.8 — Compose-specific concerns

  • Public composables should accept a Modifier = Modifier param so consumers can size/style them.
  • Don't lock consumers to a Material version — let them theme from outside via MaterialTheme.
  • Annotate stable types with @Immutable / @Stable so consumers' Compose Compiler can optimize.
  • Provide previews — @Preview-annotated functions help consumers see what your composables look like in their IDE.

3.9 — Lifecycle of your singletons

Don't create static singletons that hold Application context. Either:

  1. Provide a builder/init function consumers call from their Application.onCreate.
  2. Use App Startup library — declare an Initializer in your library's manifest; consumers don't have to remember to init.
  3. If using Hilt, ship an EntryPoint interface that consumers' Hilt graph satisfies.

3.10 — Don't bundle Hilt — provide for it

If your library uses Hilt internally, you constrain consumers to Hilt. Better:

  • Use plain constructor injection inside the library (no Hilt annotations).
  • Expose factory functions or builder objects.
  • Optionally provide a separate :foo-hilt artifact with Hilt modules for consumers who use Hilt.

3.11 — Native libs (.so)

If you ship JNI / native, place .so files under src/main/jniLibs/<abi>/. Specify ABI filters so you don't pad the AAR with architectures consumers don't need.

3.12 — Test artifacts separately

Ship :foo (production AAR) and :foo-test (testing helpers — fakes, test doubles). Consumers add testImplementation on the test artifact. Keeps test code out of release APKs.

4. Publishing — the gradle config

// build.gradle.kts
publishing {
    publications {
        create<MavenPublication>("release") {
            groupId = "com.example"
            artifactId = "player-core"
            version = "1.2.0"

            afterEvaluate {
                from(components["release"])
            }

            pom {
                name.set("Example Player Core")
                description.set("Player abstraction over ExoPlayer / Bitmovin")
                url.set("https://github.com/example/android-player")
                licenses {
                    license {
                        name.set("Apache-2.0")
                        url.set("https://www.apache.org/licenses/LICENSE-2.0.txt")
                    }
                }
                developers {
                    developer { id.set("anurag"); name.set("Anurag Somani") }
                }
                scm {
                    url.set("https://github.com/example/android-player")
                    connection.set("scm:git:git://github.com/example/android-player.git")
                }
            }
        }
    }

    repositories {
        // Maven Central
        maven {
            name = "sonatype"
            url = uri("https://s01.oss.sonatype.org/service/local/staging/deploy/maven2/")
            credentials {
                username = providers.gradleProperty("sonatypeUsername").orNull
                password = providers.gradleProperty("sonatypePassword").orNull
            }
        }

        // Or internal Artifactory
        maven {
            name = "internal"
            url = uri("https://artifactory.example.com/repo")
            credentials { /* ... */ }
        }
    }
}
  1. create<MavenPublication>("release") defines the publication. groupId:artifactId:version is the coordinate consumers use to depend on you.
  2. from(components["release"]) tells Gradle which build variant to publish — usually only release, sometimes both.
  3. pom { } is the metadata Maven Central requires — name, description, license, developers, SCM. Without it, Sonatype rejects your upload.
  4. repositories { maven { ... } } declares where to publish. Maven Central + your internal Artifactory are common.

Signing (required for Maven Central)

plugins {
    id("signing")
}

signing {
    val signingKey = providers.gradleProperty("signing.key").orNull
    val signingPassword = providers.gradleProperty("signing.password").orNull
    useInMemoryPgpKeys(signingKey, signingPassword)
    sign(publishing.publications)
}

Maven Central requires every artifact be GPG-signed. Generate a key once with gpg --gen-key, upload its public part to a keyserver, configure in ~/.gradle/gradle.properties.

Publish

./gradlew :player-core:publish                  # all repos
./gradlew :player-core:publishToSonatype        # just Sonatype
./gradlew :player-core:publishReleasePublicationToMavenLocal   # for local consumer testing

5. JitPack — for quick GitHub-published libraries

If Maven Central is overkill, JitPack builds AARs from your GitHub tags on demand. Consumers add:

// settings.gradle.kts (consumer)
dependencyResolutionManagement {
    repositories {
        google()
        mavenCentral()
        maven { url = uri("https://jitpack.io") }
    }
}

// app build.gradle.kts (consumer)
implementation("com.github.example:android-player:1.2.0")

Trade-offs: zero ops; first build can be slow as JitPack compiles; GPL on free tier (private repos cost). Good for OSS / internal experiments; not great for production-grade SLA.

6. Consumer-facing checklist (run before each release)

  • API surface stable (no accidental public on internals — run kotlinx-binary-compatibility-validator).
  • Dependencies are implementation by default; only api for types in public API.
  • Consumer ProGuard rules cover all reflection-heavy paths.
  • No surprise permissions in manifest.
  • Resource prefix set; no collisions with consumers.
  • minSdk is the lowest you can support.
  • Sample app builds against the published AAR (proves it works end-to-end).
  • CHANGELOG.md updated; SemVer bump correct.
  • README has install snippet for both Gradle Kotlin DSL and Groovy.
  • Sources JAR + Javadoc JAR attached so IDE navigation works.
  • License file present in repo and POM.
  • Smoke test: published artifact can be consumed in a fresh project.

7. Common library interview probes

Q. AAR vs JAR — when which?

JAR for pure Kotlin/Java with no Android dependencies (faster build, JVM consumers possible). AAR for anything Android-aware — references Context, ships resources, declares manifest entries, has native libs.

Q. What's the difference between api and implementation dep?

api: dep is exposed transitively — consumers see it. implementation: dep is hidden — consumers don't see it. Default to implementation; only use api when types from that dep appear in your library's public API.

Q. What's consumerProguardFiles for?

ProGuard rules shipped inside the AAR. When the consumer enables R8, your rules auto-apply. Critical if your library uses reflection (Gson types, Retrofit interfaces, kotlinx.serialization classes).

Q. How do you avoid resource name collisions?

android { resourcePrefix = "player_" }. Then every R.color, R.drawable, R.string in your library must start with that prefix — Gradle enforces it. Consumers can't accidentally clobber your resources.

Q. How do you handle Hilt in a library?

Avoid bundling Hilt — it constrains consumers. Use plain constructor injection internally. If consumers want Hilt convenience, ship a separate :foo-hilt artifact with Hilt modules. Or expose factory functions and let consumers wire DI their way.

Q. SemVer — when do you bump major?

Any breaking API change: removed/renamed public symbol, changed signature, raised minSdk, behavior change that consumers can detect. Use binary compatibility validator in CI to catch accidents.

Q. What goes in pom { } and why?

Name, description, license, developers, SCM URL. Maven Central requires these for ingest — without them Sonatype rejects the upload. Also helps tools like Renovate / Dependabot show meaningful info about your library to humans.

8. The 30-second monologue

Say this if asked "how do you publish your code as a library?"

"I switch the module to com.android.library for AAR output, set a resourcePrefix to avoid resource collisions, ship consumerProguardFiles so my reflection-heavy paths survive consumer R8, mark internals internal by default, default deps to implementation over api to hide transitive choices, and follow SemVer with a binary compatibility validator in CI. For Maven Central I configure maven-publish with full POM metadata, GPG-sign artifacts, and ship a sources JAR for IDE step-through. Hilt I avoid bundling — I use constructor injection internally and ship an optional -hilt artifact for convenience."
