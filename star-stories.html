STAR Stories — past projects

Pre-built narratives from your real Dish work. Each story is told as Situation → Task → Action → Result, with a defensive frame for the inflated areas you self-flagged.

Story 1 — EPG Compose rebuild (your strongest)

Situation: Dish's Android TV app had a legacy EPG (electronic program guide) built in XML/RecyclerView. Load times 4–5s, jank on low-end devices, OOMs scrolling past ~200 channels. Bug count climbing, customer escalations.

Task: Lead the rebuild. Cut load time, eliminate OOMs, set the foundation for future EPG features.

Action:

  • Audited the legacy code — found bitmap overhead (logos kept full-resolution), unbounded view recycling, blocking JSON parse on main.
  • Designed a Compose EPG using LazyRow of LazyColumns, with stable keys per cell.
  • Hoisted state to an EpgViewModel exposing StateFlow<EpgState>; loaded data lazily for the visible window.
  • Used remember(channelId) per-cell, marked the data class @Immutable after Compose Compiler reports flagged it as unstable.
  • Compressed channel logos via Coil with size-aware caching.
  • Profiled OOMs — found a leak in the legacy view holder retaining bitmaps after recycle. Cleared in onDispose.
  • Benchmarked startup with Macrobenchmark before/after.

Result: Load time 60–70% cut (4.5s → 1.4s on a low-end TV). Render time 50% cut. OOMs resolved (verified via 7-day Crashlytics window). Bug count down ~80%. Set foundation for 5+ subsequent features.

Defensive frame

If asked about "industry-level Compose" — "We followed standard practices: stability annotations, derivedStateOf for the visible-window flag, collectAsStateWithLifecycle, lazy keys. The rebuild was complex enough to expose all the recomposition gotchas. We didn't build a full Material 3 design system from scratch — we used the existing Dish brand tokens and built reusable composables for the channel cell and time bar. Building a full design system is a step I'd love to lead next."

Story 2 — Ad monetization stack from scratch

Situation: New Dish streaming product launching. Needed full ad monetization on Day 1 — display ads, SSAI for video, beacon tracking, lifecycle management — with no existing in-house tooling.

Task: Build it end-to-end across Android TV and tablet. Reliable, observable, resilient to ad partner outages.

Action:

  • Designed an AdsController interface with loadAd, showAd, dismissAd. Abstracted partner SDKs (AdMob NextGen, FreeWheel) behind it.
  • SSAI: parse ad pod metadata from manifest, wire beacons at quartile boundaries.
  • Pre/mid/post-roll lifecycle: pause user content during CSAI, disable seek during ad windows, restore controls after.
  • Found and fixed a memory leak in AdMob NextGen — the SDK retained ActivityContext past lifecycle. Wrapped in a weak-reference holder with explicit release in onDestroy.
  • Built analytics layer over ad lifecycle (impressions, click rate, beacon failure).
  • Negotiated with ad-partner team on retry/timeout — settled on 2 retries with 1.5s timeout per beacon.

Result: Stack shipped with launch. Ad delivery reliability ~99.5% in production. Memory leak fix prevented an estimated 5–10% of OOMs on devices with smaller heaps. Analytics surfaced 3 partner-side issues in the first month.

Defensive frame

If asked "did you design the SSAI server-side?" — "No, the server team handled stream stitching. I owned the client side — beacon firing, lifecycle integration, and the player wiring. I worked closely with the server team on the manifest format and the beacon contract."

If asked about test strategy — "Mock ad partner with Charles/Proxyman, plus chaos testing in QA: drop network mid-ad, kill app between impressions, etc."

Story 3 — Multi-module Android TV/Tablet shared codebase

Situation: Two product surfaces (TV + tablet) had diverged. Duplicated code, drifting business logic, hard to keep in sync. Build times growing.

Task: Architect a shared codebase that supports both, with platform-specific UI affordances.

Action:

  • Audited duplication — found ~70% of business logic identical, but UI patterns (10-foot vs touch) needed to differ.
  • Proposed structure: :core-domain, :core-data, :core-network, :core-player shared; :ui-tv, :ui-tablet platform-specific; :app-tv, :app-tablet entry points.
  • Built custom Gradle build variants and tasks for platform-specific resources, manifests, feature flags.
  • Defined APIs (interfaces) at module boundaries; enforced via api vs implementation dep declarations.
  • Migrated incrementally — feature by feature — over ~3 months while team kept shipping.

Result: 74% UI complexity reduction in shared code. Lean APKs optimized for low-end devices (no unused TV code in tablet APK and vice versa). Build time down ~40% on incremental builds. Two surfaces in lock-step on business logic — drift bugs eliminated.

Defensive frame

If asked "did you use dynamic feature modules?" — "We considered them but our use case was platform-split, not on-demand install — static modules with build variants were simpler and met our needs."

If asked "how did you handle Hilt across modules?" — "Hilt requires @HiltAndroidApp in the app module; we set up @InstallIn(SingletonComponent::class) modules in core modules, and each app module composes them. We use qualifiers heavily for platform-specific bindings (e.g., @TvScope vs @TabletScope)."

Story 4 — Custom analytics processing engine

Situation: We needed reliable, low-latency analytics. Off-the-shelf SDKs (Firebase, Mixpanel) were too slow for our event volume and lacked custom routing.

Task: Build a client-side processing engine — batch, dedupe, prioritize, upload 15K+ events/hour reliably.

Action:

  • Designed an AnalyticsHub with pluggable transport (HTTP, FCM data push for offline, file fallback).
  • Channel<Event> for in-process queue; consumer coroutine pulls in batches of 50 every 30s (or immediately for high-priority).
  • Persisted unsent events to Room (id, payload, priority, retry_count, last_attempt). Replayed on app start.
  • Wired to New Relic, Dynatrace, plus our own ingestion endpoint.
  • Exponential backoff on failures, max 5 retries before dead-letter queue.

Result: Reliable processing of 15K+ events/hour. Event loss rate < 0.1% (down from ~3% with previous setup). Used by product to drive 5+ decisions about EPG layout, recommendation ordering, ad placement. Engine extended for 3 new instrumentation use cases without rewriting.

Defensive frame

If asked "why not just Firebase Analytics?" — "Firebase batches lazily and offers limited routing. We needed routes to multiple backends (NewRelic, Dynatrace, internal), high-priority events flushed immediately, and custom dedupe."

Story 5 — Cold start optimization

Situation: Cold start was creeping up to 2.4s on low-end TVs (1GB RAM, slow eMMC). User-perceived "app is slow."

Task: Cut cold start materially. Target sub-1.5s.

Action:

  • Profiled with Android Studio CPU profiler + adb shell am start -W. Worst offenders: SDK init (~600ms blocking on Application.onCreate), eager network init, blocking JSON parse of cached manifest.
  • Deferred non-critical SDK init via Jetpack App Startup library.
  • Moved network-layer init off main using lifecycleScope.launch(Dispatchers.IO) post-first-frame.
  • Replaced Gson with kotlinx.serialization (faster cold start) and parsed asynchronously while showing a placeholder.
  • Generated Baseline Profiles for the cold-start path → 10–20% additional win on first launches.
  • Optimized network: payload reduction, batched calls, response caching.

Result: Cold start 2.4s → 1.1s (54% reduction) on low-end TVs. Crash rate dropped (some crashes were main-thread timeouts on slow init). Better perceived UX validated by post-launch survey.

Story 6 — Mentoring and ownership (TR4 favorite)

Situation: I was assigned to mentor 2 interns and 1 junior engineer joining the Android team.

Task: Get them productive on our codebase quickly while maintaining code quality.

Action:

  • Weekly 1:1s, structured starting tasks, pairing on architecture-heavy PRs.
  • Wrote internal docs on our module boundaries and Hilt setup since they were the trickiest gotcha.
  • Reviewed every PR with detailed feedback for the first month.
  • Gradually increased complexity of their tasks.
  • Encouraged them to lead small features end-to-end.

Result: All three independently shipped non-trivial features within 6 weeks. One intern converted to FTE. Internal docs we wrote became part of onboarding for subsequent hires.

Story 7 — A failure / learning (always have one ready)

Situation: I once shipped a "fix" for a flaky test that turned out to be a real concurrency bug. The test was failing intermittently because our coroutine setup was racy — I @Ignored it to unblock the release.

Task (self-imposed): Investigate why the test failed before just disabling it.

Action — what went wrong:

  • I assumed the test was flaky because of test infrastructure.
  • Disabled it, shipped.
  • Two weeks later, a customer-reported crash traced back to the exact race I'd dismissed.

Action — what I did to fix and learn:

  • Re-investigated with runTest and StandardTestDispatcher; reproduced the race.
  • Fixed it with proper structured concurrency (coroutineScope instead of unbounded launch).
  • Added a team norm: no @Ignore without a JIRA ticket linked + an investigation note.

Result: Bug fixed before next release. New norm caught two more "flaky" tests that were real bugs in subsequent months.

Why this story works

Shows ownership, learning, willingness to admit fault, and process improvement. Bar raisers love this combo. Don't sanitize — the "I disabled it" admission is what makes it credible.

Story 8 — Disagreement (TR4 will probe)

Situation: PM wanted to ship a new search feature without offline cache, citing time-to-market. I'd seen the data — ~25% of our user sessions had at least one network blip.

Task: Either ship as planned or push back.

Action:

  • Built a 30-min spike showing offline-cached search with a thin wrapper over our Room layer.
  • Brought numbers to PM: 25% sessions affected, latency reduction even online (60ms → 5ms for cached queries).
  • Proposed 2-day spike to integrate cleanly.
  • Compromised: shipped with cache for already-searched queries (not full pre-fetch), reducing scope but covering the worst case.

Result: Search latency dropped 50% online (P95). Offline-search worked for repeat users on subway commutes (a known persona). PM agreed to add full pre-fetch in V2.

Why this story works

Data-driven, collaborative, ended with a compromise — not "I won." Shows you bring evidence, propose alternatives, and respect product judgment.

"Why FOX?" / "Why are you leaving Dish?" (HR + TR4)

Why FOX

"FOX's DTC streaming work is at exactly the scale and complexity I want to grow on — Tubi, FOX Sports, FOX News all push different problems (ad-supported, live, news cadence). My current player work is directly relevant — DRM, SSAI, ABR — and I'd love to apply that on a team solving for tens of millions of users instead of our current scale. I've been the most senior Android engineer on my immediate team for ~2 years; I want to learn from a team where the median is more senior than me."

Why leaving Dish

"I've delivered what I set out to at Dish — full ownership of the player module, the multi-module migration, the ad stack. The natural next step for my growth is a larger DTC platform with a deeper Android engineering bench. No specific complaints about Dish — it's a growth move."

Don't trash Dish

Even mild negativity about a previous employer reads as a red flag. Frame everything as "growth" or "alignment with what I want next." If asked specifically what didn't work, say "the org chart shifted such that the Android team's roadmap moved away from the streaming work I want to do."

Delivery tips

  • Time each story — 60–90 seconds is ideal. Practice with a stopwatch.
  • Don't memorize verbatim — internalize the structure (S/T/A/R) and the numbers.
  • Pause after Action — interviewer often probes here.
  • Lead with impact in TR4: "I rebuilt our EPG in Compose — cut load time 60% and OOMs to zero. Want me to walk through how?"
  • Adapt the story to the question — same project, different angle. EPG can be told as "ownership", "performance work", "Compose deep dive", or "leading a rebuild."
  • One concrete number per story, minimum. Specifics beat adjectives.

Common bar-raiser questions to expect

Q. Tell me about a time you disagreed with your manager.

Story 8 (search caching). Frame as data + proposal + compromise.

Q. What's the toughest technical problem you've solved?

The ad lifecycle + AdMob memory leak (Story 2). Or the EPG OOM (Story 1). Pick whichever is closer to FOX's domain — ads + DRM is closest.

Q. How do you stay current with Android?

Specific sources: Now in Android podcast, Google I/O sessions, Android Developers blog, Square's tech blog, Jake Wharton on Mastodon, AOSP commits when investigating specific issues. Avoid generic "I read articles" — name actual sources.

Q. What would you do in your first 30 days at FOX?

"Listen, ramp, ship something small. Read the player module top to bottom. Pair with the senior Android engineer on the team to understand conventions. Pick up a small bug to learn the build/test/release cycle. Avoid pushing big architectural opinions in week 1."

Q. What's your weakness?

Real one with mitigation: "I lean on AI tooling more than I should — I've made it a habit this past month to write features by hand first, then use AI for review or edge cases. It's improved my speed and confidence in code review." Don't say "I'm a perfectionist" — bar raisers groan.
