1. "View value isn't being restored after screen reorientation"

What's actually happening: The Activity gets recreated on rotation. Anything stored only in field variables on the Activity/Fragment is lost. ViewModel survives rotation but not process death — so depending on where you stored the value, the cause differs.

Diagnosis checklist (in order)

  1. Is the value held in a field of the Activity/Fragment? Then it's lost on rotation by design — that field is on a destroyed instance.
  2. Compose: are you using remember { mutableStateOf(...) }? remember only survives recomposition — NOT activity recreate. Use rememberSaveable.
  3. XML: is the View a custom view that didn't override onSaveInstanceState/onRestoreInstanceState? Standard Views (EditText, etc.) restore IF they have an android:id.
  4. Did you set android:configChanges="orientation|screenSize" in the manifest? Then the Activity is NOT recreated — but anything depending on rotation-driven recreation (theme switch, layout pickup) is broken.
  5. Is the value in ViewModel? ViewModel survives rotation. If still lost, you're hitting process death — the ViewModel is recreated. Use SavedStateHandle (Bundle-backed; survives both).

Fix recipe — Compose

@Composable
fun NameField() {
    // BAD — lost on rotation
    var name by remember { mutableStateOf("") }

    // GOOD — survives configuration change AND process death (Bundle-based, < ~50KB)
    var name by rememberSaveable { mutableStateOf("") }

    TextField(name, { name = it })
}
remember stores in the slot table and only persists across recompositions. rememberSaveable additionally serializes the value into the SavedStateRegistry (Bundle-backed), so it survives Activity recreation AND a temporary process kill.

Fix recipe — ViewModel-owned input

@HiltViewModel
class FormVm @Inject constructor(
    private val savedState: SavedStateHandle    // injected by Hilt — backed by Bundle
) : ViewModel() {
    val name: StateFlow<String> = savedState.getStateFlow("name", "")
    fun onNameChange(v: String) { savedState["name"] = v }
}

@Composable
fun NameField(vm: FormVm = hiltViewModel()) {
    val name by vm.name.collectAsStateWithLifecycle()
    TextField(name, vm::onNameChange)
}
SavedStateHandle is a key-value map persisted into the saved-instance-state Bundle. getStateFlow exposes it as an observable Flow. Updates via savedState["name"] = v are written through. This survives both rotation (ViewModel) and process death (Bundle).

Fix recipe — XML custom View

class NameView : LinearLayout {
    /* ... */

    override fun onSaveInstanceState(): Parcelable? {
        return Bundle().apply {
            putParcelable("super", super.onSaveInstanceState())
            putString("name", currentText())
        }
    }
    override fun onRestoreInstanceState(state: Parcelable?) {
        val bundle = state as? Bundle ?: return super.onRestoreInstanceState(state)
        super.onRestoreInstanceState(bundle.getParcelable("super"))
        setText(bundle.getString("name", ""))
    }
}
Custom Views must explicitly serialize their internal state. The framework calls onSaveInstanceState when the Activity is being recreated; the returned Parcelable goes into the Bundle and comes back via onRestoreInstanceState on the new instance. Always carry the super.onSaveInstanceState() Parcelable forward — that's the parent class's state.

2. "Implement a cache so the app shows data offline + survives network failure"

The senior pattern: offline-first with Room as source of truth. UI observes the database via Flow; the network layer writes through to the DB. This guarantees no UI/cache drift and gives you offline reads for free.

Architecture

┌──── Network (Retrofit) ────┐
│  api.fetchNotes()          │
└────────────┬───────────────┘
             ↓ writes through
┌──── Room (source of truth) ─┐
│  dao.upsert(...)            │
│  dao.observeAll(): Flow     │
└────────────┬───────────────┘
             ↓ collects
┌──── ViewModel ─────────────┐
│  StateFlow<UiState>        │
└────────────┬───────────────┘
             ↓
┌──── UI ────────────────────┐
└────────────────────────────┘

Repository implementation

class NotesRepoImpl @Inject constructor(
    private val dao: NotesDao,
    private val api: NotesApi,
    @IoDispatcher private val io: CoroutineDispatcher
) : NotesRepo {

    // UI observes the DB. Always emits — even if offline (last-cached values).
    override fun observeAll(): Flow<List<Note>> =
        dao.observeAll().map { entities -> entities.map { it.toDomain() } }

    // refresh() is best-effort. Failure leaves DB unchanged → UI keeps showing cached data.
    override suspend fun refresh(): Result<Unit> = withContext(io) {
        runCatching {
            val remote = api.fetchNotes()                 // throws on network failure
            dao.replaceAll(remote.map { it.toEntity() })  // write-through to DB
        }
    }
}
  1. Source of truth = Room. observeAll() returns a Flow that Room emits to whenever the table changes — UI auto-updates without manual orchestration.
  2. refresh() is fail-safe: runCatching wraps the network call. If it throws, the DB isn't touched and the existing Flow values stay valid. UI shows cached data; you can surface a "couldn't refresh" snackbar separately.
  3. I/O on the right dispatcher: withContext(io) runs the network + DB write on a background thread.

ViewModel orchestration

@HiltViewModel
class NotesVm @Inject constructor(
    private val repo: NotesRepo
) : ViewModel() {

    // Offline-first: emits cached data immediately, refreshes in background.
    val notes: StateFlow<List<Note>> = repo.observeAll()
        .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), emptyList())

    private val _refreshError = MutableSharedFlow<String>(extraBufferCapacity = 1)
    val refreshError: SharedFlow<String> = _refreshError.asSharedFlow()

    init { refresh() }   // kick off background refresh on first creation

    fun refresh() = viewModelScope.launch {
        repo.refresh().onFailure { _refreshError.emit("Couldn't sync — showing cached") }
    }
}
UI gets two streams: notes (always populated from DB, never empty after first sync) and refreshError (one-shot event when a refresh fails). The user sees cached content immediately on app launch — even with no network — and a subtle banner if the refresh failed.

Background sync via WorkManager

// On app start, schedule periodic sync that respects network conditions
val sync = PeriodicWorkRequestBuilder<NotesSyncWorker>(15, TimeUnit.MINUTES)
    .setConstraints(
        Constraints.Builder()
            .setRequiredNetworkType(NetworkType.CONNECTED)
            .setRequiresBatteryNotLow(true)
            .build()
    )
    .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 30, TimeUnit.SECONDS)
    .build()

WorkManager.getInstance(ctx).enqueueUniquePeriodicWork(
    "notes-sync", ExistingPeriodicWorkPolicy.KEEP, sync
)
WorkManager handles network return for free — its constraint system waits for connectivity before running the worker, retries with exponential backoff on failure, survives reboot.

3. "App is throwing ANRs — how do you fix?"

ANR ("Application Not Responding") = the OS detects your main thread is blocked too long. Triggers: UI input not handled within 5s, BroadcastReceiver onReceive >10s, Service onStart >20s.

Diagnosis sequence

  1. Get the stack trace. Play Console → Vitals → ANR. Or /data/anr/traces.txt on a rooted device. The main thread stack at the moment of ANR is what you investigate.
  2. Look at the top frames of main. If it's BinderProxy.transact → IPC call (often ContentProvider or system service). If it's java.io.* → disk read on main. If it's HttpURLConnection → network on main (yes, people still do this).
  3. Look at locks. If main is blocked on Object.wait or a synchronized monitor, find which other thread holds the lock — usually a deadlock or a slow background operation.
  4. Reproduce in debug with StrictMode. StrictMode catches main-thread disk/network in real-time during development.

StrictMode setup (debug only)

class App : Application() {
    override fun onCreate() {
        super.onCreate()
        if (BuildConfig.DEBUG) {
            StrictMode.setThreadPolicy(StrictMode.ThreadPolicy.Builder()
                .detectDiskReads()
                .detectDiskWrites()
                .detectNetwork()
                .detectCustomSlowCalls()
                .penaltyLog()           // log violations
                .penaltyDeath()          // crash on violation — uncomment to be aggressive
                .build())
            StrictMode.setVmPolicy(StrictMode.VmPolicy.Builder()
                .detectLeakedClosableObjects()
                .detectActivityLeaks()
                .penaltyLog()
                .build())
        }
    }
}
StrictMode is the OS's lint for runtime sins. ThreadPolicy watches the main thread for disk/network operations. VmPolicy watches the whole VM for leaked Cursors, Closables, Activities. penaltyLog just logs; penaltyDeath crashes — useful in CI to fail builds on regressions.

Common ANR fixes

CauseFix
Disk I/O on main (SharedPreferences commit(), file reads)Use DataStore (async) or move to a coroutine on Dispatchers.IO
JSON parsing on mainParse on Dispatchers.Default; switch to kotlinx.serialization for less GC churn
Big onBindViewHolderPre-compute / pre-format off main; bind only display strings on main
BroadcastReceiver doing work in onReceiveEnqueue a WorkManager job from onReceive; or call goAsync() for short async
Synchronous network call (legacy)Coroutine + Retrofit's suspend functions
Held lock across slow operationReduce critical section; switch to Mutex in coroutine code
Cold-start work in Application.onCreateDefer via App Startup library or lifecycleScope.launch(IO) after first frame

Senior framing to say out loud

Senior phrase

"For ANRs I work backward from Play Console Vitals — the main-thread stack at the moment of ANR points at the culprit. I run StrictMode in debug builds to catch disk/network-on-main before it ANRs in production. The structural fix is usually to move I/O off main via coroutines + injected dispatchers, defer non-critical SDK init via App Startup, and shrink critical sections that hold locks."

4. "Smooth scrolling for a list of 10,000 items"

Compose — LazyColumn

// Stable, immutable item type
@Immutable
data class FeedItem(val id: String, val title: String, val thumbnailUrl: String)

@Composable
fun Feed(items: List<FeedItem>) {
    LazyColumn {
        items(
            items = items,
            key = { it.id },                      // stable identity → correct recomposition
            contentType = { "feed_item" }          // helps view-pool reuse for heterogeneous lists
        ) { item ->
            FeedRow(item)
        }
    }
}

@Composable
private fun FeedRow(item: FeedItem) {
    // Coil sized to the rendered area — prevents giant bitmap decode
    AsyncImage(
        model = ImageRequest.Builder(LocalContext.current)
            .data(item.thumbnailUrl)
            .size(160)                            // pixel dimension cap
            .crossfade(true)
            .build(),
        contentDescription = null,
        modifier = Modifier.size(80.dp)
    )
    Text(item.title)
}
  1. @Immutable on the data class: tells Compose it can skip recomposition when the parameter equals previous. Without it, List-typed fields make the type unstable.
  2. key = { it.id }: Compose uses this to identify items across data updates. Without keys, scroll position jumps on insertions/reorderings.
  3. contentType: helps the LazyColumn's view pool reuse compositions for items of the same shape.
  4. Coil .size(160): bitmap is decoded at request size, not source size. Critical for grids of small thumbnails.

XML — RecyclerView

class FeedAdapter : ListAdapter<FeedItem, FeedVH>(DIFF_CALLBACK) {
    override fun onCreateViewHolder(parent: ViewGroup, viewType: Int): FeedVH {
        val v = LayoutInflater.from(parent.context).inflate(R.layout.row_feed, parent, false)
        return FeedVH(v)
    }
    override fun onBindViewHolder(h: FeedVH, pos: Int) {
        h.bind(getItem(pos))
    }
    companion object {
        val DIFF_CALLBACK = object : DiffUtil.ItemCallback<FeedItem>() {
            override fun areItemsTheSame(old: FeedItem, new: FeedItem) = old.id == new.id
            override fun areContentsTheSame(old: FeedItem, new: FeedItem) = old == new
        }
    }
}

// In layout XML — fixed item size + RecyclerView optimizations
<androidx.recyclerview.widget.RecyclerView
    android:id="@+id/list"
    android:layout_width="match_parent"
    android:layout_height="match_parent"
    app:layout_behavior="@string/appbar_scrolling_view_behavior" />

// In code — when items have a fixed height
recyclerView.setHasFixedSize(true)         // skip a measure pass on inserts
recyclerView.itemAnimator = null            // disable animations during heavy scroll if needed
recyclerView.recycledViewPool.setMaxRecycledViews(VIEW_TYPE_FEED, 20)
  1. ListAdapter + DiffUtil: only recomputes/redraws changed rows when you call submitList(newList). Crucial for performance with frequent updates.
  2. setHasFixedSize(true): tells RecyclerView item size doesn't change on data changes — skips a layout pass.
  3. onBindViewHolder stays cheap: only set already-formatted strings + load images via Glide/Coil. Never compute or do I/O here.

For huge datasets — Paging 3

10K items in memory may be fine; 1M items is not. Use Paging 3 with RemoteMediator so only the visible window + buffer is loaded at any time. See Jetpack page.

5. "How do I store API keys securely?"

The honest answer first

Anything shipped in the APK can be extracted. The goal is to raise the bar, not achieve absolute security.

Layered approach

  1. Don't put real secrets in source control. Use local.properties or environment variables; load via buildConfigField.
  2. Don't ship long-lived shared secrets to clients. Use OAuth + PKCE for user auth; ephemeral tokens are fine. For service-level secrets, route calls through your backend.
  3. Obfuscate via R8. Keeps casual reverse-engineering tedious. Doesn't stop a determined attacker.
  4. Use NDK / native code for the most sensitive constants — harder to extract than Java strings, but still possible.
  5. For per-user tokens: store in EncryptedSharedPreferences (Android Keystore-backed).

Build-config approach (better than hardcoded)

// gradle.properties (gitignored or in CI secrets)
API_KEY=abc123

// app/build.gradle.kts
android {
    defaultConfig {
        buildConfigField("String", "API_KEY", "\"${project.findProperty("API_KEY")}\"")
    }
}

// In code
val key = BuildConfig.API_KEY
The key isn't in source. It still ends up as a String in the compiled APK — extractable with strings on the dex — but at least it's not in your git history or accessible to people reading your repo.

EncryptedSharedPreferences for user tokens

val masterKey = MasterKey.Builder(ctx)
    .setKeyScheme(MasterKey.KeyScheme.AES256_GCM)   // hardware-backed when available
    .build()

val prefs = EncryptedSharedPreferences.create(
    ctx, "secure_tokens", masterKey,
    EncryptedSharedPreferences.PrefKeyEncryptionScheme.AES256_SIV,
    EncryptedSharedPreferences.PrefValueEncryptionScheme.AES256_GCM
)

prefs.edit().putString("access_token", token).apply()
The master key lives in Android Keystore. On devices with hardware-backed Keystore (most modern phones), the key never leaves the secure element — even root can't extract it. The encrypted blob is stored in normal SharedPreferences XML; only the device that wrote it can read it back.

6. "Why ProGuard / R8?"

R8 = the modern code shrinker, obfuscator, and optimizer Android uses in release builds. It absorbed ProGuard's role; you'll still see "ProGuard rules" because the rule file format (proguard-rules.pro) is the same.

What R8 does (release builds only by default)

  1. Tree-shaking — removes unused classes, methods, fields. Often 20-40% APK size reduction.
  2. Optimization — inlines methods, propagates constants, removes dead code.
  3. Obfuscation — renames classes/methods to a/b/c. Smaller bytecode + harder to reverse-engineer.
  4. Resource shrinking (with shrinkResources true) — strips unused images, layouts, strings.

Enable in build.gradle

android {
    buildTypes {
        release {
            isMinifyEnabled = true            // R8 code shrink + obfuscate
            isShrinkResources = true           // strip unused resources
            proguardFiles(
                getDefaultProguardFile("proguard-android-optimize.txt"),
                "proguard-rules.pro"
            )
        }
    }
}

Common keep rules you'll write

# For reflection-heavy libs (Gson, Moshi, Retrofit @SerializedName)
-keep class com.example.app.dto.** { *; }
-keepattributes Signature
-keepattributes RuntimeVisibleAnnotations

# For Kotlinx Serialization
-keepattributes *Annotation*, InnerClasses
-dontnote kotlinx.serialization.AnnotationsKt

# Keep your Retrofit interface
-keep,allowobfuscation,allowshrinking interface com.example.app.api.**

# Don't strip data classes serialized by Gson
-keepclassmembers class com.example.app.dto.** {
    <fields>;
    <init>(...);
}
"Keep" rules tell R8 NOT to remove or obfuscate matching symbols. You need them for code accessed via reflection — JSON DTOs, Retrofit interfaces, anything called from JNI, anything referenced by class name from XML or strings. Without keep rules, R8 may rename a field that GSON tries to deserialize by name → runtime crash.

7. "AAB vs APK — what's the advantage?"

APK = a single installable Android package. Same binary delivered to every device.
AAB (Android App Bundle) = upload format for Play Store. Play uses it to generate per-device APKs with only the resources/code needed for that specific device.

What gets split per-device

  • ABI splits — only ARM64 native libs go to ARM64 devices, etc.
  • Density splits — only mdpi/hdpi/xxhdpi resources for the device's actual density.
  • Language splits — only English+French for a French-system device, etc.

Concrete numbers

Typical app: a 50 MB universal APK becomes a ~22 MB per-device download via AAB. Significant on low-end markets where data is metered.

Other AAB benefits

  • Dynamic feature modules — install rarely-used features on demand (Play Feature Delivery).
  • Asset packs — separate-from-base downloads for large assets (game levels, video catalogs).
  • Play App Signing — Google holds the app signing key; you sign with an upload key. Signing key rotation possible.

Mandatory since 2021

Play Store requires AAB for new apps. APK is still fine for sideloading or alternative stores.

8. "Memory leaks — how do you find and fix?"

Memory leak = an object that's no longer logically needed but is still reachable from a GC root, so the garbage collector can't free it. On Android, the classic case is an Activity instance retained past its destruction.

Detection

  1. LeakCanary in debug builds — automatically detects retained instances and reports the GC-root chain. Just adding it as a debug dependency is enough.
  2. Profiler heap dumps for runtime investigation — Android Studio → Profiler → Memory → "Capture heap dump" after reproducing the leak. Group by class; look for Activity/Fragment with >1 instance.
  3. StrictMode VmPolicy.detectActivityLeaks() — logs in real time when an Activity isn't GC'd after destruction.
  4. Production crash reports with patterns like "OutOfMemoryError" or unbounded memory growth in Crashlytics.

The classic causes

CauseFix
Singleton holds an Activity ContextUse Application Context (or @ApplicationContext via Hilt)
Anonymous inner class (Handler, Runnable) holds outer referenceUse static nested classes + WeakReference, or migrate to coroutines
Listeners registered, never unregisteredUnregister in onDestroy / onPause mirroring registration
GlobalScope.launch captures ActivityUse viewModelScope or lifecycleScope
ExoPlayer not releasedplayer.release() in onDestroy / onDispose
Fragment uses this as LifecycleOwner for view-bound observersUse viewLifecycleOwner instead
RxJava subscription not disposed (legacy)Add to CompositeDisposable; clear in onDestroy
Bitmap cache unboundedLruCache with maxSize in bytes; or use Coil/Glide

Heap-dump workflow (step-by-step)

  1. Run debug build on device. Open Android Studio Profiler.
  2. Reproduce the suspected leak: open the screen, rotate 5 times, navigate away.
  3. Profiler → Memory tab → click "Capture heap dump".
  4. Group by class. Type your Activity name.
  5. If >1 instance, you have a leak. Click an instance → "Show nearest GC root."
  6. The chain shows what's holding it — usually a static field, singleton, or registered listener.
  7. Fix the chain root. Re-run, dump, verify count is back to 0 (or 1 if currently visible).

9. "Multi-module app — how do you handle navigation + state sharing?"

Navigation

Don't let feature modules know about each other's destinations. Do route through interfaces.

// :feature:home — declares an interface for "where to go from here"
interface HomeNavigator {
    fun openDetail(id: String)
    fun openProfile()
}

// :app — implements it knowing about all features
class AppNavigator(private val nav: NavController) : HomeNavigator {
    override fun openDetail(id: String) = nav.navigate("detail/$id")
    override fun openProfile() = nav.navigate("profile")
}

// Hilt provides the impl
@Module @InstallIn(ActivityComponent::class)
abstract class NavigatorModule {
    @Binds abstract fun bindHome(impl: AppNavigator): HomeNavigator
}
Each feature module declares what it needs to navigate to (HomeNavigator interface) but doesn't know how. The :app module wires them all together with the actual NavController. This keeps :feature:home from depending on :feature:detail — no cycles, fast incremental builds.

State sharing across modules

  1. Repository in a shared :core:data module. Feature modules depend on the repository interface in :core:domain. Multiple features observe the same Flow → same data, automatic sync.
  2. Shared ViewModel via navGraphViewModels (XML) or shared NavBackStackEntry in Compose: scope a ViewModel to a sub-graph so multiple destinations see the same instance.
  3. Cross-module events via a shared SharedFlow in :core:common, fed by features and observed by others. Use sparingly — easy to abuse.

Shared graph ViewModel — Compose example

@Composable
fun CheckoutGraph(navController: NavHostController) {
    NavHost(navController, startDestination = "cart") {
        navigation(startDestination = "cart", route = "checkout") {
            composable("cart") { entry ->
                val parent = remember(entry) { navController.getBackStackEntry("checkout") }
                val vm: CheckoutVm = hiltViewModel(parent)   // shared across cart/payment/confirm
                CartScreen(vm)
            }
            composable("payment") { entry ->
                val parent = remember(entry) { navController.getBackStackEntry("checkout") }
                val vm: CheckoutVm = hiltViewModel(parent)
                PaymentScreen(vm)
            }
            composable("confirm") { entry ->
                val parent = remember(entry) { navController.getBackStackEntry("checkout") }
                val vm: CheckoutVm = hiltViewModel(parent)
                ConfirmScreen(vm)
            }
        }
    }
}
hiltViewModel(parent) scopes the ViewModel to the parent backstack entry — the "checkout" sub-graph. All three screens see the same CheckoutVm instance with shared state (cart contents, selected payment, etc.). The VM is cleared when the user navigates out of the entire "checkout" graph.

10. "Constructor injection vs field injection — which and why?"

Constructor injection: dependencies passed via the constructor parameters. Default for plain Kotlin classes.
Field injection: @Inject lateinit var dep: Dep set after construction. Used for Android components Hilt can't construct (Activity, Fragment, etc.).

Why constructor injection is preferred

  1. Immutability. Dependencies are val; can't be reassigned mid-life.
  2. Compile-time safety. Missing dependency → compile error, not runtime NPE.
  3. Easy unit testing. Pass fakes directly to the constructor — no DI framework needed in tests.
  4. Explicit dependency surface. Reading the constructor tells you everything the class needs.
  5. No lateinit ceremony. No risk of calling a method before injection happens.

When you must use field injection

Android constructs Activities, Fragments, Services, Receivers, Views — you don't get to control their constructors. Hilt's @AndroidEntryPoint generates the field-injection plumbing for these cases.

// Plain class — constructor injection
class FetchUseCase @Inject constructor(
    private val repo: NotesRepo,
    @IoDispatcher private val io: CoroutineDispatcher
) {
    suspend operator fun invoke(): List<Note> = withContext(io) { repo.fetchAll() }
}

// Android component — field injection (you don't own the constructor)
@AndroidEntryPoint
class HomeFragment : Fragment() {
    @Inject lateinit var analytics: AnalyticsTracker      // injected after creation

    override fun onResume() {
        super.onResume()
        analytics.screen("home")     // safe — Hilt injects before lifecycle calls
    }
}
HomeFragment is constructed by the Android framework; you can't add constructor params. Hilt's generated code injects fields right before super.onCreate runs, so any lifecycle method can safely use them.

11. "ContentProvider — what is it and when?"

ContentProvider = an Android component that exposes structured data (typically backed by SQLite or files) to other apps. URIs of the form content://authority/path/id. Used by system apps (Contacts, Calendar, MediaStore) and rarely by app-internal code.

When to use one

  • You're building a launcher / system-style app that needs to be queryable.
  • You need to share data across multiple of your own apps without a shared backend.
  • You want to hand a content:// URI to another app for it to read (e.g., letting the system Photo Picker read your gallery).

When NOT to

Within a single app, use Room directly. ContentProvider adds boilerplate, IPC overhead, and security surface area — only worth it if there's a true cross-app contract.

Typical consumer-side usage

val cursor = ctx.contentResolver.query(
    MediaStore.Images.Media.EXTERNAL_CONTENT_URI,
    arrayOf(MediaStore.Images.Media._ID, MediaStore.Images.Media.DISPLAY_NAME),
    null, null,
    "${MediaStore.Images.Media.DATE_ADDED} DESC"
)
cursor?.use {
    while (it.moveToNext()) {
        val id = it.getLong(0)
        val name = it.getString(1)
        // ...
    }
}
This queries the system MediaStore for all images, sorted newest first. The cursor.use block ensures the cursor is closed even on exception. Always project specific columns instead of null (= "all") for performance and forward compatibility.

12. "HashMap vs HashSet vs LinkedHashMap — which when?"

NeedUseWhy
Key → value lookupHashMapO(1) average
Just "is X in this set?"HashSetO(1) average; no value to store
Insertion order preservedLinkedHashMapHashMap + doubly-linked list of entries
Sorted by keyTreeMapO(log n); red-black tree
Int-keyed map (no autoboxing)SparseArrayAvoids Integer boxing on hot paths
Bounded LRU cacheLruCache (built on LinkedHashMap)Evicts oldest when full

Quick example — counting word frequencies

// HashMap — counts per key
val counts = HashMap<String, Int>()
for (word in words) counts[word] = (counts[word] ?: 0) + 1

// Or idiomatically with Kotlin
val counts = words.groupingBy { it }.eachCount()
groupingBy + eachCount is a Kotlin stdlib idiom that compiles to roughly the same logic. Read this when you spot the imperative HashMap-counter pattern in code review — replacing it makes intent clearer.

13. "Service vs WorkManager — when which?"

NeedUse
User-visible long-running task (music, location, video processing)Foreground Service (with notification)
Bound IPC to another componentBound Service
Deferrable + guaranteed task (sync, upload, indexing)WorkManager
Schedule wakeup at exact timeAlarmManager (rare; needs SCHEDULE_EXACT_ALARM permission Android 12+)
"Background" task you used to start a Service forWorkManager — background Services are blocked on Android 8+
Background service restrictions (Android 8+, API 26)

You cannot start a regular Service while the app is in the background — the OS throws IllegalStateException. The fix is either: (a) a foreground Service (must show a notification within 5s), (b) WorkManager for deferrable work, or (c) JobScheduler.

14. "Scenario: you're shipping to Play Store with a 64MB APK"

Reduction checklist

  1. Switch to App Bundle (AAB). Per-device download often cut by 50%.
  2. Enable R8: isMinifyEnabled = true, isShrinkResources = true.
  3. Convert PNGs to WebP (Android Studio → right-click → Convert to WebP). 25-50% size cut for lossy.
  4. Use vector drawables for icons. One asset, all densities.
  5. Audit dependencies. Use APK Analyzer in Android Studio. Look for huge transitive deps (Guava, big util libs).
  6. Strip native lib ABIs you don't ship: ndk { abiFilters += listOf("arm64-v8a", "armeabi-v7a") } — drop x86 if you don't need it.
  7. Dynamic feature modules for rarely-used features (Play Feature Delivery).
  8. Asset packs for large media (use Play Asset Delivery instead of bundling).

15. "Build an offline-first app from scratch — end-to-end"

Offline-first means the app is fully usable without a network. Reads come from the local DB (which is the source of truth); writes go to the DB immediately and are queued for sync to the server when connectivity returns. The user never sees a spinner waiting on the wire for a normal interaction.

This is the senior-level walkthrough that pulls together everything: cache, secure auth, optimistic writes, conflict resolution, network-aware sync, and offline UI. Treat this as the deep-dive answer to "design an offline-first notes app" or any similar prompt.

Goals + non-functional requirements (always state these first)

  • Reads work fully offline — Room is the source of truth; UI observes its Flow.
  • Writes apply locally immediately (optimistic UI), queue for server sync, retry on network return.
  • Server failures don't crash the UI or lose data — failures are observable, not catastrophic.
  • Conflicts resolve deterministically (last-write-wins by default; surface to user for important data).
  • Auth tokens stored securely; refresh on 401 transparent; tokens persist across app kills.
  • "You're offline" / "couldn't sync" UI cues are subtle, not modal.

Step 1 — schema with sync state

@Entity(tableName = "notes")
data class NoteEntity(
    @PrimaryKey val id: String,                  // client-generated UUID
    val serverId: String? = null,                 // populated after first sync
    val title: String,
    val body: String,
    val updatedAt: Long,                          // local mutation timestamp
    val version: Int = 0,                         // for optimistic concurrency
    val syncStatus: String,                       // SYNCED / PENDING_CREATE / PENDING_UPDATE / PENDING_DELETE
    val idempotencyKey: String                    // UUID per pending mutation; never re-used on retry
)

@Dao
interface NotesDao {
    @Query("SELECT * FROM notes WHERE syncStatus != 'PENDING_DELETE' ORDER BY updatedAt DESC")
    fun observeAll(): Flow<List<NoteEntity>>     // UI sees this — never includes "tombstoned" deletes

    @Query("SELECT * FROM notes WHERE syncStatus != 'SYNCED'")
    suspend fun pending(): List<NoteEntity>       // SyncWorker drains this

    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun upsert(note: NoteEntity)

    @Query("UPDATE notes SET syncStatus = 'SYNCED', serverId = :serverId, version = :version WHERE id = :id")
    suspend fun markSynced(id: String, serverId: String, version: Int)

    @Query("DELETE FROM notes WHERE id = :id")
    suspend fun hardDelete(id: String)
}
  1. Client-generated UUID (id) means the user can keep editing offline; we don't need a server round-trip to get an ID. Once synced, serverId fills in.
  2. syncStatus is the heart of the engine. Every row is either SYNCED (server agrees) or pending one of three operations.
  3. idempotencyKey is per-mutation, regenerated when the user makes a fresh edit. The SyncWorker sends it as a request header so the server dedupes our retries — even if our network died after the server processed the request, our retry won't create a duplicate.
  4. observeAll() filters out PENDING_DELETE — items the user deleted offline disappear from the UI immediately; they only get hard-deleted from the local DB after the server confirms. This is the "tombstone" pattern.

Step 2 — Repository: reads (offline-first cache)

interface NotesRepo {

    suspend fun add(title: String, body: String)

    suspend fun delete(id: String)

    private val dao: NotesDao,

        dao.observeAll().map { entities -> entities.map { it.toDomain() } }

    /* writes below */
}
The UI's Flow comes from Room directly. If the server is down, the network is gone, the user is on a plane — none of this matters for reads. The Flow keeps emitting whatever's in the local DB. New emissions happen only when the DB itself changes (via a write or a successful sync that updated rows).

Step 3 — Repository: writes (optimistic + enqueue)

override suspend fun add(title: String, body: String) = withContext(io) {
    val now = System.currentTimeMillis()
    val newNote = NoteEntity(
        id = UUID.randomUUID().toString(),
        title = title, body = body,
        updatedAt = now, version = 0,
        syncStatus = "PENDING_CREATE",
        idempotencyKey = UUID.randomUUID().toString()
    )
    dao.upsert(newNote)              // (1) UI updates immediately via Flow
    syncScheduler.scheduleSync()      // (2) tell WorkManager to try syncing now
}

override suspend fun edit(id: String, title: String, body: String) = withContext(io) {
    val current = dao.byId(id) ?: return@withContext
    val updated = current.copy(
        title = title, body = body,
        updatedAt = System.currentTimeMillis(),
        // PENDING_CREATE stays PENDING_CREATE if not yet synced once;
        // otherwise becomes PENDING_UPDATE
        syncStatus = if (current.syncStatus == "PENDING_CREATE") "PENDING_CREATE" else "PENDING_UPDATE",
        idempotencyKey = UUID.randomUUID().toString()  // fresh key for the new mutation
    )
    dao.upsert(updated)
    syncScheduler.scheduleSync()
}

override suspend fun delete(id: String) = withContext(io) {
    val current = dao.byId(id) ?: return@withContext
    if (current.serverId == null) {
        // Never made it to the server — just hard-delete locally.
        dao.hardDelete(id)
    } else {
        // Tombstone: hide from UI, tell server later.
        dao.upsert(current.copy(syncStatus = "PENDING_DELETE", idempotencyKey = UUID.randomUUID().toString()))
        syncScheduler.scheduleSync()
    }
}
  1. Optimistic UI: dao.upsert(...) runs first. Room's invalidation tracker fires. The Flow emits. UI updates within ~milliseconds. The user sees the change immediately.
  2. Schedule sync: syncScheduler.scheduleSync() kicks WorkManager. If online, the SyncWorker runs in seconds. If offline, WorkManager waits for connectivity (constraint) and runs as soon as the network returns.
  3. State machine for edits: if a note is still PENDING_CREATE (never synced) and the user edits it again, we keep it as PENDING_CREATE — the server will see only the final version, not an intermediate. This is "coalescing" pending mutations.
  4. Delete carefully: if the note never reached the server, hard-delete locally. Otherwise mark PENDING_DELETE so the UI hides it but we still know to call DELETE on the server.

Step 4 — SyncScheduler + SyncWorker

@Singleton
class SyncScheduler @Inject constructor(
    @ApplicationContext private val ctx: Context
) {
    fun scheduleSync() {
        val request = OneTimeWorkRequestBuilder<SyncWorker>()
            .setConstraints(
                Constraints.Builder()
                    .setRequiredNetworkType(NetworkType.CONNECTED)
                    .build()
            )
            .setBackoffCriteria(BackoffPolicy.EXPONENTIAL, 30, TimeUnit.SECONDS)
            .build()

        WorkManager.getInstance(ctx).enqueueUniqueWork(
            "notes-sync",
            ExistingWorkPolicy.APPEND_OR_REPLACE,   // collapse rapid bursts
            request
        )
    }

    fun schedulePeriodic() {
        val request = PeriodicWorkRequestBuilder<SyncWorker>(30, TimeUnit.MINUTES)
            .setConstraints(Constraints.Builder().setRequiredNetworkType(NetworkType.CONNECTED).build())
            .build()
        WorkManager.getInstance(ctx).enqueueUniquePeriodicWork(
            "notes-sync-periodic", ExistingPeriodicWorkPolicy.KEEP, request
        )
    }
}

@HiltWorker
class SyncWorker @AssistedInject constructor(
    @Assisted ctx: Context,
    @Assisted params: WorkerParameters,
    private val dao: NotesDao,
    private val api: NotesApi
) : CoroutineWorker(ctx, params) {

    override suspend fun doWork(): Result {
        val pending = dao.pending()
        if (pending.isEmpty()) return Result.success()

        for (entity in pending) {
            val outcome = runCatching {
                when (entity.syncStatus) {
                    "PENDING_CREATE" -> {
                        val resp = api.create(entity.toCreateDto(), entity.idempotencyKey)
                        dao.markSynced(entity.id, resp.serverId, resp.version)
                    }
                    "PENDING_UPDATE" -> {
                        // version sent for optimistic concurrency — server returns 409 if mismatched
                        val resp = api.update(entity.serverId!!, entity.toPatchDto(), entity.version, entity.idempotencyKey)
                        dao.markSynced(entity.id, resp.serverId, resp.version)
                    }
                    "PENDING_DELETE" -> {
                        api.delete(entity.serverId!!, entity.idempotencyKey)
                        dao.hardDelete(entity.id)
                    }
                }
            }
            outcome.onFailure { e ->
                if (e is HttpException && e.code() == 409) {
                    resolveConflict(entity)
                    // continue with the next item
                } else if (e is HttpException && e.code() in 400..499) {
                    // permanent client error — remove from queue, log, surface
                    dao.markFailed(entity.id, e.message ?: "")
                } else {
                    return Result.retry()  // transient network/5xx — exponential backoff
                }
            }
        }

        // Pull deltas after pushing
        runCatching {
            val since = dao.lastSyncedAt() ?: 0L
            val remote = api.delta(since)
            dao.upsertAll(remote.map { it.toEntity().copy(syncStatus = "SYNCED") })
            dao.setLastSyncedAt(System.currentTimeMillis())
        }.onFailure { return Result.retry() }

        return Result.success()
    }
}
  1. NetworkType.CONNECTED constraint means the worker waits for connectivity before running. If the user is offline when they save a note, WorkManager queues the worker; the moment the OS sees a usable network, the worker runs automatically.
  2. ExistingWorkPolicy.APPEND_OR_REPLACE coalesces rapid sync requests. If the user adds 5 notes in 3 seconds, we don't fire 5 workers in parallel — they batch.
  3. Idempotency key passed in every call — server dedupes if our retry arrives after the original was already processed. No phantom duplicates.
  4. 409 = conflict: we sent a stale version. Hand to resolveConflict (next step). Don't retry.
  5. 4xx (except 409) = permanent client error: don't retry forever. Mark the row failed, surface to the user, move on.
  6. 5xx + IO = transient: Result.retry(). WorkManager applies exponential backoff (30s, 60s, 120s...).
  7. Pull deltas after push: any changes other devices made since lastSyncedAt — keeps multi-device state consistent.

Step 5 — Conflict resolution (409 handling)

private suspend fun resolveConflict(local: NoteEntity) {
    // Server's current version of this note
    val remote = api.fetchOne(local.serverId!!).toEntity()

    // Strategy 1: last-write-wins by timestamp (simplest, lossy)
    if (local.updatedAt > remote.updatedAt) {
        // Force-push our version with the latest server version number
        val resp = api.update(local.serverId!!, local.toPatchDto(), remote.version, local.idempotencyKey)
        dao.markSynced(local.id, resp.serverId, resp.version)
    } else {
        // Server is newer — accept it locally, dropping our pending change
        dao.upsert(remote.copy(syncStatus = "SYNCED"))
    }

    // Strategy 2 (alternative): surface to user
    // emit ConflictEvent(local, remote) → UI prompts user to pick
    // Strategy 3 (Notion-level): three-way merge on diff of (commonAncestor, local, remote)
}
Pick a strategy upfront and call it out: (a) last-write-wins (timestamp-driven, simple, can lose user input), (b) prompt user (good UX for important data — emit a one-shot event, let UI show a "Local vs Server" picker), (c) three-way merge (Notion/Figma-level — usually overkill). Pick one per data type. For notes: prompting is reasonable. For ephemeral things like "last read position": last-write-wins is fine.

Step 6 — Network awareness: trigger immediate sync on reconnect

@Singleton
class NetworkObserver @Inject constructor(
    @ApplicationContext private val ctx: Context,
    private val syncScheduler: SyncScheduler,
    appScope: AppScope
) {
    private val cm = ctx.getSystemService(ConnectivityManager::class.java)

    val isOnline: StateFlow<Boolean> = callbackFlow {
        val callback = object : ConnectivityManager.NetworkCallback() {
            override fun onAvailable(network: Network) {
                trySend(true)
                syncScheduler.scheduleSync()    // network came back — drain the queue now
            }
            override fun onLost(network: Network) { trySend(false) }
        }
        cm.registerDefaultNetworkCallback(callback)
        awaitClose { cm.unregisterNetworkCallback(callback) }
    }.stateIn(appScope, SharingStarted.Eagerly, initialValue = isCurrentlyOnline())

    private fun isCurrentlyOnline(): Boolean {
        val net = cm.activeNetwork ?: return false
        val caps = cm.getNetworkCapabilities(net) ?: return false
        return caps.hasCapability(NetworkCapabilities.NET_CAPABILITY_INTERNET)
    }
}
Two roles:
  1. UI banner driver — isOnline is a StateFlow the UI binds to: show "You're offline, changes will sync later" when false; hide when true.
  2. Eager sync trigger — when the OS reports a network became available (onAvailable), immediately call scheduleSync(). WorkManager would eventually pick this up via its constraint system, but observing the callback fires it in a few ms instead of waiting for the system to re-evaluate constraints.
callbackFlow + awaitClose ensure we unregister the system callback when nothing's collecting — no leak.

Step 7 — Auth integration in this same flow

The SyncWorker calls go through your standard OkHttpClient with AuthInterceptor + TokenAuthenticator:

  • Tokens stored in EncryptedSharedPreferences (Keystore-backed) — survive app kill, can't be read by other apps.
  • AuthInterceptor attaches the access token on every request including SyncWorker's.
  • TokenAuthenticator handles 401 transparently — refreshes under a Mutex, retries the original request.
  • If refresh itself fails (refresh token revoked / expired beyond grace) — clear tokens, return null. SyncWorker sees the request fail, marks rows as needing re-auth, returns Result.failure(). UI surfaces "please sign in again" banner. Local data is preserved; nothing is lost.
Don't sync without auth

Before doWork() walks the queue, check tokens.refreshToken() != null. If null (logged out), exit early — no point hammering the server. The pending mutations stay queued until the user signs back in, then sync drains as expected.

Step 8 — UI: offline banner + sync-failure feedback

@Composable
fun NotesScreen(
    vm: NotesVm = hiltViewModel(),
    network: NetworkObserver = LocalNetworkObserver.current
) {
    val state by vm.state.collectAsStateWithLifecycle()
    val online by network.isOnline.collectAsStateWithLifecycle()

    Column {
        if (!online) {
            // Subtle, non-modal banner. Doesn't block interaction.
            OfflineBanner("You're offline — changes will sync when you're back online")
        }

        // Always-render content from cache. Notes flow continues to emit even offline.
        NotesList(
            notes = state.notes,
            onAdd = { title, body -> vm.onIntent(NotesIntent.Add(title, body)) },
            onEdit = { id, title, body -> vm.onIntent(NotesIntent.Edit(id, title, body)) },
            onDelete = { id -> vm.onIntent(NotesIntent.Delete(id)) }
        )
    }

    // One-shot effects from the VM
    LaunchedEffect(Unit) {
        vm.effects.collect { e ->
            when (e) {
                is NotesEffect.SyncFailed -> snackbarHostState.showSnackbar("Couldn't sync: ${e.message}")
                is NotesEffect.AuthExpired -> navController.navigate("auth")
            }
        }
    }
}
Three UI behaviors stacked:
  1. Offline banner — informative, not blocking. The user can still add/edit/delete; their changes apply locally and queue for sync.
  2. Notes list always renders from the cached Flow. Whether online or offline doesn't change reads at all.
  3. One-shot effects — sync failures surface as snackbars (auto-dismiss); auth expiry navigates to login. Both come via SharedFlow(replay=0) so a screen rotation doesn't replay yesterday's snackbars.
