KMP / CMP — theoretical knowledge

You don't need to have shipped KMP, but you should be able to articulate what it is, when to use it, and the tradeoffs. FOX likely won't deep-dive here.

Kotlin Multiplatform Compose Multiplatform commonMain / androidMain / iosMain expect / actual shared business logic, native UI shared UI with CMP Kotlin/Native + LLVM

1. What is KMP?

Kotlin Multiplatform lets you share Kotlin code across Android, iOS, web, desktop, and server. The Kotlin compiler targets JVM (Android/server), Native (iOS, macOS, Linux, Windows), JS, and Wasm.

Typical adoption pattern: share business logic + data + networking + DB; keep UI native. Android UI in Compose, iOS UI in SwiftUI; both call the same Kotlin :shared module.

Module layout

:shared                    (Kotlin Multiplatform module)
   commonMain/             — code shared everywhere
   commonTest/
   androidMain/            — Android-specific (uses Android APIs)
   iosMain/                — iOS-specific (uses iOS via Kotlin/Native)
   jvmMain/                — server / desktop JVM
:androidApp                (Android app — depends on :shared)
:iosApp                    (Xcode project — links the .framework from :shared)

expect / actual — the platform escape hatch

// commonMain
expect class Settings(name: String) {
    fun getString(key: String, default: String): String
    fun putString(key: String, value: String)
}

// androidMain
actual class Settings actual constructor(name: String) {
    private val prefs = appCtx.getSharedPreferences(name, Context.MODE_PRIVATE)
    actual fun getString(key: String, default: String) = prefs.getString(key, default) ?: default
    actual fun putString(key: String, value: String) { prefs.edit().putString(key, value).apply() }
}

// iosMain
actual class Settings actual constructor(name: String) {
    private val defaults = NSUserDefaults.standardUserDefaults
    actual fun getString(key: String, default: String) = defaults.stringForKey(key) ?: default
    actual fun putString(key: String, value: String) { defaults.setObject(value, key) }
}

2. What is CMP (Compose Multiplatform)?

Compose Multiplatform is JetBrains's port of Compose to other platforms — desktop (JVM), iOS, web. You write the UI once in Compose; CMP renders it on each target via the appropriate backend (Android Skia, iOS Skia, Desktop Skiko, Web Wasm/Skia).

As of 2024, CMP iOS is in stable. CMP Web is experimental.

Adoption shapes

ShapeWhat you share
KMP-onlyBusiness logic + data; native UI per platform
KMP + CMP-DesktopSame as above + desktop port
KMP + CMP-iOSBusiness + UI on Android & iOS — single Compose UI
KMP + CMP everywhereBusiness + UI everywhere (most ambitious)

4. When to use KMP — the senior framing

Locked-in answer

"KMP shines when you have significant duplicated business logic across Android and iOS — state machines, validation rules, networking layer, data models, sync engines, complex calculation logic. Sharing eliminates 'fixed on Android, still broken on iOS' bug drift. I wouldn't share UI unless the team is unified — sharing UI couples your release cadence and forces design compromises. The classic shape is KMP for business logic, native UI per platform."

6. Common interview probes

Q. Difference between KMP and CMP?

KMP = share Kotlin logic across platforms. CMP = share Compose UI across platforms. CMP runs on top of KMP — you can do KMP without CMP, but not the reverse for shared UI.

Q. Why not React Native or Flutter?

RN and Flutter use a non-native UI runtime (JS bridge / Skia). KMP+native UI gives fully native UX on each platform — best fit for OS conventions, performance, accessibility. CMP-iOS is also Skia-rendered (like Flutter) — accept that tradeoff if you choose it.

Q. How does Kotlin run on iOS?

Kotlin/Native compiles to LLVM bitcode → native ARM/x86 binary (or .framework that Xcode links). No JVM on iOS.

Q. What's the iOS framework integration like?

Build the shared module → produces a .framework (or .xcframework for multi-arch) → Xcode imports it. Swift calls Kotlin classes; Kotlin can call Objective-C / Swift via cinterop.

Q. Can you reuse Coroutines across platforms?

Yes — kotlinx.coroutines is multiplatform. On iOS you typically expose a Flow as a callback wrapper (e.g., via SKIE or a manual CFlow wrapper) since Swift can't natively consume Kotlin Flows.

7. If asked "have you shipped KMP?" and you haven't

Honest framing

"I haven't shipped a KMP module to production. I've evaluated it twice — once for a shared validation layer at Dish, once for a sync engine. We didn't move forward both times because the iOS team's Swift/RxSwift codebase wasn't ready to integrate. I follow the ecosystem closely — kotlinx.coroutines, kotlinx.serialization, Ktor, SQLDelight, Koin are the typical stack. For a greenfield product I'd seriously consider it for business logic; I'd avoid CMP-iOS until the team's iOS engineers are aligned."

8. Senior signal phrases

  • "Share business logic, keep UI native."
  • "expect/actual for platform escape hatches."
  • "Kotlin/Native compiles via LLVM to ARM .framework."
  • "Ktor + kotlinx.serialization + SQLDelight + Koin is the canonical KMP stack."
