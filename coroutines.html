Coroutines & Flows

The single deepest topic in modern Android. Structured concurrency, every dispatcher, every scope, exception handling, Flow operators, and how to test the whole stack.

1. The mental model

A coroutine is a suspendable computation. Not a thread — an instance of work that runs on a thread (managed by a Dispatcher) and can suspend cheaply (no thread blocking) at suspend points. The runtime parks the work and frees the thread for other coroutines.

Coroutine = a "lightweight thread" — a unit of asynchronous work that can pause and resume without holding a thread. Thousands run cheaply on a few real threads. Suspend function = a function that can pause; only callable from another suspend function or coroutine builder. Dispatcher = the thread-pool that actually runs your coroutine code.
The 4 pillars to recite

(1) Suspend functions — pause/resume without blocking. (2) Coroutine builders — launch, async, runBlocking. (3) Coroutine context — Job + Dispatcher + CoroutineExceptionHandler + CoroutineName. (4) Structured concurrency — parent/child relationships; failure and cancellation propagate.

2. Builders — launch vs async vs runBlocking

// launch — fire and forget, returns Job
val job: Job = scope.launch { /* ... */ }

// async — returns Deferred<T>; await() to get result
val deferred: Deferred<User> = scope.async { repo.fetch(id) }
val user = deferred.await()

// runBlocking — blocks the current thread; only for tests / main()
fun main() = runBlocking {
    val u = async { fetchUser() }
    println(u.await())
}
async exception handling — the trap

Exceptions in async are held in the Deferred until await(). They don't trigger a CoroutineExceptionHandler. Always await or wrap in try/catch. In a coroutineScope, the failure still propagates to siblings via the parent.

5. Structured concurrency in detail

// coroutineScope — child failure cancels siblings + propagates up
suspend fun loadDashboard(): Dashboard = coroutineScope {
    val a = async { api.userInfo() }
    val b = async { api.notifications() }
    val c = async { api.feed() }
    Dashboard(a.await(), b.await(), c.await())
}

// supervisorScope — children independent
suspend fun loadFeedAndAds(): Page = supervisorScope {
    val feed = async { api.feed() }
    val ads = async { api.ads() }
    val feedRes = feed.await()
    val adsRes = runCatching { ads.await() }.getOrDefault(emptyList())
    Page(feedRes, adsRes)
}
  1. coroutineScope { } — creates a child scope. async { } launches three concurrent coroutines. If any one fails, the others are cancelled and the exception propagates. The function returns only when all three complete.
  2. supervisorScope { } — same shape, but children are independent. If the ads API fails, the feed API keeps running. runCatching { ads.await() } catches the exception just for ads; the feed still loads. Use this when you want partial-success behavior.
Senior phrase

"Structured concurrency means a coroutine cannot outlive its scope — that's how I avoid the GlobalScope footguns I saw in pre-coroutine codebases. coroutineScope for atomic groups, supervisorScope for resilient parallel fetches."

6. Cancellation — cooperative

val job = scope.launch {
    while (isActive) {                  // check the cancellation flag
        compute()
        ensureActive()                  // alternative — throws CancellationException if cancelled
    }
}
job.cancel()
job.cancelAndJoin()                     // wait for clean shutdown

Cleanup with NonCancellable

scope.launch {
    try {
        // download / process
    } finally {
        withContext(NonCancellable) {
            db.persistProgress()        // must run even after cancel
        }
    }
}
Common cancellation bugs

(1) CPU loops that don't check isActive ignore cancellation. (2) try { ... } catch (e: Exception) { } — swallows CancellationException! Always rethrow if you must catch Throwable: if (e is CancellationException) throw e. Or catch specific exceptions only.

7. Exception handling — every flavor

Inside launch

val handler = CoroutineExceptionHandler { _, e -> Log.e("coro", "boom", e) }
val scope = CoroutineScope(SupervisorJob() + Dispatchers.Default + handler)
scope.launch { throw IOException("x") }    // caught by handler

Inside async

val deferred = scope.async { throw IOException("x") }
try { deferred.await() } catch (e: Exception) { /* caught here */ }
// CoroutineExceptionHandler does NOT catch async exceptions

Per-coroutine try/catch

scope.launch {
    try {
        api.fetch()
    } catch (e: IOException) {
        showError()
    } catch (e: CancellationException) {
        throw e   // rethrow!
    }
}
Senior framing

"CoroutineExceptionHandler only catches uncaught exceptions in launch. async defers them to await(). With a SupervisorJob, sibling coroutines aren't cancelled by one failure. I use handler as a global safety net for telemetry, with per-coroutine try/catch for recoverable errors."

8. Suspending utilities — withTimeout, withContext, yield

val r = withTimeoutOrNull(5_000) { api.slow() }   // null on timeout
val r2 = withTimeout(5_000) { api.slow() }         // throws TimeoutCancellationException

withContext(Dispatchers.IO) { /* shift dispatcher, no new coroutine */ }

repeat(1_000_000) {
    yield()        // play nice — suspend + check cancel
    process(it)
}

9. Mutex & Channel

val mutex = Mutex()
suspend fun safeAdd(item: T) = mutex.withLock { list.add(item) }   // coroutine-friendly lock

// Channel — producer/consumer
val ch = Channel<Int>(capacity = Channel.BUFFERED)
launch { for (i in 1..100) ch.send(i); ch.close() }
launch { for (item in ch) consume(item) }

For most state/event needs, prefer Flow over Channel. Channels shine for true producer/consumer with fan-out semantics.

10. Flows — cold streams

fun search(q: String): Flow<List<User>> = flow {
    emit(emptyList())                       // initial loading state
    val results = api.search(q)
    emit(results)
}.flowOn(Dispatchers.IO)                    // upstream runs on IO
 .catch { e -> emit(emptyList()) }          // upstream errors only
 .onStart { /* ... */ }
 .onCompletion { /* cleanup */ }

// StateFlow
private val _state = MutableStateFlow<UiState>(UiState.Loading)
val state: StateFlow<UiState> = _state.asStateFlow()
_state.update { it.copy(isLoading = true) }      // safe atomic update

// SharedFlow for one-shot events
private val _events = MutableSharedFlow<Event>(replay = 0, extraBufferCapacity = 1)
val events: SharedFlow<Event> = _events.asSharedFlow()
viewModelScope.launch { _events.emit(Event.Navigate("home")) }
Why isn't my SharedFlow event delivered?

Default has replay=0, extraBufferCapacity=0: if no collector is active, emit() suspends. For events, set extraBufferCapacity=1 with BufferOverflow.DROP_OLDEST and use tryEmit for fire-and-forget.

Why isn't my StateFlow update visible?

StateFlow uses structural equality for dedupe. If you re-set the same data class instance with the same content, no emission. Use update { it.copy(...) } with actual value changes, and ensure your data class's equals reflects all changing fields.

12. stateIn / shareIn — cold to hot

val users: StateFlow<List<User>> = repo.observeUsers()
    .stateIn(
        scope = viewModelScope,
        started = SharingStarted.WhileSubscribed(5_000),
        initialValue = emptyList()
    )

SharingStarted.WhileSubscribed(5000) — share while at least one collector; keep alive for 5s after last unsubscribe (handles config changes / brief navigation). The standard.

StrategyBehavior
Eagerlystart immediately when scope starts
Lazilystart on first collector, never stop
WhileSubscribed(timeout)start on first collector, stop after timeout post-last-unsubscribe

13. flatMapLatest — search-as-you-type

val results: StateFlow<List<User>> = queryFlow
    .debounce(300)
    .distinctUntilChanged()
    .flatMapLatest { q -> if (q.isBlank()) flowOf(emptyList()) else repo.search(q) }
    .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), emptyList())
Walk through what each operator does to the keystroke stream:
  1. queryFlow emits the current text every time the user types a character.
  2. debounce(300) waits 300ms — if no new keystroke arrives in that window, the latest value passes through. Filters out bursts of typing.
  3. distinctUntilChanged() — if the user typed and immediately deleted (same value as before), don't re-search.
  4. flatMapLatest { repo.search(q) } — for each query, start a search. If a new query arrives before the previous search finishes, cancel the previous and start the new one. Saves wasted API calls.
  5. stateIn(...) turns the cold Flow into a hot StateFlow with an empty initial value. WhileSubscribed(5000) keeps the upstream alive 5s after the last collector unsubscribes — so rotating the device doesn't restart the whole pipeline.

14. callbackFlow — bridge old APIs

fun locationUpdates(client: FusedLocationProviderClient): Flow<Location> = callbackFlow {
    val cb = object : LocationCallback() {
        override fun onLocationResult(result: LocationResult) {
            result.lastLocation?.let { trySend(it) }
        }
    }
    client.requestLocationUpdates(LocationRequest.create(), cb, Looper.getMainLooper())
    awaitClose { client.removeLocationUpdates(cb) }
}

15. Collecting Flow safely from UI

In Compose

// Lifecycle-aware: stops collecting in STOPPED
val state by vm.state.collectAsStateWithLifecycle()

In Activity/Fragment

lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        vm.state.collect { render(it) }
    }
}
Senior phrase

"Bare flow.collect in Activity is a resource leak waiting to happen — keeps the upstream alive in the background. repeatOnLifecycle(STARTED) in XML, collectAsStateWithLifecycle in Compose. Both stop collection when the screen leaves STARTED state."

18. Common interview probes — quick fire

Q. What's the difference between launch and async?

Both start a coroutine inside a scope. launch returns Job, fire-and-forget; uncaught exception propagates to parent immediately. async returns Deferred<T>; exception held until await(). Use async when you need a result; launch when you don't.

Q. What happens if a child fails inside coroutineScope?

The failure cancels all sibling children and propagates up to the scope, which rethrows from the suspending call. Use supervisorScope if you want siblings to keep running.

Q. Cold vs hot flow?

Cold (Flow): producer block runs per collector; no collector → no work. Hot (StateFlow, SharedFlow): emit independently of collectors; collectors see whatever is currently being emitted (or replay).

Q. Why WhileSubscribed(5000)?

Keeps the upstream alive 5s after the last collector unsubscribes — handles configuration change (rotate device) without re-fetching. Without this timeout, every rotation re-runs the upstream Flow.

Q. How do you make a CPU-heavy loop cancellable?

Call ensureActive() or yield() inside the loop. Suspending functions (e.g., delay) check cancellation automatically; pure CPU loops do not.

Q. How do you handle a 401 + token refresh in coroutines?

OkHttp Authenticator handles refresh + retry transparently. To dedupe concurrent 401s (multiple parallel calls hitting 401 at once), wrap the refresh in a Mutex.withLock so only one refresh happens.

Q. What's GlobalScope and when is it OK?

A scope tied to the application process lifetime. Almost never use it — it bypasses structured concurrency. Acceptable in a library's top-level entry point if explicitly app-scoped, but better to inject a custom CoroutineScope(SupervisorJob() + ...) created in the Application class.

Q. Why does my SharedFlow not get my events?

Default config: replay = 0, extraBufferCapacity = 0. With no collector, emit() suspends; tryEmit() drops. For events: MutableSharedFlow(replay = 0, extraBufferCapacity = 1, onBufferOverflow = DROP_OLDEST).

Q. Channel vs SharedFlow for one-shot events?

Both work. SharedFlow is the modern recommendation — better composability, less ceremony, integrates with Flow operators. Channel is fine for true producer/consumer queues with fan-out semantics, but for events from VM to UI use SharedFlow.

19. Senior signal phrases to drop naturally

  • Structured concurrency — say it when explaining why you don't use GlobalScope.
  • Cooperative cancellation — say it when explaining cleanup.
  • Conflation in StateFlow — say it when explaining why duplicates aren't emitted.
  • Backpressure via buffer/conflate/collectLatest — say it when discussing fast producers.
  • flowOn affects upstream only — say it when explaining dispatcher boundaries.
  • SharingStarted.WhileSubscribed(5000) — say the literal config; signals you've shipped this.
  • repeatOnLifecycle / collectAsStateWithLifecycle — say either; shows you don't naive-collect.
  • awaitClose in callbackFlow — say it when bridging callback APIs.
