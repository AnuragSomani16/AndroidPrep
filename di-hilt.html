Dependency Injection — Hilt

Every annotation, every component, qualifiers, multibinding, assisted injection, and the integration points (WorkManager, Compose, Navigation). Built on Dagger 2 — Hilt is opinionated wiring on top.

compile-time DI graph component hierarchy scoped vs unscoped @Binds beats @Provides for interfaces qualifier annotations multibinding (@IntoSet / @IntoMap) assisted injection SavedStateHandle wiring Hilt + WorkManager bridge ApplicationContext qualifier
Dependency Injection (DI) = a class receives its collaborators from outside (constructor / setter / field) rather than constructing them itself. Hilt = Google's DI library for Android, built on top of Dagger 2. It generates the boilerplate Dagger components for the standard Android lifecycle owners (Application, Activity, ViewModel, Fragment) so you don't have to. The DI graph is verified at compile time — missing bindings fail the build, not the app.

1. Why DI — and why Hilt over plain Dagger

Senior framing

"DI gives me decoupling (depend on interfaces), testability (swap real for fake), lifecycle-aware object graphs, and a single source of truth for how dependencies are constructed. Hilt is opinionated DI on top of Dagger — it generates the boilerplate Dagger components for the standard Android lifecycle owners (Application, Activity, ViewModel) so I get scoped graphs without writing them myself. Compile-time wiring means errors surface at build time, not runtime — unlike service locators or runtime-reflection DI."

@AndroidEntryPointActivity, Fragment, View, Service, BroadcastReceiverAllows field injection into that component
@HiltViewModelViewModelHilt provides this VM via the ViewModelComponent

4. The component (scope) hierarchy

SingletonComponent           — @Singleton                — application lifetime
  └─ ActivityRetainedComponent — @ActivityRetainedScoped  — survives config changes
        └─ ViewModelComponent  — @ViewModelScoped         — viewmodel lifetime
        └─ ActivityComponent   — @ActivityScoped          — activity lifetime
              └─ FragmentComponent — @FragmentScoped      — fragment lifetime
                    └─ ViewComponent — @ViewScoped        — view lifetime
              └─ ViewWithFragmentComponent
  └─ ServiceComponent          — @ServiceScoped           — service lifetime
Scope rule

"Wider scope can be injected into narrower; not vice versa. A @Singleton can be injected anywhere. A @FragmentScoped can NOT be injected into a @Singleton — its lifetime would be wrong."

5. The 4 module patterns you'll be quizzed on

Pattern 1 — @Provides for third-party / configured types

@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {

    @Provides
    @Singleton
    fun provideJson() = Json {
        ignoreUnknownKeys = true
        coerceInputValues = true
    }

    @Provides
    @Singleton
    fun provideOkHttp(authInterceptor: AuthInterceptor): OkHttpClient =
        OkHttpClient.Builder()
            .addInterceptor(authInterceptor)
            .addInterceptor(HttpLoggingInterceptor().apply {
                level = if (BuildConfig.DEBUG) HttpLoggingInterceptor.Level.BODY
                        else HttpLoggingInterceptor.Level.NONE
            })
            .build()

    @Provides
    @Singleton
    fun provideRetrofit(client: OkHttpClient, json: Json): Retrofit =
        Retrofit.Builder()
            .baseUrl("https://api.example.com/")
            .client(client)
            .addConverterFactory(json.asConverterFactory("application/json".toMediaType()))
            .build()

    @Provides @Singleton fun userApi(r: Retrofit): UserApi = r.create()
}
  1. @Module — declares this object holds bindings. @InstallIn(SingletonComponent::class) — these bindings live for the application lifetime.
  2. @Provides on each function — Hilt calls these functions to create the bound type. Parameters of the function are themselves resolved from the graph (Hilt finds OkHttpClient from another binding to give to provideRetrofit).
  3. @Singleton — the result is cached for the lifetime of the SingletonComponent. Without it, every injection would call the function again.
  4. Order doesn't matter — Hilt's compile-time resolver figures out construction order from the dependency DAG. If you forget to provide one of the params (e.g., AuthInterceptor), the build fails with a clear "missing binding" error.

Pattern 2 — @Binds for interface → impl

@Module
@InstallIn(SingletonComponent::class)
abstract class RepoModule {
    @Binds @Singleton
    abstract fun bindUserRepo(impl: UserRepoImpl): UserRepo

    @Binds
    abstract fun bindNotesRepo(impl: NotesRepoImpl): NotesRepo
}
@Binds vs @Provides

For "interface → impl with no construction logic", @Binds is more efficient — Dagger can skip generating a factory class. Use @Provides when you need to construct the instance yourself or it's a third-party type.

Pattern 3 — Qualifiers (multiple bindings of same type)

@Qualifier @Retention(AnnotationRetention.BINARY) annotation class IoDispatcher
@Qualifier @Retention(AnnotationRetention.BINARY) annotation class MainDispatcher
@Qualifier @Retention(AnnotationRetention.BINARY) annotation class DefaultDispatcher

@Module
@InstallIn(SingletonComponent::class)
object DispatchersModule {
    @IoDispatcher      @Provides fun io(): CoroutineDispatcher = Dispatchers.IO
    @MainDispatcher    @Provides fun main(): CoroutineDispatcher = Dispatchers.Main
    @DefaultDispatcher @Provides fun default(): CoroutineDispatcher = Dispatchers.Default
}

class Repo @Inject constructor(
    @IoDispatcher private val io: CoroutineDispatcher
) { /* ... */ }

Pattern 4 — Multibinding (collect impls)

interface AnalyticsTracker { fun screen(name: String) }

@Module
@InstallIn(SingletonComponent::class)
abstract class AnalyticsModule {
    @Binds @IntoSet abstract fun newRelic(impl: NewRelicTracker): AnalyticsTracker
    @Binds @IntoSet abstract fun crashlytics(impl: CrashlyticsTracker): AnalyticsTracker
    @Binds @IntoSet abstract fun internal(impl: InternalTracker): AnalyticsTracker
}

class AnalyticsHub @Inject constructor(
    private val trackers: Set<@JvmSuppressWildcards AnalyticsTracker>
) : AnalyticsTracker {
    override fun screen(name: String) = trackers.forEach { runCatching { it.screen(name) } }
}

Map multibinding

@Module
@InstallIn(SingletonComponent::class)
abstract class FeatureRoutesModule {
    @Binds @IntoMap @StringKey("home")
    abstract fun home(impl: HomeRouter): Router
    @Binds @IntoMap @StringKey("player")
    abstract fun player(impl: PlayerRouter): Router
}
// inject Map<String, @JvmSuppressWildcards Router>

6. Assisted injection — runtime + DI args

class UserDetailsVm @AssistedInject constructor(
    private val repo: UserRepo,
    @Assisted val userId: String
) : ViewModel() {
    @AssistedFactory
    interface Factory {
        fun create(userId: String): UserDetailsVm
    }
}

// Use it
class UserActivity : AppCompatActivity() {
    @Inject lateinit var vmFactory: UserDetailsVm.Factory
    private val vm: UserDetailsVm by viewModels {
        viewModelFactory {
            initializer { vmFactory.create(intent.getStringExtra("id")!!) }
        }
    }
}
For ViewModels with route args, prefer SavedStateHandle

If you're on Navigation Compose, just inject SavedStateHandle into a normal @HiltViewModel — Hilt + Nav auto-wires the route args. Use Assisted only when the runtime arg can't go through SavedStateHandle.

7. Hilt + ViewModel + Navigation Compose

@HiltViewModel
class NoteDetailVm @Inject constructor(
    private val savedState: SavedStateHandle,
    private val getNote: GetNoteUseCase
) : ViewModel() {
    private val noteId: String = checkNotNull(savedState["id"])
    val state: StateFlow<NoteDetailState> = flow { emit(getNote(noteId)) }
        .map { NoteDetailState.Success(it) as NoteDetailState }
        .catch { emit(NoteDetailState.Error(it.message ?: "")) }
        .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5000), NoteDetailState.Loading)
}

// In a NavHost:
composable("note/{id}") { entry ->
    val vm: NoteDetailVm = hiltViewModel()  // SavedStateHandle has "id" automatically
    NoteDetailScreen(vm)
}

8. Hilt + WorkManager

@HiltWorker
class SyncWorker @AssistedInject constructor(
    @Assisted ctx: Context,
    @Assisted params: WorkerParameters,
    private val repo: NotesRepository
) : CoroutineWorker(ctx, params) {
    override suspend fun doWork(): Result =
        repo.refresh().fold(
            onSuccess = { Result.success() },
            onFailure = { Result.retry() }
        )
}

// Application
@HiltAndroidApp
class App : Application(), Configuration.Provider {
    @Inject lateinit var workerFactory: HiltWorkerFactory
    override val workManagerConfiguration: Configuration
        get() = Configuration.Builder()
            .setWorkerFactory(workerFactory)
            .setMinimumLoggingLevel(android.util.Log.INFO)
            .build()
}

// AndroidManifest — disable default initializer
<provider
    android:name="androidx.startup.InitializationProvider"
    android:authorities="${applicationId}.androidx-startup"
    tools:node="merge">
    <meta-data
        android:name="androidx.work.WorkManagerInitializer"
        android:value="androidx.startup"
        tools:node="remove" />
</provider>

10. @EntryPoint — for non-Hilt callers

// You have a class that's not @AndroidEntryPoint (e.g., a third-party callback)
// and you need a binding from the SingletonComponent.

@EntryPoint
@InstallIn(SingletonComponent::class)
interface AnalyticsEntryPoint {
    fun analytics(): AnalyticsHub
}

class LegacyCallback(private val appContext: Context) {
    private val analytics = EntryPointAccessors
        .fromApplication(appContext, AnalyticsEntryPoint::class.java)
        .analytics()
}

11. Common Hilt errors and fixes

"MissingBinding: cannot be provided without an @Inject constructor or @Provides-annotated method"

The class isn't @Inject constructor-annotated and there's no @Provides/@Binds for it. Add one. If it's a third-party type, write a @Provides in a module installed in the right component.

"Cannot be injected into @Singleton because it's @ActivityScoped"

Wider scope can't depend on narrower. Either widen the dependency to @Singleton, or move the consumer to a narrower scope.

"Duplicate bindings"

Two @Provides for the same type with no qualifiers. Add a @Qualifier to disambiguate, or remove the duplicate.

"@HiltAndroidApp not found at compile time"

You forgot to set android:name=".App" in AndroidManifest.xml, or your Application class isn't annotated.

"Hilt component for AndroidEntryPoint requires a parent component"

You annotated a Fragment with @AndroidEntryPoint, but its hosting Activity is missing the same annotation. Annotate the Activity too.

12. Best practices — what seniors do differently

  • Inject interfaces, not impls. val repo: NotesRepo, not val repo: NotesRepoImpl. Easier to fake in tests.
  • Inject coroutine dispatchers. Don't hardcode Dispatchers.IO. Inject @IoDispatcher so tests swap a StandardTestDispatcher.
  • Avoid @Singleton reflexively. Only use it when the object truly should live for the whole app. Otherwise scope appropriately.
  • Modules per concern: NetworkModule, DatabaseModule, RepoModule, DispatchersModule. Don't put everything in one giant module.
  • Use @Binds for interface → impl. More efficient than @Provides for that case.
  • Use multibinding for plugin-style architectures (analytics trackers, feature flag handlers, deep link routers).
  • Don't field-inject when constructor injection works. Field injection (@Inject lateinit var) is for Android components you can't construct yourself (Activity, Fragment); everywhere else use constructor injection.
  • Test modules: replace production modules in tests with @TestInstallIn(replaces = [...]).

14. Hilt vs Koin (you may be asked)

Hilt (Dagger)Koin
WiringCompile-timeRuntime (service locator)
ErrorsBuild-timeApp-launch / first-access
SetupMore boilerplateLess, just Kotlin DSL
PerformanceZero runtime costSlight startup + per-resolution cost
Recommended forLarge apps, multi-module, official Google guidanceSmall apps, KMP (Hilt is Android-only)
Senior phrase

"For Android single-platform, I default to Hilt — compile-time DI graph means missing bindings fail the build, not the app. For KMP I'd consider Koin since Hilt is Android-only."
