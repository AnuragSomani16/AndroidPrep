Android SDK Core

The foundation: Activity / Fragment lifecycles, every launch mode, Services, Broadcast Receivers, Content Providers, Intents, and the Application class.

configuration change vs process death back stack mutation singleTop deduplication singleTask task affinity foreground vs background service dynamic vs manifest receivers LocalBroadcastManager (deprecated) Intent vs PendingIntent FLAG_IMMUTABLE on Android 12+ doze mode + app standby
Activity = a single screen with its own lifecycle (onCreate → onResume → onPause → onDestroy). Fragment = a reusable UI piece embedded in an Activity, with a separate view lifecycle. Service = long-running background work without UI (foreground for user-visible, bound for IPC). BroadcastReceiver = listener for system or app-wide events (boot complete, network change, custom). ContentProvider = a data interface other apps can query. Intent = a message describing an action ("open this URL", "share this text"); explicit (you name the target) or implicit (the system picks).

1. The four core components

ComponentUse for
ActivityUI screen with lifecycle
ServiceLong-running background work without UI
Broadcast ReceiverReact to system or app-wide events
Content ProviderCross-app data sharing (rarely used in modern apps)

2. Activity lifecycle — full cycle

onCreate    — view binding, ViewModel attach. Bundle for process-death restore.
onStart     — Activity becomes visible (but not interactive yet).
onResume    — Activity is in foreground, receiving input.
            ─── (USER INTERACTS) ───
onPause     — losing focus (dialog, multi-window, going home). Save lightweight state.
onStop      — no longer visible. Heavier teardown OK.
onRestart   — back from stopped (returning to this activity).
onDestroy   — final cleanup. Always pairs with onCreate IF not killed unexpectedly.

State diagram (simplified)

onCreate → onStart → onResume ↔ (foreground)
                     ↓
                 onPause → onStop
                     ↓        ↓
                 onResume   onRestart → onStart → onResume
                     ↓
                 onDestroy

Configuration change vs process death

TriggerWhat happensWhat survives
Rotation, font scale, theme switchActivity recreatedViewModel survives. onSaveInstanceState Bundle restored.
Process death (background, low memory)Process killed; on return, Activity recreatedViewModel does NOT survive. Only the onSaveInstanceState Bundle / SavedStateHandle survives.
User backs outActivity destroyed for goodNothing.
 
class MainActivity : ComponentActivity() {
    private val vm: HomeVm by viewModels()           // survives config change

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        // savedInstanceState is non-null on rebirth (rotation OR process death)
        // For rotation, vm has data. For process death, only the Bundle.
        setContent { HomeScreen(vm) }
    }

    override fun onSaveInstanceState(outState: Bundle) {
        super.onSaveInstanceState(outState)
        outState.putString("scrollKey", lastVisibleId)
    }
}
Senior phrase

"For state I'd be sad to lose I use SavedStateHandle in the ViewModel — survives both config change AND process death. ViewModel-only state survives rotation but not background-kill. UI-only state (text input, scroll position) goes in rememberSaveable."

3. Fragment lifecycle

onAttach        — fragment attached to host activity
onCreate        — instance created (before view)
onCreateView    — inflate the view
onViewCreated   — view ready, bind UI
onStart         — visible
onResume        — foreground
onPause
onStop
onDestroyView   — view destroyed (fragment may live on, e.g., in back stack)
onDestroy       — fragment destroyed
onDetach        — detached from activity
The view-vs-fragment lifecycle gotcha

A fragment can outlive its view (back stack). Use viewLifecycleOwner for UI-bound observers, not this. Otherwise observers leak when the view is destroyed but the fragment persists.

override fun onViewCreated(v: View, s: Bundle?) {
    viewLifecycleOwner.lifecycleScope.launch {
        repeatOnLifecycle(Lifecycle.State.STARTED) {
            vm.state.collect { render(it) }
        }
    }
}

4. Launch modes — every one with concrete example

Set in AndroidManifest android:launchMode or via Intent flags. Affects how the activity is added to the task back stack.

ModeBehaviorUse case
standard(default) Always create a new instance. Stack: A → A → A allowed.Most screens.
singleTopIf activity is already at the top of the stack, reuse it (fires onNewIntent). Else create new.Notification → screen that's already on top (avoid stacking duplicates).
singleTaskSingle instance across the task. If exists anywhere in the stack, bring to top + clear above. Fires onNewIntent.Home / launcher activity. Browser entry point.
singleInstanceLike singleTask but lives alone in its own task. No other activity can join its task.System dialogs, lock screen — almost never used in apps.

Concrete scenarios

// Scenario: tap a notification when the same screen is already on top.
// AndroidManifest:
<activity android:name=".DetailActivity" android:launchMode="singleTop" />

// In DetailActivity, override onNewIntent to update the existing instance:
override fun onNewIntent(intent: Intent) {
    super.onNewIntent(intent)
    setIntent(intent)        // important — getIntent() will return new one
    val newId = intent.getStringExtra("id")
    vm.load(newId)
}

Intent flags — runtime equivalents

// FLAG_ACTIVITY_SINGLE_TOP — same as singleTop just for this navigation
intent.addFlags(Intent.FLAG_ACTIVITY_SINGLE_TOP)

// FLAG_ACTIVITY_CLEAR_TOP — pop everything above the existing instance of this activity
intent.addFlags(Intent.FLAG_ACTIVITY_CLEAR_TOP)

// Common combo: "go back to home, clearing stack"
intent.flags = Intent.FLAG_ACTIVITY_NEW_TASK or Intent.FLAG_ACTIVITY_CLEAR_TASK

"Trace the stack" drill — what's on the stack?

Q. App opens A → B → C. Tap notification that opens A again with android:launchMode="singleTask". What's on the stack?

Stack: A. singleTask brings A to top and clears everything above it (B, C destroyed). onNewIntent fires on A.

Q. Same but singleTop?

Stack: A → B → C → A. singleTop only dedupes if already at the top. Since C is on top, a new A is pushed. (No onNewIntent; new onCreate.)

Q. With FLAG_ACTIVITY_CLEAR_TOP and standard launch mode, what's on the stack?

Stack: A. CLEAR_TOP pops B and C. With standard, A is also recreated by default — unless you also set FLAG_ACTIVITY_SINGLE_TOP or A's launch mode is singleTop/singleTask, in which case onNewIntent fires on the existing A.

5. Tasks & back stack — the senior framing

A task is a stack of activities the user navigates through. Each task has its own back-stack. Tasks are isolated — pressing back in one doesn't pop another. Multi-window or recents shows tasks, not activities.

Task affinity (manifest android:taskAffinity) — by default, all your activities share an affinity (your package). singleTask/FLAG_ACTIVITY_NEW_TASK uses affinity to decide which task to use.

6. Services — foreground / background / bound

Service typeLifecycleUse for
Foreground servicePersistent notification; OS keeps the process aliveMusic playback, location tracking, video processing
Background serviceHidden; killed eagerly by OS on Android 8+ (heavily restricted)Almost never — use WorkManager instead
Bound serviceLives only while clients are boundIPC, exposing API to other components

Foreground service for media playback

class PlaybackService : MediaSessionService() {     // Media3 type
    private lateinit var session: MediaSession

    override fun onCreate() {
        super.onCreate()
        val player = ExoPlayer.Builder(this).build()
        session = MediaSession.Builder(this, player).build()
    }

    override fun onGetSession(controllerInfo: MediaSession.ControllerInfo): MediaSession = session

    override fun onDestroy() {
        session.release()
        super.onDestroy()
    }
}

// AndroidManifest
<service
    android:name=".PlaybackService"
    android:exported="true"
    android:foregroundServiceType="mediaPlayback">
    <intent-filter>
        <action android:name="androidx.media3.session.MediaSessionService"/>
    </intent-filter>
</service>
Background service restrictions (Android 8+ / Oreo)

If your app is in the background, you cannot start a regular Service — the OS will throw IllegalStateException. Use: (1) WorkManager for deferrable work, (2) Foreground service with notification for user-visible long work, (3) JobIntentService (deprecated; use WorkManager).

7. Broadcast Receivers

Manifest-declared (statically registered)

<receiver android:name=".BootReceiver" android:exported="true">
    <intent-filter>
        <action android:name="android.intent.action.BOOT_COMPLETED" />
    </intent-filter>
</receiver>

Useful for system-wide events. Heavily restricted on Android 8+ — most implicit broadcasts can't be received via manifest anymore (e.g., CONNECTIVITY_CHANGE).

Dynamically registered

val receiver = object : BroadcastReceiver() {
    override fun onReceive(ctx: Context, intent: Intent) {
        if (intent.action == ConnectivityManager.CONNECTIVITY_ACTION) {
            handleConnectivityChange()
        }
    }
}
ContextCompat.registerReceiver(
    ctx, receiver,
    IntentFilter(ConnectivityManager.CONNECTIVITY_ACTION),
    ContextCompat.RECEIVER_NOT_EXPORTED
)
// don't forget to unregister in onStop / onDestroy
Modern alternatives

For app-internal events: SharedFlow / EventBus / DI singleton. Old code used LocalBroadcastManager — deprecated. For network state: ConnectivityManager.NetworkCallback. For lifecycle events: ProcessLifecycleOwner.

8. Content Providers (briefly)

Used to share data across app boundaries (e.g., ContactsContract, MediaStore). Modern apps rarely create their own. Common consumer use:

val cursor = ctx.contentResolver.query(
    MediaStore.Images.Media.EXTERNAL_CONTENT_URI,
    arrayOf(MediaStore.Images.Media._ID, MediaStore.Images.Media.DISPLAY_NAME),
    null, null,
    "${MediaStore.Images.Media.DATE_ADDED} DESC"
)

If asked: "I'd reach for a Content Provider only if I needed to expose data to other apps. Within my app, I use Room or DataStore."

9. Intents — explicit, implicit, and PendingIntent

Explicit Intent (you know the target)

val i = Intent(ctx, DetailActivity::class.java).apply {
    putExtra("id", noteId)
}
ctx.startActivity(i)

Implicit Intent (system picks the handler)

val share = Intent(Intent.ACTION_SEND).apply {
    type = "text/plain"
    putExtra(Intent.EXTRA_TEXT, "Check this out: $url")
}
ctx.startActivity(Intent.createChooser(share, "Share via"))

PendingIntent — for notifications, alarms, widgets

val pi = PendingIntent.getActivity(
    ctx, 0,
    Intent(ctx, DetailActivity::class.java).apply { putExtra("id", id) },
    PendingIntent.FLAG_UPDATE_CURRENT or PendingIntent.FLAG_IMMUTABLE   // IMMUTABLE required Android 12+
)
FLAG_IMMUTABLE — Android 12+

Required on API 31+. Prevents another app from modifying the wrapped intent. FLAG_MUTABLE only when you genuinely need it (e.g., for inline notification reply where the system fills in the result).

10. The Application class

@HiltAndroidApp
class App : Application(), Configuration.Provider {
    @Inject lateinit var workerFactory: HiltWorkerFactory
    @Inject lateinit var crashReporter: CrashReporter

    override fun onCreate() {
        super.onCreate()
        // Keep this method LIGHT — every ms here is cold start time
        crashReporter.install()
        // Defer heavier init to App Startup library or post-first-frame
    }

    override val workManagerConfiguration: Configuration
        get() = Configuration.Builder()
            .setWorkerFactory(workerFactory)
            .setMinimumLoggingLevel(android.util.Log.INFO)
            .build()
}

App Startup library — defer init

class TelemetryInitializer : Initializer<Telemetry> {
    override fun create(ctx: Context): Telemetry = Telemetry(ctx)

<provider

    android:exported="false"
    tools:node="merge">
    <meta-data
        android:name="com.x.TelemetryInitializer"
        android:value="androidx.startup" />
</provider>

11. Process lifecycle (Android & you)

Your app's process can be killed at any time when in the background. Priority order:

  1. Foreground (visible Activity, foreground service) — protected.
  2. Visible (Activity behind a translucent dialog) — protected.
  3. Service running — moderate protection.
  4. Background activity — low protection.
  5. Empty — first to be killed.

ProcessLifecycleOwner

class App : Application(), DefaultLifecycleObserver {
    override fun onCreate() {
        super.onCreate()
        ProcessLifecycleOwner.get().lifecycle.addObserver(this)
    }
    override fun onStart(o: LifecycleOwner) { /* app foregrounded */ }
    override fun onStop(o: LifecycleOwner) { /* app backgrounded */ }
}

12. Doze mode & App Standby

  • Doze — device unplugged + screen off + stationary → OS defers background CPU/network. Wakes briefly periodically. Foreground services + high-priority FCM bypass.
  • App Standby — apps not used recently get bucketed (active / working set / frequent / rare / restricted). Lower buckets → tighter restrictions on jobs and alarms.

Implication for you: WorkManager handles this. Avoid setExactAndAllowWhileIdle alarms unless you genuinely need wall-clock precision.

13. Permissions — runtime & install-time

// Compose runtime permission (Accompanist or AndroidX)
val permission = rememberPermissionState(Manifest.permission.CAMERA)

LaunchedEffect(Unit) {
    if (!permission.status.isGranted) permission.launchPermissionRequest()
}

when {
    permission.status.isGranted -> CameraScreen()
    permission.status.shouldShowRationale -> RationaleScreen { permission.launchPermissionRequest() }
    else -> PermissionDeniedScreen()
}

"Special" permissions to be aware of

  • POST_NOTIFICATIONS — runtime on Android 13+.
  • SCHEDULE_EXACT_ALARM — Android 12+; needs special grant via Settings.
  • FOREGROUND_SERVICE_* — typed foreground service permissions on Android 14+.
  • READ_MEDIA_IMAGES / VIDEO / AUDIO — replaces READ_EXTERNAL_STORAGE on Android 13+.

14. Common gotchas

Q. My BroadcastReceiver runs but onReceive is "too slow."

onReceive runs on main; you have ~10 seconds before ANR. For longer work: enqueue a WorkManager job from onReceive, or use goAsync() + finish() after work completes.

Q. My Service is being killed even though I started it.

Background service restrictions on Android 8+. Convert to foreground service with a notification, or move work to WorkManager.

Q. My PendingIntent throws on Android 12+.

Add PendingIntent.FLAG_IMMUTABLE (or FLAG_MUTABLE if you really need mutable). API 31+ requires explicit choice.

Q. Activity recreated unexpectedly.

Either configuration change (rotation, dark mode, font scale, multi-window) or process death restoration. Handle with ViewModel + SavedStateHandle. Don't override configChanges in manifest unless you really know what you're doing — it's a band-aid that often hides bugs.
