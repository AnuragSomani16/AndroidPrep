Recent Releases — Android & Compose

What changed in the last three Android platform releases (14 / 15 / 16) and the recent Compose BOMs. The "what does this mean in practice for a senior" framing — drop these in TR3/TR4 to signal you keep current.

Verify before quoting numbers

Specific API names, deprecation dates, and behavior gates can shift between betas and stable releases. Use this as a structural map; for an authoritative current state, check developer.android.com/about/versions and the Compose release notes.

Android 14 (API 34) — released October 2023

Theme of the release: tightening foreground service rules, granular media access, and predictive back as a first-class developer feature.

Foreground service types — now mandatory

Apps targeting API 34 must declare a foregroundServiceType for every startForeground() call. Examples: mediaPlayback, location, dataSync, connectedDevice, shortService, specialUse. Each type has its own permissions and runtime restrictions.

// AndroidManifest
<service android:name=".PlaybackService"
    android:foregroundServiceType="mediaPlayback" />

<uses-permission android:name="android.permission.FOREGROUND_SERVICE" />
<uses-permission android:name="android.permission.FOREGROUND_SERVICE_MEDIA_PLAYBACK" />

What this means: a music app needs FOREGROUND_SERVICE_MEDIA_PLAYBACK; misclassifying or omitting throws SecurityException. Audit every startForeground call.

Partial Photo Picker access

Users can grant access to a subset of photos rather than the whole library. New READ_MEDIA_VISUAL_USER_SELECTED permission flow.

What this means: avoid demanding full media access — use the Photo Picker system component, which doesn't need a permission at all and respects user choice automatically.

Predictive back gesture — opt in

Users see a peek of the underlying screen as they swipe back. Apps opt in via android:enableOnBackInvokedCallback="true". The legacy OnBackPressedCallback works alongside but doesn't show the predictive animation.

Granular media permissions (carried from 13)

READ_MEDIA_IMAGES, READ_MEDIA_VIDEO, READ_MEDIA_AUDIO instead of one READ_EXTERNAL_STORAGE.

Other notable A14 items

  • Per-app language preferences — production stable; users override the system locale per app.
  • OpenJDK 17 — apps can use Java 17 source-level features.
  • App store rules — apps installed from store must support API 23+; lower targets blocked.
  • Health Connect as part of the platform (no separate APK needed).
  • Cross-profile contacts — work-profile contact merge.

Android 15 (API 35) — released October 2024

Theme of the release: edge-to-edge becomes default, predictive back becomes default, and stricter rules for apps that target API 35.

Edge-to-edge enforced

Apps targeting API 35 are edge-to-edge by default. android:fitsSystemWindows stops being honored automatically; the system bars don't insets your content unless you handle it.

What this means: every Activity must use WindowCompat.setDecorFitsSystemWindows(window, false) (already implicit on API 35) and apply WindowInsets on its root layout / Compose tree (Modifier.systemBarsPadding(), Modifier.safeDrawingPadding(), etc.). Forget this and your content goes under the status/nav bars.

// Compose — handle insets explicitly
Column(modifier = Modifier
    .fillMaxSize()
    .systemBarsPadding()      // pad both top + bottom system bars
    .imePadding()              // pad the keyboard when it's up
) { /* content */ }

Predictive back — default for new targets

Apps targeting API 35 get predictive back animations without opt-in. The android:enableOnBackInvokedCallback="true" manifest flag is the new default for these apps.

16 KB page size support

Android 15 supports devices with 16 KB memory page sizes (vs. the historical 4 KB). Native libraries (.so) must be aligned to 16 KB or they fail to load on these devices.

What this means: if you ship native code (NDK), rebuild with the AGP / NDK that produces 16 KB-aligned libs. Pure Kotlin/Java apps are unaffected.

Partial Screen Sharing

New MediaProjection APIs let users share a single app instead of the whole screen. Better privacy, easier integration.

In-app camera controls

Camera2 / CameraX gain low-light boost, in-app extension control, more precise exposure controls. Useful if your app does any camera capture.

Other notable A15 items

  • App archiving via the system package installer — Play can offload unused apps.
  • Notification cooldown — system attenuates noisy notification streams.
  • Stronger app signature requirements for v3.1+ block downgrade attacks.
  • Foreground service launch restrictions tightened — even dataSync has stricter rules around starting from background.
  • Selectable Text in Compose stabilizations (SelectionContainer performance).

Sample edge-to-edge migration

// Before (Android < 15) — content fits below status bar by default
class MainActivity : ComponentActivity() {
    override fun onCreate(b: Bundle?) {
        super.onCreate(b)
        setContent { MyApp() }
    }
}

// After (Android 15 target) — must explicitly handle insets
class MainActivity : ComponentActivity() {
    override fun onCreate(b: Bundle?) {
        super.onCreate(b)
        WindowCompat.setDecorFitsSystemWindows(window, false)  // explicit, even though it's the default on 35
        setContent {
            Box(Modifier.fillMaxSize().systemBarsPadding()) {
                MyApp()
            }
        }
    }
}

Android 16 (API 36) — released 2025

Theme of the release: Material 3 Expressive — a new emphasis on motion, shape, and personality across the system. Plus continued tightening of background restrictions and adaptive performance APIs.

Material 3 Expressive

An expanded design language with more motion, larger shape variation, and richer typography hierarchies. Components like FloatingActionButton, NavigationBar, and dialogs gain emphasis variants.

What this means: Material 3 stays compatible — the Expressive variants are additive. You'll see new APIs in androidx.compose.material3 for motion-rich containers and shape-shifting components.

Adaptive Refresh Rate

System more aggressively chooses display refresh rate based on content. Apps that animate at fixed rates can hint via Surface / SurfaceView APIs.

What this means: for video / scrolling apps, smoother on high-refresh devices; potentially more battery savings on static screens.

PdfRenderer enhancements

Stronger PDF rendering primitives in the platform, including form-field handling and better text extraction.

Foreground service restrictions continued

Each cycle tightens the rules around when an app can start a foreground service from the background. By API 36, apps must be in a "valid" state (recent user interaction, FCM, etc.) to start most service types. Plan WorkManager-first.

Other notable A16 items

  • Adaptive layouts — better support for foldables and large screens via window size class APIs.
  • Improved auto-fill for Compose — system can fill text fields and credit-card inputs more reliably.
  • Accessibility — more granular semantics for Compose screens; system actions for dynamic content.

Compose BOM — recent highlights

BOM (Bill of Materials) = a meta-package that locks all Compose artifacts to consistent versions. Bumping the BOM brings the whole stack forward together.

Compose BOM 2024.10 (Compose 1.7.x) — October 2024

  • Strong Skipping Mode is now the default — Compose Compiler treats more types as stable automatically (including types with stable val fields). Means less need for @Stable / @Immutable annotations; recomposition skipping is more aggressive.
  • Drag-and-drop API stable — system-wide drag-and-drop with a Compose-friendly API.
  • LookaheadScope improvements — for shared element transitions and bounds-aware layouts.
  • Modifier.animateBounds() — implicit animation of size/position changes.
  • Improved nested-scroll performance.
  • AnimatedContent with default contentTransform — less ceremony for simple transitions.

Compose BOM 2024.12 — December 2024

  • ContextualFlowRow / ContextualFlowColumn — flow layouts that can show "+N more" overflow indicators.
  • Performance regression fixes from 1.7.
  • Pager stability improvements.

Compose BOM 2025.x (Compose 1.8) — early 2025

  • Auto-fill first-class API for Compose text fields.
  • Adaptive layouts library stabilization — supports list-detail, supporting-pane, three-pane layouts that adapt to window size class.
  • Performance — further reductions in recomposition cost; smaller bytecode generated by the compiler.
  • Compose UI testing improvements — better semantics handling, faster test execution.

What this means for senior interviews

Drop these naturally
  • "Targeting Android 15 means I have to handle edge-to-edge by default — explicit systemBarsPadding on roots."
  • "Foreground services need typed declarations since 14, and starting them from background is increasingly restricted — I default to WorkManager."
  • "Strong Skipping Mode in Compose 1.7 reduces the need for stability annotations — I still mark @Immutable for clarity but the compiler is smarter now."
  • "16 KB page size support is a quiet but important Android 15 thing — if we ship native libs we have to verify alignment."
  • "Predictive back is default in API 35 targets — old onBackPressed overrides should migrate to OnBackPressedCallback."

Migration checklist when bumping targetSdk

  1. Audit every startForeground() — declare types, request matching permissions.
  2. Apply edge-to-edge insets to all root layouts / Compose trees.
  3. Migrate onBackPressed to OnBackPressedCallback + enableOnBackInvokedCallback.
  4. If shipping NDK, rebuild for 16 KB page size.
  5. Replace READ_EXTERNAL_STORAGE with granular media permissions or Photo Picker.
  6. Test foreground service start from each app state (foreground / recent / cold).
  7. Re-record Baseline Profiles — code paths that JIT'd may have changed.
  8. Run Compose Compiler reports — Strong Skipping may surface or hide previous issues.
