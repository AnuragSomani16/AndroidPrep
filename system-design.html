HLD / LLD / Architectures

A senior's 6-step framework, plus full walk-throughs: Tubi-style streaming app, Image loader (Glide-clone), Offline notes, Chat client, Analytics in Compose & XML, Full playlist DB, Push pipeline, Photo upload.

clarify scope first functional vs non-functional data model + API contract tradeoff articulation dependency inversion at boundaries idempotency keys conflict resolution cursor pagination backpressure on producers SDUI for layout flexibility

1. The 6-step framework — never skip step 1

  1. Clarify scope — "When you say 'X', do you mean Y or Z? Scale? Online/offline? Platforms?" (5–8 minutes)
  2. Functional + non-functional reqs — features, latency, offline, scale, accessibility.
  3. Data model + API contract — what does the input/output look like.
  4. Sketch the architecture — boxes, modules, data flow.
  5. Deep dive — pick the part the interviewer cares about, go 3 layers deep on tradeoffs.
  6. Iterate — "what if X fails?" / "how to scale 10×?"
Senior phrase to open every design

"Before I start drawing, can I clarify a few things? It'll change the design significantly: (1) What's the scale? (2) Online-only or offline-capable? (3) Is real-time required? (4) Platforms — phone only, or tablet/TV/web? (5) Auth — anonymous, social, full account?"

2. Mobile-specific dimensions to clarify

  • Online-only or offline-first? Offline-first changes everything (Room as source of truth, sync engine, conflict resolution).
  • Real-time? WebSocket / SSE / FCM push.
  • Multi-device sync? Phone + tablet + TV — what's authoritative?
  • Pagination? Cursor (preferred) vs page-number.
  • Caching? Cache-aside, read-through, write-through, write-behind.
  • Auth? OAuth (PKCE on mobile), refresh rotation, session expiry.
  • Background work? WorkManager, foreground service, push.

3. HLD — Tubi-style streaming app (THE one for FOX)

Step 1 — clarify

  • "Free ad-supported, like Tubi?" (Yes, AVOD).
  • "VOD only or live?" (mostly VOD, some live news/sports).
  • "Targets — phone/tablet/TV/web?" (all).
  • "Offline downloads?" (premium feature, optional).
  • "Personalization?" (yes, recommendation rows).
  • "Scale?" (millions DAU, 100Ks concurrent at peak).

Step 2 — requirements

Functional: browse rows of categories, watch with ads (SSAI), sign-in (optional), continue-watching, search, personalized rows, captions, audio tracks.

Non-functional: cold start < 1.5s, time-to-first-frame < 2s, 60fps row scrolling on low-end TVs, offline-friendly browsing, 99.9% playback success.

4. LLD — Image Loader (Glide / Coil clone)

Clarify

  • Memory + disk cache?
  • Cancel on view recycle?
  • Transformations (round, blur, resize)?
  • Placeholder + error?
  • Priority (visible items first)?

Architecture

ImageRequest (url, target, transformations, placeholder)
    ↓
RequestQueue (dedupe by key, prioritize)
    ↓
Memory cache (LruCache<Key, Bitmap>) ── hit → bind to target
    ↓ miss
Disk cache (LRU on disk) ── hit → decode → bind, also store in memory
    ↓ miss
Network (OkHttp) → decode → store both caches → bind

API + key

data class ImageRequest(
    val url: String,
    val target: ImageView,
    val width: Int,
    val height: Int,
    val transformations: List<Transform> = emptyList(),
    val placeholder: Int? = null,
    val error: Int? = null
)

// Cache key includes size + transforms (different sizes ≠ same cache entry)
data class CacheKey(
    val url: String,
    val w: Int, val h: Int,
    val transforms: List<String>
)

Skeleton

class ImageLoader(
    private val memoryCache: LruCache<CacheKey, Bitmap>,
    private val diskCache: DiskLruCache,
    private val client: OkHttpClient,
    private val ioScope: CoroutineScope
) {
    private val activeRequests = mutableMapOf<ImageView, Job>()

    fun load(req: ImageRequest) {
        cancel(req.target)
        val key = req.cacheKey()

        memoryCache[key]?.let {
            req.target.setImageBitmap(it)
            return
        }

        req.placeholder?.let { req.target.setImageResource(it) }

        activeRequests[req.target] = ioScope.launch {
            val bitmap = diskCache.get(key)?.let(::decode)
                ?: download(req.url)?.also { storeBoth(key, it) }
                ?: return@launch withContext(Dispatchers.Main) {
                    req.error?.let { req.target.setImageResource(it) }
                }
            withContext(Dispatchers.Main) {
                if (req.target.tag == key) req.target.setImageBitmap(bitmap)
            }
        }
        req.target.tag = key
    }

    fun cancel(target: ImageView) { activeRequests.remove(target)?.cancel() }
}

Sizing the LruCache

val maxKb = (Runtime.getRuntime().maxMemory() / 1024 / 8).toInt()    // 1/8 of heap
val memCache = object : LruCache<CacheKey, Bitmap>(maxKb) {
    override fun sizeOf(k: CacheKey, v: Bitmap): Int = v.byteCount / 1024
}

5. LLD — Offline-first sync engine

Pattern for any offline-capable CRUD app (notes, todos, drafts).

Schema

@Entity
data class NoteEntity(
    @PrimaryKey val id: String,           // client UUID
    val serverId: String? = null,
    val title: String,
    val body: String,
    val updatedAt: Long,
    val version: Int = 0,
    val status: String = "PENDING_CREATE"  // PENDING_CREATE / SYNCED / PENDING_UPDATE / PENDING_DELETE
)

Sync worker

@HiltWorker
class SyncWorker @AssistedInject constructor(
    @Assisted ctx: Context, @Assisted params: WorkerParameters,
    private val dao: NoteDao, private val api: NoteApi
) : CoroutineWorker(ctx, params) {

    override suspend fun doWork(): Result {
        // Pushes
        dao.pendingByStatus("PENDING_CREATE").forEach { entity ->
            runCatching { api.create(entity.toCreateDto()) }
                .onSuccess { resp -> dao.upsert(entity.copy(serverId = resp.id, status = "SYNCED")) }
                .onFailure { return Result.retry() }
        }
        dao.pendingByStatus("PENDING_UPDATE").forEach { entity ->
            runCatching { api.update(entity.serverId!!, entity.toPatchDto(), entity.version) }
                .onSuccess { dao.upsert(entity.copy(status = "SYNCED", version = entity.version + 1)) }
                .onFailure { e ->
                    if (e is ConflictException) resolveConflict(entity)
                    else return Result.retry()
                }
        }
        dao.pendingByStatus("PENDING_DELETE").forEach { entity ->
            runCatching { api.delete(entity.serverId!!) }
                .onSuccess { dao.hardDelete(entity.id) }
                .onFailure { return Result.retry() }
        }

        // Pulls
        val since = dao.lastSyncedAt() ?: 0L
        runCatching { api.delta(since) }
            .onSuccess { remote -> dao.upsertAll(remote.map { it.toEntity() }); dao.setLastSyncedAt(System.currentTimeMillis()) }
            .onFailure { return Result.retry() }

        return Result.success()
    }
}
Conflict resolution choices

Pick a strategy upfront and call it out: (a) last-write-wins (simple — server timestamp decides), (b) prompt user (good UX, complex), (c) three-way merge (Notion/Figma-level — usually overkill). Sketch the chosen one and acknowledge the others as alternatives.

6. LLD — Chat client

Components

  • WebSocket for real-time inbound messages.
  • Room for local persistence; UI observes via Flow.
  • Send queue with WorkManager; client-generated UUIDs for idempotency.
  • Reconnection with exponential backoff; on reconnect, request "messages since lastDeliveredId."
  • Pagination via keyset on messageId DESC.

Send pattern

fun send(text: String) = scope.launch {
    val clientId = UUID.randomUUID().toString()
    val msg = MessageEntity(id = clientId, text = text, status = "PENDING", createdAt = now())
    dao.upsert(msg)   // appears in UI immediately, marked pending
    enqueueSendWorker(clientId)
}

class SendWorker(...) : CoroutineWorker(...) {
    override suspend fun doWork(): Result {
        val pending = dao.byId(clientId) ?: return Result.success()
        return runCatching { api.send(SendDto(pending.text, clientId)) }
            .onSuccess { resp -> dao.upsert(pending.copy(serverId = resp.id, status = "SENT")) }
            .fold(onSuccess = { Result.success() }, onFailure = { Result.retry() })
    }
}

7. Analytics integration — Compose AND XML

Architecture: AnalyticsHub fans out to multiple trackers

interface AnalyticsTracker {
    fun screen(name: String, props: Map<String, Any> = emptyMap())
    fun event(name: String, props: Map<String, Any> = emptyMap())
}

@Module
@InstallIn(SingletonComponent::class)
abstract class AnalyticsModule {
    @Binds @IntoSet abstract fun newRelic(impl: NewRelicTracker): AnalyticsTracker
    @Binds @IntoSet abstract fun crashlytics(impl: CrashlyticsTracker): AnalyticsTracker
    @Binds @IntoSet abstract fun internal(impl: InternalTracker): AnalyticsTracker
}

class AnalyticsHub @Inject constructor(
    private val trackers: Set<@JvmSuppressWildcards AnalyticsTracker>,
    @IoDispatcher private val io: CoroutineDispatcher
) : AnalyticsTracker {
    private val scope = CoroutineScope(SupervisorJob() + io)
    override fun screen(name: String, props: Map<String, Any>) {
        scope.launch { trackers.forEach { runCatching { it.screen(name, props) } } }
    }
    override fun event(name: String, props: Map<String, Any>) {
        scope.launch { trackers.forEach { runCatching { it.event(name, props) } } }
    }
}

// Track scroll depth
@Composable
fun TrackScrollDepth(listState: LazyListState, screen: String) {
    val analytics = LocalAnalytics.current
    LaunchedEffect(listState) {
        snapshotFlow { listState.firstVisibleItemIndex }
            .filter { it > 0 }
            .distinctUntilChanged()
            .collect { idx -> analytics.event("scroll_depth", mapOf("screen" to screen, "idx" to idx)) }
    }

Custom in-memory event engine (your Dish work)

class InternalTracker @Inject constructor(
    private val dao: AnalyticsDao,
    private val api: AnalyticsApi,
    private val scope: AppScope
) : AnalyticsTracker {
    private val channel = Channel<Event>(Channel.BUFFERED)

    init {
        scope.launch { drainBatched() }
    }

    override fun event(name: String, props: Map<String, Any>) {
        channel.trySend(Event(name, props, System.currentTimeMillis()))
    }

    private suspend fun drainBatched() {
        val batch = mutableListOf<Event>()
        while (true) {
            val first = channel.receive()
            batch += first
            // Drain available within 30s window
            withTimeoutOrNull(30_000) {
                while (batch.size < 50) batch += channel.receive()
            }
            // Persist + send
            dao.insertAll(batch.map { it.toEntity() })
            runCatching { api.upload(batch) }
                .onSuccess { dao.markSent(batch) }
                .onFailure { /* will retry next drain cycle */ }
            batch.clear()
        }
    }
}

8. LLD — Photo upload pipeline

  • Pick photo → ViewModel → emit Uri.
  • WorkManager foreground service for upload (user-visible long work).
  • Resumable upload: chunked PUT (S3 multipart, Tus). Persist chunk progress.
  • Progress: setProgress(workDataOf("pct" to 50)); observe via workManager.getWorkInfoByIdFlow.
  • Constraints: NetworkType.UNMETERED if user toggles wifi-only.
class UploadWorker(...) : CoroutineWorker(...) {
    override suspend fun getForegroundInfo(): ForegroundInfo {
        val notif = makeNotification(progress = 0)
        return ForegroundInfo(NOTIFICATION_ID, notif)
    }
    override suspend fun doWork(): Result {
        val uri = inputData.getString("uri")?.toUri() ?: return Result.failure()
        val chunks = chunkStrategy(uri)
        chunks.forEachIndexed { i, chunk ->
            uploadChunk(chunk)
            setProgress(workDataOf("pct" to (i + 1) * 100 / chunks.size))
            setForeground(ForegroundInfo(NOTIFICATION_ID, makeNotification((i + 1) * 100 / chunks.size)))
        }
        return Result.success()
    }
}

9. LLD — Push notification pipeline

Components

  • FCM token registered on app start; sent to backend.
  • Token rotation: FirebaseMessagingService.onNewToken → re-register.
  • Notification channels (Android 8+): create on app start; user-controlled importance per channel.
  • Backstack: TaskStackBuilder when deep-linking from notification.
  • Silent push for data sync — priority: "high", no display.
  • Privacy: redact sensitive content on lockscreen via setVisibility.
class FoxFcmService : FirebaseMessagingService() {
    @Inject lateinit var tokens: TokenStore
    override fun onNewToken(token: String) {
        scope.launch { api.registerFcm(token) }
    }
    override fun onMessageReceived(msg: RemoteMessage) {
        val type = msg.data["type"]
        when (type) {
            "silent_sync" -> enqueueSyncWorker()
            "new_episode" -> showEpisodeNotification(msg.data)
            else -> showGenericNotification(msg)
        }
    }
}

11. Common HLD/LLD interview probes

Q. How do you scale this from 100K to 10M users?

Backend scaling is the primary lever (CDN, caching, sharding) — say so. Mobile-side: more aggressive caching, batched API calls (or GraphQL with sparse fields), feature flags for risky rollouts, robust telemetry, A/B testing infra, baseline profiles for cold start at p95.

Q. How do you handle a flaky network?

Retries with exponential backoff for transient (5xx, IO). Idempotency keys for retried POSTs. Cache last successful response (OkHttp HTTP cache + app-level cache). Surface "offline mode" UI when consecutive failures exceed N. Auto-resume when network returns via ConnectivityManager.NetworkCallback.

Q. What if backend changes the home layout daily?

SDUI (Server-Driven UI) — server returns module descriptors, client renders generically. Layout changes ship without app release. Tradeoff: client must support all module types in advance; new types require app update.

Q. How do you handle two devices editing the same note offline?

Three options: last-write-wins (simple, lossy), version-based 409 + user prompt (better UX), three-way merge (complex). Pick one and call out the alternatives. Server-generated content hash can dedupe identical edits.

Q. How do you design pagination across multiple sources?

Paging 3 with RemoteMediator: DB is source of truth, mediator fetches from network and writes through. Cursor-based, not page-numbered. Handle "empty page" as end-of-pagination signal.

Q. How do you implement deep links safely?

Don't trust URL params. Validate IDs against current user session. Use Android App Links (verified by domain) over plain custom-scheme links — proves your domain ownership. Prefer Navigation Compose's navDeepLink for routing.

12. Design-round closing tips

  • Always articulate tradeoffs: "I'm picking X over Y because Z; downside is W."
  • Be honest about what you'd punt: "I'd not build the recommendation engine in V1 — start with categorical rows, add personalization in V2."
  • Bring numbers when possible: "We targeted < 1.5s cold start, P99."
  • If asked "how would you fail this?" — engage. Senior interviewers love this question.
  • Don't draw 50 boxes. Keep it readable; add detail only on the part being probed.
