1. MVVM vs MVI vs Clean — the deep dive

MVVM = ViewModel exposes UI state via observable streams; the View observes and renders; events flow back via method calls on the ViewModel. The default Android pattern. MVI = a stricter MVVM with one State object, sealed Intents, and Effects on a separate channel — best for state-heavy screens. Clean Architecture = a layering concern (presentation / domain / data) orthogonal to MVVM/MVI; you compose them.

Side-by-side

MVVMMVI
State exposuremultiple flows (notes, loading, error)one StateFlow<UiState>
Events from UImethod calls (vm.load(), vm.delete())vm.onIntent(Intent.Load)
One-shot eventsoften hacky (SingleLiveEvent, Channel)SharedFlow<Effect>
Best forsimple CRUD screens — most app screensstate-heavy: player, multi-step forms, search
Testabilitygoodexcellent — pure reducer
Boilerplatelowmedium-high

Clean Architecture — the layering

┌──────────────── presentation ────────────────┐
│  Compose / XML + ViewModel                   │
│  Knows about Android, knows about domain     │
└──────────────────┬───────────────────────────┘

┌──────────────── domain ──────────────────────┐  ← PURE Kotlin (no Android)
│  Use Cases, Domain Models, Repository INTERFACES│

                   ↑ implements (dependency inversion)
┌──────────────── data ────────────────────────┐
│  Repository implementations, Retrofit, Room  │
│  Knows about Android, knows about domain     │
└──────────────────────────────────────────────┘
Senior framing — say this

"I default to MVVM with Clean layering for most screens. I lift to MVI on specific screens where state interactions get complex enough that 'what's the screen showing right now' becomes hard to answer from scattered flows. The win of MVI is single source of truth for screen state — every UI state can be reproduced in a test or @Preview by just constructing the State object."

2. Build a movies fetcher — end-to-end

This is the centerpiece. If TR1 says "fetch movies from an API and show a list" — this is the answer.

2.1 — The contract (clarify scope first)

  • Show a list of trending movies on a screen.
  • Loading spinner while fetching.
  • Error message if fetch fails, with a retry button.
  • Empty state if the list is empty.
  • Tap a movie to open detail (out of TR1 scope but architecture should support it).

2.2 — Layer-by-layer (Clean + MVVM)

Walk this from the API surface inward. Each layer has a clear responsibility.

Layer 1 — Data (network DTO + Retrofit + DataSource)

// data/remote/MovieDto.kt — what the API returns (DTO = Data Transfer Object)
@Serializable
data class MovieDto(
    val id: Int,
    val title: String,
    @SerialName("poster_path") val posterPath: String?,
    @SerialName("vote_average") val voteAverage: Double,
    @SerialName("release_date") val releaseDate: String
)

@Serializable
data class MoviesPageDto(
    val page: Int,
    val results: List<MovieDto>,
    @SerialName("total_pages") val totalPages: Int
)

// data/remote/MoviesApi.kt — Retrofit interface
interface MoviesApi {
    @GET("trending/movie/week")
    suspend fun trending(@Query("page") page: Int = 1): MoviesPageDto
}

// data/remote/MoviesRemoteDataSource.kt — wraps Retrofit; could be swapped for fakes
class MoviesRemoteDataSource @Inject constructor(
    private val api: MoviesApi,
    @IoDispatcher private val io: CoroutineDispatcher
) {
    suspend fun trending(page: Int = 1): List<MovieDto> = withContext(io) {
        api.trending(page).results
    }
}
  1. DTO mirrors the API exactly — including the wire-format names (poster_path) via @SerialName. The rest of the app should NEVER see a DTO.
  2. Retrofit interface is suspend — Retrofit's coroutine adapter handles the HTTP call asynchronously and resumes your coroutine when the response is ready. No callbacks, no Result wrappers needed at this layer.
  3. DataSource wraps the API — gives you a place to hook caching, logging, retries, dispatcher choice. withContext(io) moves the call off main even if the caller is on Main.

Layer 2 — Data (Repository + mappers)

// data/MovieMapper.kt — converts wire format to domain
fun MovieDto.toDomain() = Movie(
    id = id.toString(),
    title = title,
    posterUrl = posterPath?.let { "https://image.tmdb.org/t/p/w500$it" },
    rating = voteAverage,
    releaseDate = releaseDate.takeIf { it.isNotBlank() }?.let(LocalDate::parse)
)

// data/MoviesRepositoryImpl.kt
class MoviesRepositoryImpl @Inject constructor(
    private val remote: MoviesRemoteDataSource
) : MoviesRepository {        // implements the interface from the domain layer
    override suspend fun trending(page: Int): List<Movie> =
        remote.trending(page).map { it.toDomain() }
}
The repository does TWO jobs: (1) coordinates data sources (here just one — but later we could add a Room cache and choose between them), and (2) maps DTOs to domain models. After this layer, no caller can be coupled to the wire format.

Layer 3 — Domain (model + repository interface + use case)

// domain/model/Movie.kt — pure Kotlin, no Android, no DTO
data class Movie(
    val id: String,
    val title: String,
    val posterUrl: String?,
    val rating: Double,
    val releaseDate: LocalDate?
)

// domain/repository/MoviesRepository.kt — interface only
interface MoviesRepository {
    suspend fun trending(page: Int = 1): List<Movie>
}

// domain/usecase/GetTrendingMoviesUseCase.kt
class GetTrendingMoviesUseCase @Inject constructor(
    private val repo: MoviesRepository
) {
    suspend operator fun invoke(): List<Movie> =
        repo.trending().filter { it.rating >= 6.0 }   // example business rule
}
  1. Domain layer is pure Kotlin — no Android imports, no Retrofit, no Room. JVM unit tests run in milliseconds with no emulator.
  2. Repository INTERFACE lives here, IMPL in data — that's the dependency inversion. The data layer depends on the domain (implements its interfaces), not the other way around.
  3. Use Case has operator fun invoke — call site reads getTrendingMovies(), not getTrendingMovies.execute(). This is the senior-Kotlin idiom.
  4. Use cases hold the business rules — here, "rating >= 6.0". If multiple ViewModels need this filter, they all share one place to fix it.

Layer 4 — Presentation (ViewModel — MVVM style)

// presentation/MoviesUiState.kt — sealed for exhaustive when
sealed interface MoviesUiState {
    data object Loading : MoviesUiState
    data class Success(val movies: List<Movie>) : MoviesUiState
    data class Error(val message: String) : MoviesUiState
}

// presentation/MoviesViewModel.kt
@HiltViewModel
class MoviesViewModel @Inject constructor(
    private val getTrending: GetTrendingMoviesUseCase
) : ViewModel() {

    private val _state = MutableStateFlow<MoviesUiState>(MoviesUiState.Loading)
    val state: StateFlow<MoviesUiState> = _state.asStateFlow()

    init { load() }

    fun load() = viewModelScope.launch {
        _state.value = MoviesUiState.Loading
        runCatching { getTrending() }
            .onSuccess { movies ->
                _state.value = if (movies.isEmpty()) MoviesUiState.Success(emptyList())
                                else MoviesUiState.Success(movies)
            }
            .onFailure { e ->
                _state.value = MoviesUiState.Error(e.message ?: "Couldn't load movies")
            }
    }

    fun retry() = load()
}
  1. One StateFlow<UiState> exposes the entire screen state — Loading / Success / Error. The View renders one of three things based on which subclass is current.
  2. viewModelScope.launch — coroutine cancelled when ViewModel is cleared (configuration change, finish). No leaks.
  3. runCatching { } .onSuccess { } .onFailure { } — handles the success/error fork without nested try/catch. Internal use only — never return Result from a public API.
  4. retry() just calls load() again — the View binds its retry button to this.

Layer 5a — UI (Compose)

@Composable
fun MoviesScreen(vm: MoviesViewModel = hiltViewModel()) {
    val state by vm.state.collectAsStateWithLifecycle()

    when (val s = state) {
        is MoviesUiState.Loading -> LoadingView()
        is MoviesUiState.Error   -> ErrorView(s.message, onRetry = vm::retry)
        is MoviesUiState.Success -> if (s.movies.isEmpty())
            EmptyView() else MoviesList(s.movies)
    }
}

@Composable
private fun LoadingView() {
    Box(Modifier.fillMaxSize(), contentAlignment = Alignment.Center) {
        CircularProgressIndicator()
    }
}

@Composable
private fun ErrorView(msg: String, onRetry: () -> Unit) {
    Column(
        Modifier.fillMaxSize().padding(24.dp),
        verticalArrangement = Arrangement.Center,
        horizontalAlignment = Alignment.CenterHorizontally
    ) {
        Text(msg, style = MaterialTheme.typography.bodyLarge)
        Spacer(Modifier.height(16.dp))
        Button(onClick = onRetry) { Text("Retry") }
    }
}

@Composable
private fun MoviesList(movies: List<Movie>) {
    LazyColumn {
        items(movies, key = { it.id }) { movie ->
            MovieRow(movie)
        }
    }
}

@Composable
private fun MovieRow(movie: Movie) {
    Row(Modifier.fillMaxWidth().padding(12.dp)) {
        AsyncImage(
            model = movie.posterUrl,
            contentDescription = null,
            modifier = Modifier.size(80.dp).clip(RoundedCornerShape(8.dp))
        )
        Spacer(Modifier.width(12.dp))
        Column {
            Text(movie.title, style = MaterialTheme.typography.titleMedium)
            Text("⭐ ${movie.rating}", style = MaterialTheme.typography.bodySmall)
        }
    }
}
  1. collectAsStateWithLifecycle() — collects the StateFlow only while the screen is at least STARTED. No background work when the user navigates away.
  2. Exhaustive when (val s = state) — sealed interface forces handling every state. Adding a fourth state (e.g., Refreshing) becomes a compile error in every consumer until you handle it.
  3. key = { it.id } on the LazyColumn — Compose uses this to identify items across data updates. Without it, scroll position jumps and animations target wrong items.
  4. Method reference vm::retry — stable lambda equivalent. Compose can skip recomposition of ErrorView when nothing else changes.

Layer 5b — UI (XML alternative)

@AndroidEntryPoint
class MoviesFragment : Fragment(R.layout.fragment_movies) {

    private val vm: MoviesViewModel by viewModels()

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        val binding = FragmentMoviesBinding.bind(view)
        val adapter = MoviesAdapter()
        binding.list.adapter = adapter
        binding.btnRetry.setOnClickListener { vm.retry() }

        viewLifecycleOwner.lifecycleScope.launch {
            repeatOnLifecycle(Lifecycle.State.STARTED) {
                vm.state.collect { state ->
                    binding.progress.isVisible = state is MoviesUiState.Loading
                    binding.errorView.isVisible = state is MoviesUiState.Error
                    binding.list.isVisible = state is MoviesUiState.Success

                    when (state) {
                        is MoviesUiState.Loading -> { /* spinner already shown */ }
                        is MoviesUiState.Error   -> binding.errorMsg.text = state.message
                        is MoviesUiState.Success -> adapter.submitList(state.movies)
                    }
                }
            }
        }
    }
}
  1. viewLifecycleOwner not this — fragments outlive their views (back stack). Using this would leak observers when the view is destroyed.
  2. repeatOnLifecycle(STARTED) — XML equivalent of collectAsStateWithLifecycle. Cancels collection when the screen is no longer visible.
  3. Visibility toggles drive UI — same one StateFlow drives all three views (progress, error, list). XML doesn't have exhaustive when for views, so isVisible bools handle it.
  4. ListAdapter + DiffUtil (in MoviesAdapter, not shown) — only redraws changed rows on submitList.

2.3 — Same screen in MVI style

If TR1 specifically asks for MVI, here's the diff vs MVVM:

// State (same as before — already a sealed UiState)

// Intents — what the user can do
sealed interface MoviesIntent {
    data object Load : MoviesIntent
    data object Retry : MoviesIntent
    data class Click(val movie: Movie) : MoviesIntent
}

// Effects — one-shot events (navigation, snackbar)
sealed interface MoviesEffect {
    data class NavigateToDetail(val id: String) : MoviesEffect
    data class ShowSnackbar(val msg: String) : MoviesEffect
}

@HiltViewModel
class MoviesViewModel @Inject constructor(
    private val getTrending: GetTrendingMoviesUseCase
) : ViewModel() {

    private val _state = MutableStateFlow<MoviesUiState>(MoviesUiState.Loading)
    val state: StateFlow<MoviesUiState> = _state.asStateFlow()

    private val _effects = MutableSharedFlow<MoviesEffect>(extraBufferCapacity = 1)
    val effects: SharedFlow<MoviesEffect> = _effects.asSharedFlow()

    init { onIntent(MoviesIntent.Load) }

    fun onIntent(intent: MoviesIntent) {
        when (intent) {
            MoviesIntent.Load, MoviesIntent.Retry -> load()
            is MoviesIntent.Click -> viewModelScope.launch {
                _effects.emit(MoviesEffect.NavigateToDetail(intent.movie.id))
            }
        }
    }

    private fun load() = viewModelScope.launch {
        _state.value = MoviesUiState.Loading
        runCatching { getTrending() }
            .onSuccess { _state.value = MoviesUiState.Success(it) }
            .onFailure {
                _state.value = MoviesUiState.Error(it.message ?: "")
                _effects.emit(MoviesEffect.ShowSnackbar("Couldn't load — tap retry"))
            }
    }
}

// In the Compose screen
@Composable
fun MoviesScreen(vm: MoviesViewModel = hiltViewModel(), nav: NavController) {
    val state by vm.state.collectAsStateWithLifecycle()

    LaunchedEffect(Unit) {
        vm.effects.collect { effect ->
            when (effect) {
                is MoviesEffect.NavigateToDetail -> nav.navigate("movie/${effect.id}")
                is MoviesEffect.ShowSnackbar -> { /* via SnackbarHost */ }
            }
        }
    }

    /* same when (state) rendering as MVVM */
}
The MVI delta from MVVM: (1) a single onIntent entry point instead of separate methods, (2) Effects on a SharedFlow for one-shot events (navigation, snackbar) — replay=0 means rotation doesn't replay them, (3) the View is even more passive — it dispatches Intents and renders State.

3. Suspend functions

Suspend function = a function that can pause its execution and resume later, without blocking the underlying thread. Marked with the suspend keyword. Only callable from another suspend function or a coroutine builder (launch, async, runBlocking). The compiler transforms each suspend function into a state machine that saves local state at each suspension point.
// Looks synchronous — but doesn't block the calling thread
suspend fun fetchUser(id: String): User {
    val response = api.user(id)        // suspends here — thread is free
    val avatar = api.avatar(id)        // suspends again
    return User(response, avatar)
}

Common interview probes

  • "Why use suspend instead of callbacks?" — sequential code, no callback hell, exceptions propagate naturally, structured concurrency.
  • "Does suspend mean it runs on a background thread?" — No. It runs on whatever Dispatcher the coroutine is in. Use withContext(Dispatchers.IO) to actually move it.
  • "Can you call a suspend function from a regular function?" — No. You need a coroutine builder.

Full coverage on the Coroutines page.

4. Exception handling

4 patterns to know

// PATTERN 1 — try/catch in a suspend function
suspend fun fetch(): Result<User> = try {
    Result.success(api.user())
} catch (e: IOException) {
    Result.failure(e)
} catch (e: CancellationException) {
    throw e                          // ALWAYS rethrow cancellation
}

// PATTERN 2 — runCatching (idiomatic for "try and convert to Result")
suspend fun fetch(): Result<User> = runCatching { api.user() }

// PATTERN 3 — CoroutineExceptionHandler (catches uncaught launch exceptions)
val handler = CoroutineExceptionHandler { _, e ->
    Log.e("App", "uncaught", e)
}
val scope = CoroutineScope(SupervisorJob() + Dispatchers.Default + handler)
scope.launch { /* throw IOException("x") — handler catches */ }

// PATTERN 4 — async holds exception until await()
val deferred = scope.async { throw IOException("x") }
try { deferred.await() } catch (e: IOException) { /* handle */ }
// CoroutineExceptionHandler does NOT catch async exceptions until awaited
The cancellation trap

CancellationException extends Exception — a generic catch (e: Exception) swallows it, breaking cooperative cancellation. Always rethrow it: if (e is CancellationException) throw e. Or catch only the specific exceptions you mean to handle.

5. UI updates — XML (LiveData) vs Compose (StateFlow)

Coming from XML, the key insight: StateFlow is the modern replacement for LiveData. Same role (lifecycle-aware observable from ViewModel), different mechanics underneath.

LiveData (XML era)StateFlow (modern)
Libraryandroidx.lifecycle (Android-only)kotlinx.coroutines (Kotlin Multiplatform)
Initial valueOptionalRequired
Lifecycle-aware on its own?Yes (auto-pauses observers off-screen)No — needs collectAsStateWithLifecycle (Compose) or repeatOnLifecycle (XML)
Composes with operators?Limited (map, switchMap)Full Flow operator suite (combine, flatMapLatest, debounce, etc.)
ThreadingPosts updates to main threadHonors caller's dispatcher
Equality dedupeNoYes (no emit if new equals old)
Recommended forExisting XML codebasesNew code, especially Compose

Same VM, two UIs

// ViewModel — StateFlow style (works for both Compose AND XML)
@HiltViewModel
class HomeVm @Inject constructor() : ViewModel() {
    private val _state = MutableStateFlow<UiState>(UiState.Loading)
    val state: StateFlow<UiState> = _state.asStateFlow()
}

// Compose UI
@Composable
fun HomeScreen(vm: HomeVm = hiltViewModel()) {
    val state by vm.state.collectAsStateWithLifecycle()
    when (state) { /* render */ }
}

// XML UI
class HomeFragment : Fragment() {
    override fun onViewCreated(v: View, b: Bundle?) {
        viewLifecycleOwner.lifecycleScope.launch {
            repeatOnLifecycle(Lifecycle.State.STARTED) {
                vm.state.collect { /* update UI */ }
            }
        }
    }
}
The lifecycle-aware collection means the Flow's collector is started when the screen reaches STARTED, cancelled when it drops below. Without it, you'd have a coroutine still consuming updates in the background after the user navigated away — leaking work and possibly memory.

6. Handling error states + loading

Senior pattern: model loading and error as part of the state, not as separate signals. The View's when over the sealed UiState handles all three uniformly.

sealed interface UiState {
    data object Loading : UiState
    data class Success(val items: List<Item>) : UiState
    data class Error(val message: String, val canRetry: Boolean = true) : UiState
}

// In the View
when (val s = state) {
    UiState.Loading      -> CircularProgressIndicator()
    is UiState.Success   -> if (s.items.isEmpty()) EmptyView() else List(s.items)
    is UiState.Error     -> ErrorView(s.message, onRetry = if (s.canRetry) vm::retry else null)
}

For partial states (data + refreshing simultaneously)

// When you want to show STALE data with a refresh spinner on top
data class UiState(
    val items: List<Item> = emptyList(),
    val isRefreshing: Boolean = false,
    val error: String? = null
)
// Now Loading is just isRefreshing = true; data stays visible underneath
Senior framing

"For pure-fetch screens I use a sealed UiState (Loading / Success / Error) — one type, three branches, exhaustive. For screens that show cached data and refresh in the background (offline-first), I use a data class with isRefreshing + items + error fields so I can show the existing data with a refresh indicator on top. The choice depends on whether the user benefits from seeing previous data."

7. Lifecycle awareness

Lifecycle-aware = an observer or component that automatically starts/stops its work in sync with an Activity/Fragment lifecycle. Prevents leaks (long-running work after destroy) and wasted CPU (observers running while screen is hidden).

The toolkit

APIBehavior
viewModelScopecancelled when ViewModel.onCleared (config change OR screen finished)
lifecycleScopecancelled when Lifecycle DESTROYED. Use viewLifecycleOwner.lifecycleScope in Fragments
repeatOnLifecycle(STARTED) { }(re)starts/cancels at STARTED — for Flow collection in XML
collectAsStateWithLifecycle()Compose equivalent — same behavior
DefaultLifecycleObserverOld-school onStart/onStop hooks — rarely needed in modern code

The view-vs-fragment lifecycle gotcha

// WRONG — fragment outlives view in the back stack; this leaks observers
viewModel.state.observe(this) { /* ... */ }

// RIGHT — observer dies with the view
viewModel.state.observe(viewLifecycleOwner) { /* ... */ }
A fragment can outlive its view (back stack). The fragment instance persists, but its view (and all its bindings) is destroyed. Using this as LifecycleOwner ties observers to the fragment's longer lifecycle — but the view they reference is gone. Always use viewLifecycleOwner for UI-bound observers.

8. Parallel vs sequential coroutines

// SEQUENTIAL — second call starts only after first returns. Total time = sum.
suspend fun loadSequential(): Dashboard {
    val user = api.user()           // wait...
    val notifications = api.notifications()  // ...wait again
    val feed = api.feed()           // ...and again
    return Dashboard(user, notifications, feed)
}

// PARALLEL — all three run at once. Total time = max of the three.
suspend fun loadParallel(): Dashboard = coroutineScope {
    val user = async { api.user() }
    val notifications = async { api.notifications() }
    val feed = async { api.feed() }
    Dashboard(user.await(), notifications.await(), feed.await())
}

// PARALLEL with partial failures tolerated — feed is optional
suspend fun loadResilient(): Dashboard = coroutineScope {
    val user = async { api.user() }
    val notifications = async { api.notifications() }
    val feed = async { runCatching { api.feed() }.getOrDefault(emptyList()) }
    Dashboard(user.await(), notifications.await(), feed.await())
}
  1. Sequential when each call depends on the previous one's result. val token = login(); val data = fetch(token) — fetch needs token.
  2. Parallel with async + await when calls are independent. Use coroutineScope { } as the parent so any failure cancels siblings (atomic group).
  3. Resilient parallel with supervisorScope or per-call runCatching when partial failure is acceptable.
Common bug

val a = async { ... }.await(); val b = async { ... }.await() is secretly sequential — the first .await() blocks before the second async starts. Always declare both asyncs first, then await at the end.

9. Jetpack Compose — revision

The 6 things you'll be quizzed on

  1. Recomposition: re-running composables when state inputs change. Skip happens when all inputs are stable and unchanged.
  2. Stability: a type Compose can trust to mean "if equal to before, output is unchanged." Mark data classes @Immutable when they have List / Map fields.
  3. remember vs rememberSaveable: remember survives recomposition; rememberSaveable additionally survives configuration change AND process death (Bundle).
  4. Side effects: LaunchedEffect(key) for coroutines, DisposableEffect(key) for cleanup-required resources, SideEffect for "sync to non-Compose" after each composition, derivedStateOf for "only recompose when result changes."
  5. State hoisting: push state up; pass state + onEvent down. Stateless leaves are easier to test/preview/reuse.
  6. LazyColumn with key = { it.id } — always. Without it, scroll position jumps on data updates.

Quick reference — building a Compose screen from scratch

@Composable
fun MyScreen(vm: MyVm = hiltViewModel()) {
    val state by vm.state.collectAsStateWithLifecycle()

    // Coroutine triggered ONCE on enter
    LaunchedEffect(Unit) { vm.load() }

    // Cleanup pattern
    DisposableEffect(Unit) {
        val listener = SomeListener { /* ... */ }
        someService.register(listener)
        onDispose { someService.unregister(listener) }
    }

    // Render based on state
    when (state) { /* ... */ }
}

10. SOLID principles in Android — with code

SOLID = five OOP principles that guide design toward code that's easier to change, test, and extend. Coined by Robert Martin. They're heuristics, not laws — apply them where they pay off, skip them where they'd add ceremony for nothing.

S — Single Responsibility Principle

"A class should have one reason to change."

Bad — VM does everything

class MoviesVm : ViewModel() {
    private val _state = MutableStateFlow<UiState>(...)
    val state = _state.asStateFlow()

    private val client = OkHttpClient()       // network
    private val gson = Gson()                  // parsing
    private val cache = HashMap<Int, Movie>()  // caching
    private val analytics = FirebaseAnalytics  // telemetry

    fun load() {
        // ...does network call, parses JSON, caches, logs analytics, updates state
    }
}

Good — split by responsibility

class MoviesApi { /* HTTP only */ }
class MoviesCache { /* LRU only */ }
class Analytics { /* events only */ }

class MoviesRepository @Inject constructor(
    private val api: MoviesApi,
    private val cache: MoviesCache
) { /* coordinates only */ }

@HiltViewModel
class MoviesVm @Inject constructor(
    private val repo: MoviesRepository,
    private val analytics: Analytics
) : ViewModel() { /* UI state only */ }

Senior framing: "VM holds UI state and orchestrates intents — it doesn't know HTTP. Repository coordinates data sources — it doesn't know UI. Each class can be tested with one fake."

O — Open/Closed Principle

"Open for extension, closed for modification." Adding new behavior shouldn't require editing existing code.

Bad — must edit when adding states

// Every new state requires changing MoviesUiState
class MoviesUiState {
    var isLoading: Boolean = false
    var movies: List<Movie> = emptyList()
    var error: String? = null
    var isOffline: Boolean = false      // new — added this
    var isRefreshing: Boolean = false   // new — added this
}

// And every consumer's render() needs new branches:
when {
    state.isLoading -> ...
    state.isOffline -> ...    // edited
    state.isRefreshing -> ... // edited
    state.error != null -> ...
    else -> ...
}

Good — sealed, extend by adding subclass

sealed interface MoviesUiState {
    data object Loading : MoviesUiState
    data object Refreshing : MoviesUiState     // added; no edits to siblings
    data object Offline : MoviesUiState         // added; no edits to siblings
    data class Success(val movies: List<Movie>) : MoviesUiState
    data class Error(val msg: String) : MoviesUiState
}

// when (state) — compiler enforces all branches handled
// adding a new state forces every consumer to handle it

Senior framing: "Sealed hierarchies + exhaustive when = OCP made concrete. The new UiState subclass requires no edit to siblings; it requires consumers to acknowledge it (which is a feature, not a bug)."

L — Liskov Substitution Principle

"Subtypes must be substitutable for their base type without surprising callers." An impl that violates the interface contract violates LSP.

Bad — fake violates contract

interface UserRepo {
    /** @throws IOException on network failure */
    suspend fun fetchUser(id: String): User
}

class FakeUserRepo : UserRepo {
    override suspend fun fetchUser(id: String): User {
        // Returns null wrapped in unchecked cast — caller crashes with ClassCastException
        return null as User
    }
}

Good — fake honors the contract

class FakeUserRepo(
    private val users: Map<String, User> = emptyMap(),
    private val shouldFail: Boolean = false
) : UserRepo {
    override suspend fun fetchUser(id: String): User {
        if (shouldFail) throw IOException("fake network error")
        return users[id] ?: throw NoSuchElementException("user $id")
    }
}

Senior framing: "LSP is most often violated by fakes that return defaults instead of throwing the contract's documented exceptions. Then the real impl crashes the app where the test passed."

I — Interface Segregation Principle

"Don't force consumers to depend on methods they don't use." Many small interfaces beat one fat one.

Bad — fat interface

interface UserRepo {
    suspend fun fetch(id: String): User
    suspend fun search(q: String): List<User>
    suspend fun create(u: User)
    suspend fun update(u: User)
    suspend fun delete(id: String)
    suspend fun listAll(): List<User>
    suspend fun count(): Int
    fun observeChanges(): Flow<UserEvent>
}

// Profile screen only needs fetch — but its test must
// fake the entire interface (or use a half-implemented mock).

Good — segregate

interface UserReader {
    suspend fun fetch(id: String): User
    suspend fun search(q: String): List<User>
}

interface UserWriter {
    suspend fun create(u: User)
    suspend fun update(u: User)
    suspend fun delete(id: String)
}

interface UserObserver {
    fun observeChanges(): Flow<UserEvent>
}

// Real impl can implement all three; consumers depend on only what they need.
class UserRepoImpl @Inject constructor(...) : UserReader, UserWriter, UserObserver { /*...*/ }

class ProfileVm @Inject constructor(private val reader: UserReader) : ViewModel()

Senior framing: "Don't shotgun this — most repos can stay one interface. Split when the consumers truly only need a slice (read-only screens, write-heavy workers) AND your test surface is suffering. Premature splitting is its own anti-pattern."

D — Dependency Inversion Principle

"High-level modules should not depend on low-level modules. Both should depend on abstractions." The data layer is "low-level" (network, DB); the domain is "high-level" (business rules). The domain shouldn't depend on Retrofit or Room.

Bad — domain knows about Retrofit

// in domain layer
class GetMoviesUseCase(
    private val retrofit: Retrofit       // ← couples domain to network impl
) {
    suspend operator fun invoke(): List<Movie> {
        val api = retrofit.create(MoviesApi::class.java)
        return api.trending().results.map { it.toDomain() }
    }
}

Good — domain depends on its own interface

// in domain layer (pure Kotlin)
interface MoviesRepository {
    suspend fun trending(): List<Movie>
}

class GetTrendingMoviesUseCase @Inject constructor(
    private val repo: MoviesRepository    // domain interface
) {
    suspend operator fun invoke(): List<Movie> = repo.trending()
}

// in data layer (Android, Retrofit)
class MoviesRepositoryImpl @Inject constructor(
    private val api: MoviesApi
) : MoviesRepository {
    override suspend fun trending(): List<Movie> =
        api.trending().results.map { it.toDomain() }
}

// Hilt wires the impl
@Module @InstallIn(SingletonComponent::class)
abstract class RepoModule {
    @Binds abstract fun bind(impl: MoviesRepositoryImpl): MoviesRepository
}

Senior framing: "DIP is what makes Clean Architecture's domain layer pure — JVM unit tests, no emulator, no mocks of Retrofit. The arrow points inward: data depends on domain, not the reverse."

SOLID quick decision tree

  • Class is > 300 lines or hard to describe in one sentence? SRP violation — split it.
  • Adding a new state forces edits in 5+ places? OCP violation — sealed hierarchy + exhaustive when.
  • Fake/mock crashes where real impl works? LSP violation — match the contract.
  • Test must implement 20 methods to fake one usage? ISP violation — split the interface.
  • Domain code imports retrofit2.* or androidx.*? DIP violation — flip the dependency via an interface.
Senior framing — say this

"SOLID isn't dogma — it's a heuristic. The rule I actually follow: depend on interfaces at module boundaries, keep classes focused enough you can describe them in one sentence, and don't optimize for an extension point you don't need yet. Premature interface explosion is its own anti-pattern. The principles I lean on hardest in Android are SRP (split fat ViewModels) and DIP (interfaces in domain, impls in data) — those two pay for themselves immediately in testability."

Why split them — say this

"Pure domain layer means I can run business-logic unit tests in milliseconds — no emulator, no mocks of Android types. Swap the data implementation (network → local file → mock) without touching domain. The DTO/Entity types stay confined to data; the rest of the app sees only domain models. Mappers at the boundary are where format translation happens."

12. Dependency Injection — overview + samples

Dependency Injection (DI) = a class receives its collaborators from outside (constructor, setter, field) rather than constructing them itself. Hilt = Android's official DI library, built on Dagger 2. The graph is verified at compile time — missing bindings fail the build, not the app at runtime.

The 4 patterns to know cold

// PATTERN 1 — @Inject constructor: Hilt knows how to build this class
class GetMoviesUseCase @Inject constructor(
    private val repo: MoviesRepository
)

// PATTERN 2 — @Provides for third-party / configured types
@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {
    @Provides @Singleton
    fun retrofit(): Retrofit = Retrofit.Builder()
        .baseUrl("https://api.tmdb.org/3/")
        .addConverterFactory(/* ... */)
        .build()

    @Provides @Singleton
    fun moviesApi(r: Retrofit): MoviesApi = r.create()
}

// PATTERN 3 — @Binds for interface → impl (more efficient than @Provides)
@Module
@InstallIn(SingletonComponent::class)
abstract class RepositoryModule {
    @Binds abstract fun bindMoviesRepo(impl: MoviesRepositoryImpl): MoviesRepository
}

// PATTERN 4 — @Qualifier for disambiguating same-type bindings
@Qualifier annotation class IoDispatcher
@Qualifier annotation class MainDispatcher

@Module @InstallIn(SingletonComponent::class)
object DispatchersModule {
    @IoDispatcher @Provides fun io(): CoroutineDispatcher = Dispatchers.IO
    @MainDispatcher @Provides fun main(): CoroutineDispatcher = Dispatchers.Main
}

// usage
class Repo @Inject constructor(
    @IoDispatcher private val io: CoroutineDispatcher
)

Constructor injection vs field injection

// CONSTRUCTOR INJECTION — preferred for everything you control
class Repo @Inject constructor(private val api: Api)         // immutable, testable

// FIELD INJECTION — only for Android components Hilt can't construct (Activity, Fragment, View)
@AndroidEntryPoint
class HomeFragment : Fragment() {
    @Inject lateinit var analytics: AnalyticsTracker         // injected after construction
}
Why constructor > field

(1) Immutable — dependencies are val, can't be reassigned. (2) Easier unit tests — pass fakes directly, no Hilt setup. (3) Compile-time guarantee a dep was provided. (4) Explicit dependency surface — reading the constructor tells you exactly what the class needs. Use field injection only when you have to.
