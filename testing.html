Testing — Compose + XML

JUnit, MockK / Mockito, coroutine-aware tests, Compose UI tests, Espresso for legacy XML. Test pyramid & what to invest in.

test pyramid fakes over mocks StandardTestDispatcher runTest virtual time Compose semantics tree composeTestRule HiltAndroidTest turbine for Flow assertions Robolectric Espresso onView/onData

1. The test pyramid for Android

LayerToolingSpeedCoverage target
Unit (domain, VM, mappers)JUnit + MockK / Mockito + Turbinems> 80%
Integration (DAO + DB, repo + fake API)JUnit + Robolectric or instrumentationsecondskey flows
UI (Compose / XML)composeTestRule / Espressoseconds-minutescritical user journeys
E2E (full app)UI Automator + real devicesminutessmoke
Senior framing

"Most coverage at the unit layer; selective UI tests for critical journeys. UI tests are slow and flaky — we don't write them for trivial things. We use fakes over mocks when behavior matters; mocks for behavior verification only."

2. Testing coroutines & flows

@OptIn(ExperimentalCoroutinesApi::class)
class FetchVmTest {
    private val testDispatcher = StandardTestDispatcher()

    @Before fun setup() { Dispatchers.setMain(testDispatcher) }
    @After fun tearDown() { Dispatchers.resetMain() }

    @Test
    fun `loading then success`() = runTest {
        val repo = FakeNotesRepo(notes = listOf(note1, note2))
        val vm = NotesVm(getNotes = GetNotesUseCase(repo, testDispatcher))

        vm.onIntent(NotesIntent.Load)

        // Loading state appears synchronously
        assertEquals(true, vm.state.value.isLoading)

        advanceUntilIdle()       // virtual clock runs to completion

        with(vm.state.value) {
            assertEquals(false, isLoading)
            assertEquals(2, notes.size)
            assertNull(error)
        }
    }
}

Turbine — assert flow emissions cleanly

@Test
fun `state transitions on load`() = runTest {
    vm.state.test {
        assertEquals(NotesState(), awaitItem())             // initial
        vm.onIntent(NotesIntent.Load)
        assertTrue(awaitItem().isLoading)
        assertFalse(awaitItem().isLoading)
        cancelAndConsumeRemainingEvents()
    }
}

3. Fakes vs Mocks — when which

Fake (preferred for state-rich)

class FakeNotesRepo(
    private var notes: List<Note> = emptyList()
) : NotesRepository {
    private val _flow = MutableStateFlow(notes)
    override fun observeAll() = _flow.asStateFlow()
    override suspend fun add(n: Note) {
        notes = notes + n
        _flow.value = notes
    }
    override suspend fun delete(id: String) {
        notes = notes.filterNot { it.id == id }
        _flow.value = notes
    }
}

Mock (for behavior verification)

// MockK
val repo = mockk<NotesRepository>()
coEvery { repo.add(any()) } just Runs

vm.onIntent(NotesIntent.Add(note))
advanceUntilIdle()
coVerify(exactly = 1) { repo.add(note) }
When to use which

"Fakes when the test cares about state evolution: 'after I add, observeAll() emits the new list.' Mocks when the test cares about that a specific call happened: 'analytics.event was fired exactly once with the right params.' Mock-heavy test suites tend to over-couple to implementation."

4. Compose UI tests

class NotesScreenTest {
    @get:Rule val rule = createComposeRule()

    @Test fun `clicking add invokes callback`() {
        val onAdd = mockk<() -> Unit>(relaxed = true)
        rule.setContent {
            NotesScreen(state = NotesState(), onIntent = { if (it is NotesIntent.Add) onAdd() })
        }
        rule.onNodeWithContentDescription("Add note").performClick()
        verify(exactly = 1) { onAdd() }
    }

    @Test fun `shows error state`() {
        rule.setContent {
            NotesScreen(state = NotesState(error = "boom"), onIntent = {})
        }
        rule.onNodeWithText("boom").assertIsDisplayed()
    }
}

Finder methods (memorize the common ones)

FinderUse
onNodeWithText("X")find by visible text
onNodeWithContentDescription("X")find by content description (semantic)
onNodeWithTag("X")find by Modifier.testTag("X")
onAllNodesWithText("X")multiple matches

6. Room DAO tests (instrumentation, fast)

@RunWith(AndroidJUnit4::class)
class NotesDaoTest {
    private lateinit var db: AppDb
    private lateinit var dao: NotesDao

    @Before fun setUp() {
        db = Room.inMemoryDatabaseBuilder(
            ApplicationProvider.getApplicationContext(), AppDb::class.java
        ).allowMainThreadQueries().build()
        dao = db.notesDao()
    }
    @After fun tearDown() { db.close() }

    @Test
    fun upsertThenObserve() = runTest {
        dao.upsert(NoteEntity("1", "title", "body", 0L))
        val first = dao.observeAll().first()
        assertEquals(1, first.size)
    }
}

7. Espresso (XML codebases)

@RunWith(AndroidJUnit4::class)
class HomeFragmentTest {
    @Test fun `clicks navigate to detail`() {
        launchFragmentInContainer<HomeFragment>()
        onView(withId(R.id.list))
            .perform(RecyclerViewActions.actionOnItemAtPosition<NoteVH>(0, click()))
        onView(withText("Detail")).check(matches(isDisplayed()))
    }
}

Common Espresso matchers / actions

  • withId(R.id.x), withText("X"), withContentDescription("X")
  • onView(...).check(matches(isDisplayed()))
  • onView(...).perform(click(), typeText("hi"), scrollTo())
  • onData(...) for ListView/Spinner
  • RecyclerViewActions.actionOnItemAtPosition<VH>(pos, click())

8. Robolectric — JVM tests for Android types

Run framework-dependent code on JVM (no emulator). Faster than instrumentation but slower than pure unit. Useful for testing Activity/Fragment lifecycle, SharedPreferences, etc.

@RunWith(RobolectricTestRunner::class)
class HomeActivityTest {
    @Test fun `creates activity`() {
        val controller = Robolectric.buildActivity(HomeActivity::class.java).setup()
        assertNotNull(controller.get())
    }
}

9. Tracking flakiness — senior tactic

  • CI tracks per-test pass rate over last N runs.
  • Flaky tests get a JIRA ticket — never just @Ignore without one.
  • Flaky >5% of the time is treated as a real bug — usually a concurrency issue, not "test infra."
Don't @Ignore without investigation

That's how you ship the bug to production. Root-cause first; if it's truly flaky test infrastructure, document and fix the infra. Senior signal: never silence tests reflexively.

10. Senior signal phrases for testing

  • "Fakes over mocks for state-rich collaborators."
  • "Inject dispatchers so tests can use StandardTestDispatcher."
  • "Turbine for Flow assertions."
  • "Compose semantic tree drives UI tests."
  • "In-memory Room for DAO tests."
