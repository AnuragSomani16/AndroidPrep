State holders & reactive types

StateFlow vs mutableStateOf (the question you asked)

StateFlow (kotlinx.coroutines)mutableStateOf (Compose)
Where it livesViewModel (or any non-Compose layer)Inside Compose runtime — only meaningful inside a composition
Survives recomposition?Yes — independent of UIYes (with remember)
Survives configuration change?Yes (lives in ViewModel)No on its own — needs rememberSaveable; even then only Bundle-sized
Visible to non-Compose code?Yes — collectable from anywhere (Activity, Service)No — Compose-only
ThreadingHonors the caller's dispatcherMust be read on Main
How UI subscribesCompose: flow.collectAsStateWithLifecycle(). XML: repeatOnLifecycle(STARTED) { collect { } }Compose-native: val x by mutableStateOf(...) read directly inside @Composable
Equality dedupeYes — won't re-emit if new equals oldYes — won't trigger recomposition if equal
Can be combined / mappedFull Flow operator suiteLimited — usually combine via derivedStateOf
Test outside AndroidYes (pure Kotlin, KMP-ready)Needs Compose test runtime
Which to use when

StateFlow for state that lives in ViewModel and needs to be observable from both Compose and XML, or that survives configuration change. mutableStateOf for state local to a composable (text input, toggle, scroll position) — simpler, integrates natively with Compose's read-tracking. Don't mirror StateFlow into mutableStateOf inside the composable — just collectAsStateWithLifecycle() the StateFlow directly.

StateFlow vs SharedFlow

StateFlowSharedFlow
Initial valueRequiredOptional (start empty)
Replay to new collectorsAlways 1 (the current value)Configurable (0..N)
Equality dedupeYes — won't emit if equal to currentNo — every emit goes through
ConflatedYes — fast emits coalesceConfigurable via onBufferOverflow

Which to use when

StateFlow for "what is the screen showing right now?" — it always has an answer. SharedFlow(replay=0, extraBufferCapacity=1) for one-shot events that must NOT replay on rotation (snackbar, navigate-to-detail).

StateFlow vs LiveData

StateFlowLiveData
Librarykotlinx.coroutines (multiplatform)androidx.lifecycle (Android-only)
Lifecycle awarenessNot on its own — needs collectAsStateWithLifecycle / repeatOnLifecycleBuilt-in — auto-pauses observers off-screen

Which to use when

StateFlow for new code, especially Compose-heavy or KMP-bound. LiveData only when contributing to a legacy XML codebase that's already standardized on it. The lifecycle-awareness gap is closed by repeatOnLifecycle + collectAsStateWithLifecycle.

Channel vs SharedFlow (for events)

ChannelSharedFlow
Hot / coldHotHot
Multiple consumersEach item delivered to exactly one consumer (fan-out)Each item delivered to all active collectors (broadcast)

Which to use when

For VM → UI events use SharedFlow(replay=0, extraBufferCapacity=1, DROP_OLDEST). Channel for true producer/consumer queues with single-delivery semantics — rare in app code.

collectAsState vs collectAsStateWithLifecycle

collectAsStatecollectAsStateWithLifecycle
Librarycompose-runtimelifecycle-runtime-compose (added)
Stops collecting off-screenNo — keeps collecting in backgroundYes — stops below STARTED
Battery / CPU impactWastes if upstream is expensiveCheap when hidden
Which to use when

Always collectAsStateWithLifecycle for production. collectAsState is fine for previews / non-critical local flows.

Sequential vs parallel coroutines

// SEQUENTIAL — total time = sum
val a = api.fetchA()           // wait
val b = api.fetchB()           // then wait

// PARALLEL — total time = max
coroutineScope {
    val a = async { api.fetchA() }
    val b = async { api.fetchB() }
    a.await() to b.await()
}

Sequential when each call depends on the previous one's result. Parallel when calls are independent. The classic bug: async { }.await() back-to-back — that's secretly sequential.

derivedStateOf vs plain expression

// BAD — recomposes on every scroll pixel
val showFab = listState.firstVisibleItemIndex == 0

// GOOD — recomposes only when the boolean flips
val showFab by remember { derivedStateOf { listState.firstVisibleItemIndex == 0 } }

derivedStateOf when a derivation depends on rapidly-changing state but the derived value changes rarely. Without it, every scroll pixel triggers a recomposition. With it, only the threshold-cross does.

compositionLocalOfstaticCompositionLocalOf

@Stable vs @Immutable

Both tell Compose "trust me, equality is enough to know if output is unchanged." Difference is intent:

  • @Immutable — the type's public API cannot change after construction. Stronger contract.
  • @Stable — the type can change, but its equals reflects all relevant changes (so equality means rendered-output equivalence).

For data classes wrapping a List: prefer @Immutable (signals you guarantee the list isn't mutated).

AspectMVVMMVI
State exposureMultiple flows: notes, isLoading, errorOne flow: StateFlow<UiState> sealed
Events from UIMethod calls: vm.load(), vm.delete(id)vm.onIntent(Intent.Load) / onIntent(Intent.Delete(id))
One-shot eventsOften hacky — SingleLiveEvent, ChannelDedicated SharedFlow<Effect>

Clean Architecture vs MVVM

They're not alternatives — they're orthogonal. MVVM is a presentation pattern (how UI gets its data); Clean is a layering pattern (where types and dependencies live). You compose them: "MVVM with Clean layering" or "MVI with Clean layering."

Preferences DataStore vs Proto DataStore

Preferences = untyped key-value, like SharedPreferences but async. Proto = a single typed schema (protobuf), better for structured app-wide settings. Use proto when you have >5 settings or want compile-time type safety.

The "decision criterion" cheat-sheet

Memorize these one-liners
  • StateFlow for VM state, SharedFlow(replay=0) for events, mutableStateOf for composable-local UI state.
  • LiveData only in legacy XML; everywhere else StateFlow.
  • MVVM by default; lift to MVI when "what's the screen showing right now" requires reading 3+ flows.
  • @Binds for interface→impl, @Provides for configured types.
  • Constructor injection for everything you control; field injection only for Android components.
  • Room for queryable data, DataStore for settings, EncryptedSharedPreferences for tokens.
  • Kotlinx Serialization for new code; Gson only in legacy.
  • WorkManager for deferrable; Foreground Service for user-visible long work.
  • singleTop dedupes when already on top; singleTask dedupes across the task.
  • Compose for new UI; XML for legacy maintenance and very-custom Views.
  • HLS+CMAF+Widevine = one packaging serves Android + iOS.
