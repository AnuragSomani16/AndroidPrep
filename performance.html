Performance & Profiling

Cold start, jank, memory leaks, ANRs, APK size, Baseline Profiles, Macrobenchmark — and the tooling workflows seniors use to find the regression in 10 minutes.

cold / warm / hot start baseline profiles R8 + resource shrinking 16.6ms frame budget (60Hz) StrictMode in debug LeakCanary JankStats + Macrobenchmark app startup tracing Time to Initial Display (TTID) Time to Full Display (TTFD)
Cold start = process newly created (first launch, or after Android killed it). Application.onCreate runs, classes load, resources inflate. Slowest. Warm start = process exists, but the Activity was destroyed; just onCreate runs again. Hot start = process AND Activity exist; just brought to foreground. Fastest. Baseline Profile = compile-hint file shipped in the APK so Android Runtime (ART) pre-compiles your hot startup paths AOT instead of JIT-ing them on first run.

2. Cold start — every win in priority order

  1. Baseline Profiles — biggest single win. Ships AOT-compile hints; ART pre-compiles your hot startup paths. Often 20-40% improvement on cold start.
  2. Trim Application.onCreate — every ms here is cold start. Defer non-critical SDK init (Crashlytics, Analytics, AdMob) via App Startup library or lifecycleScope.launch(Dispatchers.IO) after first frame.
  3. R8 + resource shrinking — smaller APK, faster class loading. minifyEnabled true; shrinkResources true in release.
  4. Lazy DI graph — Hilt's @Singleton are lazy by default. Don't reference unnecessary singletons in onCreate.
  5. Splash screen API — Android 12+ SplashScreen compat lib. Don't draw your own splash.
  6. Avoid blocking I/O on main — synchronous SharedPreferences, Disk I/O, parsing JSON.
  7. Network layer init off-main — Retrofit is fine to construct on main, but don't make first network call inline.

Baseline Profiles in 60 seconds

// Generate via macrobenchmark:
@RunWith(AndroidJUnit4::class)
class BaselineProfileGenerator {
    @get:Rule val rule = BaselineProfileRule()
    @Test fun generate() = rule.collect(packageName = "com.example.app") {
        pressHome()
        startActivityAndWait()
        // simulate critical user journey:
        device.findObject(By.text("Home")).click()
        device.wait(Until.hasObject(By.text("Featured")), 5000)
    }
}
// Run on rooted/debug build → produces baseline-prof.txt → check into src/main/baselineProfiles/

Defer SDK init via App Startup

class CrashlyticsInitializer : Initializer<Unit> {
    override fun create(ctx: Context) {
        FirebaseApp.initializeApp(ctx)
        FirebaseCrashlytics.getInstance().setCrashlyticsCollectionEnabled(true)
    }
    override fun dependencies() = emptyList<Class<out Initializer<*>>>()
}

3. Frame timing & jank

DisplayPer-frame budget
60 Hz16.6 ms
90 Hz11.1 ms
120 Hz8.3 ms

Common jank sources

  • Heavy work on main thread — JSON parsing, large bitmap decode, expensive measure/layout passes.
  • Compose: unstable parameters causing entire subtrees to recompose.
  • RecyclerView onBindViewHolder doing I/O or heavy work.
  • Overdraw — too many opaque layers stacked. Use Layout Inspector → "Show overdraw".
  • Bitmap loading without sampling. Use Coil/Glide.

Tools

  • JankStats library — runtime jank detection in production-like builds.
  • Profiler → System Trace — see frame timeline, missed deadlines.
  • Layout Inspector → Recomposition counts (Compose).
  • Choreographer.FrameCallback — programmatic frame timing.
val jankStats = JankStats.createAndTrack(window) { frameData ->
    if (frameData.isJank) {
        log("jank ${frameData.frameDurationUiNanos / 1_000_000}ms")
    }
}
override fun onResume() { super.onResume(); jankStats.isTrackingEnabled = true }
override fun onPause() { jankStats.isTrackingEnabled = false; super.onPause() }

4. Memory leaks

The classics

  • Holding Activity/Context in a Singleton.
  • Anonymous inner classes (Handler, Runnable) holding implicit outer reference.
  • Listeners registered but never unregistered.
  • Coroutine scope leaking (e.g., GlobalScope.launch capturing Activity).
  • ExoPlayer not released.
  • RxJava subscriptions not disposed (legacy).
  • Bitmaps cached unboundedly.

LeakCanary in debug builds

// build.gradle (debug only)
debugImplementation("com.squareup.leakcanary:leakcanary-android:2.13")
// no further setup needed — auto-installs and notifies on leak

Heap dump workflow

  1. Reproduce: do the suspected leak action 5x (rotate, navigate in/out).
  2. Profiler → Memory tab → "Capture heap dump."
  3. Group by class, look for > 1 instance of your Activity / ViewModel / Compose Composer.
  4. "Show nearest GC root" → find the chain pinning the leaked object.

5. ANRs (Application Not Responding)

Triggered when:

  • UI thread blocked > 5 seconds for input.
  • BroadcastReceiver onReceive > 10 seconds.
  • Service onStart > 20 seconds.

StrictMode for early detection (debug only)

if (BuildConfig.DEBUG) {
    StrictMode.setThreadPolicy(StrictMode.ThreadPolicy.Builder()
        .detectDiskReads()
        .detectDiskWrites()
        .detectNetwork()
        .penaltyLog()
        .build())
    StrictMode.setVmPolicy(StrictMode.VmPolicy.Builder()
        .detectLeakedClosableObjects()
        .detectActivityLeaks()
        .penaltyLog()
        .build())
}

Reading ANR traces

From /data/anr/traces.txt (root) or Play Console "Vitals." Look at the main thread stack at the moment of ANR — usually points to whatever blocked it.

6. APK / app size

  • R8 — obfuscation + shrinking. Default enabled in release.
  • Resource shrinking: shrinkResources true.
  • App Bundles (.aab) — Play Store builds per-device APKs (only architectures, density buckets, languages relevant to that device).
  • WebP for images, vector drawables for icons.
  • Dynamic feature modules (Play Feature Delivery) for rarely-used features.
  • Audit dependencies — large transitives like Guava can be replaced with smaller alternatives.

Inspect APK / AAB contents

./gradlew bundleRelease
bundletool build-apks --bundle=app-release.aab --output=apks.apks
unzip apks.apks
# Or in Android Studio: Build → Analyze APK

7. Network performance

  • HTTP/2 + connection pooling — OkHttp does this by default.
  • Compression — Accept-Encoding: gzip handled automatically.
  • Batch endpoints — collapse N small calls into 1 batched call.
  • Caching — Cache-Control headers, OkHttp HTTP cache, app-level cache.
  • Reduce payload — server-side: GraphQL, sparse fieldsets, pagination size right-sized.
  • Prefetch hot data on cold start while user is on splash/auth screen.

8. Compose-specific performance

  • Stable parameters → recomposition skipping.
  • derivedStateOf to gate downstream recomposition (e.g., scroll position →"isAtTop" boolean).
  • key = { it.id } and contentType in lazy lists.
  • Don't allocate lambdas in body when passed to stable composables — extract.
  • Avoid Modifier.composed where a direct modifier works (composed allocates).
  • Compose Compiler reports → identify non-skippable composables.

9. Profiler workflows you should know

Workflow A — "the screen is sluggish"

  1. Profiler → CPU → "System Trace" recording.
  2. Reproduce the slowness, stop recording.
  3. Find frames with red markers (jank).
  4. Zoom in. Look at main thread. The biggest red bar is the culprit.
  5. If Compose: check Layout Inspector recomposition counts. Run Compose Compiler reports.

Workflow B — "the app is OOMing"

  1. Reproduce → Profiler → Memory → record allocations during.
  2. Look at allocation graph. Spike? Steady growth? Large bitmaps?
  3. Heap dump at peak. Group by class. Anything > 100MB? Anything growing unboundedly?
  4. For bitmap-related: confirm via "Bitmap" filter — usually the cache or unbounded image loading.

Workflow C — "cold start regressed"

  1. Macrobenchmark StartupBenchmark — capture before/after numbers.
  2. System trace during cold start. Look at Application.onCreate, Activity.onCreate, first inflate.
  3. Identify any new SDK init or large class init pulled in by the change.
  4. Defer or move off-main.

12. Common interview probes

Q. How do you measure cold start?

adb shell am start -W for quick reads. Macrobenchmark StartupBenchmark for stable, CI-friendly numbers. Profiler "System Trace" for one-time deep dive.

Q. What's TTID vs TTFD?

Time to Initial Display — first frame drawn. Time to Full Display — content fully populated. Mark with reportFullyDrawn(). Macrobenchmark distinguishes them.

Q. How do Baseline Profiles work?

Pre-AOT-compile hints shipped in the APK. ART pre-compiles those classes/methods so first execution doesn't pay JIT cost. Generated via Macrobenchmark recording the critical path.

Q. Why does my LazyColumn recompose unnecessarily?

(1) Missing key; (2) unstable item parameter (e.g., a List field); (3) lambda allocated per composition. Run Compose Compiler reports to confirm.

Q. Strategies to reduce APK size?

R8 + shrinkResources, App Bundles, WebP/vector drawables, audit dependencies, dynamic feature modules. Bonus: language-specific splits via Bundles drop unused locales per device.
