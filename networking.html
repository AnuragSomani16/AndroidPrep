Networking & Auth

Retrofit + interceptors, JSON parsing libraries compared, REST patterns, OAuth 2.0 + PKCE, refresh-token rotation, and secure token storage with full code.

Interceptor vs Authenticator OAuth 2.0 with PKCE refresh token rotation EncryptedSharedPreferences SSL pinning via CertificatePinner kotlinx serialization (KSP, no reflection) logging interceptor (debug-only body) connection pool tuning idempotent retries on POST 5xx vs 4xx handling
Retrofit = a Square library that turns a Kotlin interface into an HTTP client. You annotate methods (@GET, @POST); Retrofit generates a runtime proxy. OkHttp = the underlying HTTP client. Connection pooling, gzip, HTTP/2, redirects, caching, interceptors. Interceptor = a function in OkHttp's pipeline that runs on every request (add headers, log, retry). Authenticator = a function that runs on a 401 response specifically — refreshes the token and retries the original request.

1. Why Retrofit + OkHttp

Senior framing

"Retrofit converts Kotlin interfaces into HTTP clients at runtime via dynamic proxies. OkHttp under the hood handles connection pooling, gzip, HTTP/2, redirects, and caching. Together they give me type-safe API surfaces, interceptor pipelines for cross-cutting concerns (auth, logging, telemetry), and first-class coroutine integration (suspend functions return parsed types directly). Their replaceability matters too — I can swap the JSON converter without touching API code."

@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {
    @Provides @Singleton
    fun json() = Json {
        ignoreUnknownKeys = true        // forward-compat: server adds field, we don't crash
        coerceInputValues = true        // null in non-null field falls back to default
        explicitNulls = false           // don't write nulls in serialized output
    }

    @Provides @Singleton
    fun loggingInterceptor() = HttpLoggingInterceptor().apply {
        level = if (BuildConfig.DEBUG) HttpLoggingInterceptor.Level.BODY
                else HttpLoggingInterceptor.Level.NONE         // never BODY in release — leaks PII
    }

    @Provides @Singleton
    fun okHttp(
        auth: AuthInterceptor,
        authenticator: TokenAuthenticator,
        logging: HttpLoggingInterceptor
    ): OkHttpClient = OkHttpClient.Builder()
        .addInterceptor(auth)              // appends Authorization: Bearer ...
        .authenticator(authenticator)       // refreshes on 401
        .addInterceptor(logging)
        .connectTimeout(15, TimeUnit.SECONDS)
        .readTimeout(20, TimeUnit.SECONDS)
        .writeTimeout(20, TimeUnit.SECONDS)
        .retryOnConnectionFailure(true)
        .build()

    @Provides @Singleton
    fun retrofit(client: OkHttpClient, json: Json): Retrofit = Retrofit.Builder()
        .baseUrl("https://api.fox.com/")
        .client(client)
        .addConverterFactory(json.asConverterFactory("application/json".toMediaType()))
        .build()

    @Provides @Singleton fun userApi(r: Retrofit): UserApi = r.create()
}

3. Interceptor vs Authenticator — locked-in distinction

InterceptorAuthenticator
RunsEvery requestOnly on 401 response
Use forAdd headers, logging, metrics, retry on 5xxRefresh access token, retry original request
Network or AppBoth flavorsApp-level only

AuthInterceptor — adds the Bearer token

class AuthInterceptor @Inject constructor(
    private val tokens: TokenStore
) : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        val token = runBlocking { tokens.accessToken() }     // OK in interceptor (off main thread)
        val req = if (token != null) {
            chain.request().newBuilder()
                .header("Authorization", "Bearer $token")
                .build()
        } else chain.request()
        return chain.proceed(req)
    }
}

TokenAuthenticator — refresh on 401

@Singleton
class TokenAuthenticator @Inject constructor(
    private val tokens: TokenStore,
    private val authApi: dagger.Lazy<AuthApi>    // Lazy to break Retrofit cycle
) : Authenticator {

    private val refreshMutex = Mutex()

    override fun authenticate(route: Route?, response: Response): Request? {
        val expiredToken = response.request.header("Authorization")
            ?.removePrefix("Bearer ") ?: return null

        return runBlocking {
            refreshMutex.withLock {
                // Another in-flight 401 may have already refreshed
                val current = tokens.accessToken()
                if (current != null && current != expiredToken) {
                    return@withLock buildRequest(response.request, current)
                }

                val refresh = tokens.refreshToken() ?: return@withLock null
                val resp = runCatching { authApi.get().refresh(RefreshDto(refresh)) }.getOrNull()
                    ?: run { tokens.clear(); return@withLock null }

                tokens.save(resp.accessToken, resp.refreshToken)
                buildRequest(response.request, resp.accessToken)
            }
        }
    }

    private fun buildRequest(orig: Request, token: String) =
        orig.newBuilder().header("Authorization", "Bearer $token").build()
}
Walk through what happens when the access token expires:
  1. App makes a request. AuthInterceptor attaches the (expired) Bearer token.
  2. Server responds 401 Unauthorized. OkHttp invokes TokenAuthenticator.authenticate().
  3. The authenticator acquires the mutex. If 5 parallel requests all 401'd at once, only the first will hold the lock; the rest wait.
  4. Inside the lock, it checks current token vs the expired one. If they differ, another caller already refreshed — just retry with the new token.
  5. If still stale, call /refresh with the refresh token. On success, save the new pair. On failure, clear tokens (force re-login) by returning null.
  6. Return a new Request with the fresh Bearer token. OkHttp retries automatically.
The Mutex matters: without it, all 5 parallel 401s would hammer /refresh simultaneously, possibly invalidating each other's tokens (server-side rotation).
Senior framing

"Refresh under a Mutex — multiple parallel requests hitting 401 simultaneously would each try to refresh. The mutex makes only the first one refresh; the rest see the new token and retry."

Why GSON exists / "what is Gson?"

Google's classic JSON serializer for Java/Android. JSON is a format; Gson is a library that converts between JSON text and Java/Kotlin objects. It uses runtime reflection — flexible but slower and Kotlin-unaware. Code: gson.fromJson(jsonString, User::class.java).

"Why GSON?" — interview answer

"Honest answer: legacy. For a new Kotlin app I'd never pick Gson. It uses runtime reflection (slower cold start, larger app), doesn't understand Kotlin nullability or default args (you can hit "null in non-null field" crashes), and doesn't handle sealed classes natively. Kotlinx Serialization with KSP generates parsers at compile time — faster, type-safe, multiplatform-ready. I keep Gson around in legacy modules where I haven't migrated yet."

6. Error handling pattern

sealed interface ApiResult<out T> {
    data class Success<T>(val data: T) : ApiResult<T>
    data class HttpError(val code: Int, val body: String?) : ApiResult<Nothing>
    data class NetworkError(val cause: Throwable) : ApiResult<Nothing>
}

suspend inline fun <T> safeApiCall(crossinline call: suspend () -> T): ApiResult<T> = try {
    ApiResult.Success(call())
} catch (e: HttpException) {
    ApiResult.HttpError(e.code(), e.response()?.errorBody()?.string())
} catch (e: IOException) {
    ApiResult.NetworkError(e)
} catch (e: CancellationException) {
    throw e   // never swallow!
} catch (e: Throwable) {
    ApiResult.NetworkError(e)
}

// usage
suspend fun fetchUser(id: String): ApiResult<User> = safeApiCall { api.user(id).toDomain() }

8. SSL / Certificate pinning

val pinner = CertificatePinner.Builder()
    // pin to the SPKI hash of the intermediate, not the leaf — survives leaf rotation
    .add("api.fox.com", "sha256/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=")
    .add("api.fox.com", "sha256/BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB=")  // backup pin
    .build()

OkHttpClient.Builder()
    .certificatePinner(pinner)
    .build()
Pinning operational burden

Pinning protects against rogue CAs / MITM but can brick your app if the cert rotates and your release lags. Always pin to multiple keys (current + backup) and have a kill-switch via Remote Config to disable pinning if needed. Senior signal: knowing this tradeoff.

9. OAuth 2.0 with PKCE — full implementation

Why PKCE (Proof Key for Code Exchange)

The mobile-safe flavor of OAuth 2.0. Mobile apps can't keep a static client secret safe (it ships in the APK). PKCE replaces the secret with a per-flow random "verifier" that the server only sees in the second step. Even if attackers intercept the auth code, they can't exchange it without the verifier.

The flow

1. App generates code_verifier (random 43-128 chars).
2. App computes code_challenge = BASE64URL(SHA256(code_verifier)).
3. App opens Chrome Custom Tab to:
   https://auth.example.com/authorize?
       response_type=code
       &client_id=mobile
       &redirect_uri=myapp://callback
       &code_challenge=<challenge>
       &code_challenge_method=S256
       &state=<random>
4. User logs in. Server redirects to: myapp://callback?code=xyz&state=<random>
5. App POST /token with code + code_verifier:
   { code, code_verifier, redirect_uri, client_id }
6. Server validates SHA256(code_verifier) == code_challenge and returns:
   { access_token, refresh_token, expires_in }
7. App stores tokens in EncryptedSharedPreferences.

Code verifier + challenge generation

fun generateCodeVerifier(): String {
    val bytes = ByteArray(64).also { SecureRandom().nextBytes(it) }
    return Base64.encodeToString(bytes, Base64.URL_SAFE or Base64.NO_PADDING or Base64.NO_WRAP)
}

fun codeChallenge(verifier: String): String {
    val sha = MessageDigest.getInstance("SHA-256").digest(verifier.toByteArray(Charsets.US_ASCII))
    return Base64.encodeToString(sha, Base64.URL_SAFE or Base64.NO_PADDING or Base64.NO_WRAP)
}

Open Chrome Custom Tab

class AuthScreen(private val ctx: Context, private val tokens: TokenStore) {
    fun startLogin() {
        val verifier = generateCodeVerifier()
        val challenge = codeChallenge(verifier)
        val state = UUID.randomUUID().toString()
        tokens.savePending(verifier, state)

        val url = Uri.parse("https://auth.fox.com/authorize").buildUpon()
            .appendQueryParameter("response_type", "code")
            .appendQueryParameter("client_id", "mobile-android")
            .appendQueryParameter("redirect_uri", "myapp://callback")
            .appendQueryParameter("code_challenge", challenge)
            .appendQueryParameter("code_challenge_method", "S256")
            .appendQueryParameter("state", state)
            .appendQueryParameter("scope", "openid profile offline_access")
            .build()

        CustomTabsIntent.Builder().build().launchUrl(ctx, url)
    }
}

Handle redirect deep link

// AndroidManifest
<activity android:name=".AuthCallbackActivity">
    <intent-filter>
        <action android:name="android.intent.action.VIEW"/>
        <category android:name="android.intent.category.DEFAULT"/>
        <category android:name="android.intent.category.BROWSABLE"/>
        <data android:scheme="myapp" android:host="callback"/>
    </intent-filter>
</activity>

@AndroidEntryPoint
class AuthCallbackActivity : AppCompatActivity() {
    @Inject lateinit var tokens: TokenStore
    @Inject lateinit var authApi: AuthApi

    override fun onCreate(b: Bundle?) {
        super.onCreate(b)
        val data = intent.data ?: run { finish(); return }
        val code = data.getQueryParameter("code")
        val returnedState = data.getQueryParameter("state")
        val pending = tokens.pending() ?: run { finish(); return }
        require(returnedState == pending.state) { "state mismatch — possible CSRF" }

        lifecycleScope.launch {
            runCatching {
                val resp = authApi.token(
                    TokenRequest(
                        grantType = "authorization_code",
                        code = code!!,
                        codeVerifier = pending.verifier,
                        redirectUri = "myapp://callback",
                        clientId = "mobile-android"
                    )
                )
                tokens.save(resp.accessToken, resp.refreshToken)
                tokens.clearPending()
            }.onSuccess { startActivity(Intent(this@AuthCallbackActivity, MainActivity::class.java)) }
             .onFailure { showError(it) }
            finish()
        }
    }
}

10. Secure token storage with EncryptedSharedPreferences

class TokenStore @Inject constructor(@ApplicationContext private val ctx: Context) {

    private val prefs by lazy {
        val masterKey = MasterKey.Builder(ctx)
            .setKeyScheme(MasterKey.KeyScheme.AES256_GCM)
            .build()

        EncryptedSharedPreferences.create(
            ctx,
            "secure_tokens",
            masterKey,
            EncryptedSharedPreferences.PrefKeyEncryptionScheme.AES256_SIV,
            EncryptedSharedPreferences.PrefValueEncryptionScheme.AES256_GCM
        )
    }

    suspend fun accessToken(): String? = withContext(Dispatchers.IO) { prefs.getString("access", null) }
    suspend fun refreshToken(): String? = withContext(Dispatchers.IO) { prefs.getString("refresh", null) }

    suspend fun save(access: String, refresh: String) = withContext(Dispatchers.IO) {
        prefs.edit().putString("access", access).putString("refresh", refresh).apply()
    }

    suspend fun clear() = withContext(Dispatchers.IO) { prefs.edit().clear().apply() }

    // Pending PKCE state (verifier + state) during auth
    fun savePending(verifier: String, state: String) {
        prefs.edit().putString("pending_v", verifier).putString("pending_s", state).apply()
    }
    fun pending(): Pending? {
        val v = prefs.getString("pending_v", null) ?: return null
        val s = prefs.getString("pending_s", null) ?: return null
        return Pending(v, s)
    }
    fun clearPending() = prefs.edit().remove("pending_v").remove("pending_s").apply()

    data class Pending(val verifier: String, val state: String)
}
Senior framing

"Tokens go in EncryptedSharedPreferences — backed by Android Keystore so the encryption key is hardware-isolated on devices that support it. Never store tokens in plain SharedPreferences or files. Refresh tokens are sensitive — if they leak, an attacker has long-lived access. Refresh token rotation on the server side mitigates this: each refresh issues a new refresh token and revokes the old one."

12. Common networking gotchas

Q. Why am I getting CleartextTrafficException on Android 9+?

API 28+ defaults to HTTPS-only. Either fix to HTTPS (correct) or override via Network Security Config cleartextTrafficPermitted="true" for specific dev domains. Never ship cleartext to prod.

Q. My HttpLoggingInterceptor is leaking tokens to logcat in release.

Set level = NONE in release. Even better: gate via BuildConfig.DEBUG. Or use a custom interceptor that redacts the Authorization header.

Q. How do I retry transient 5xx but not 4xx?

An Interceptor: catch the response, check code in 500..599, sleep with exponential backoff, retry up to N times. Don't retry 4xx (client error — won't fix itself). Don't retry POST without idempotency keys.

Q. Connection timeout vs read timeout vs write timeout?

Connect: TCP handshake. Read: time between bytes received. Write: time between bytes sent. Defaults are reasonable; tune read for slow APIs.

Q. Authenticator infinite-loops on 401.

Check response.priorResponse count and bail after N attempts. Also: if refresh itself returned 401 (refresh token invalid), clear tokens and force re-login — don't retry.
