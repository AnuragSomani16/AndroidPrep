Kotlin — basics to advanced

Everything from null safety to reified generics, with OOP, idiomatic patterns, and "spot the bug" drills you can practice with collapsible answers.

1. Why Kotlin over Java (have this answer locked)

Senior framing

Kotlin gives you null safety at the type system level, data classes that eliminate boilerplate, coroutines for structured async, sealed hierarchies for exhaustive state modeling, and extension functions for clean API surfaces. On the JVM it's 100% interoperable with Java; on Android it's the official language since 2019. The combined effect is roughly 30-40% less boilerplate, fewer NPEs, and significantly more expressive concurrency.

Concrete differences worth saying out loud

  • Null safety: String? vs String — Java has no compile-time NPE protection. Compile-time null checks.
  • Data classes: 1 line vs 50+ lines of Java POJO with equals/hashCode/toString.
  • Coroutines: structured concurrency vs callback hell or RxJava's complexity.
  • Extension functions: extend types you don't own, statically dispatched (no runtime cost).
  • Smart casts: after a type check, the variable is auto-cast in that branch.
  • Default + named args: kills builder pattern boilerplate.
  • Type inference: less ceremony, same type safety.
  • Higher-order functions + lambdas: first-class, with inline to avoid object allocation.

2. Variables, types, and null safety

Null safety = Kotlin's type system distinguishes types that can hold null (String?) from those that can't (String). The compiler enforces a null check before any access on a nullable. The result: most NullPointerExceptions become impossible at compile time. Operators: ?. (safe call — returns null if receiver is null), ?: (Elvis — fallback for null), !! (force-unwrap, throws if null — code smell).
val a: Int = 5            // immutable reference (still mutable object if it's mutable)
var b: Int = 5            // mutable
const val PI = 3.14       // compile-time constant; only top-level / object / companion
val name: String? = null  // nullable
val len = name?.length ?: 0   // safe call + Elvis
name?.let { println(it) }     // scope only if non-null

// Smart cast
fun describe(x: Any) {
    if (x is String) println(x.length)   // x auto-cast to String here
}
Avoid !! in production

The !! operator throws NullPointerException if null. Acceptable in tests; dangerous in app code. Better: ?.let { }, ?: error("expected non-null"), or checkNotNull(x) { "msg" } which signals intent.

3. Classes, OOP, and inheritance

Class basics — primary constructor, init, secondary

class User(val id: String, val name: String) {       // primary constructor
    init { require(id.isNotBlank()) { "id required" } }
    constructor(name: String) : this(UUID.randomUUID().toString(), name)  // secondary
}

Visibility modifiers

  • public (default) — visible everywhere.
  • internal — visible within the same Gradle module. Module boundary — useful in multi-module builds.
  • protected — class + subclasses (no top-level use).
  • private — file (top-level) or class (member).

Inheritance — open/final by default

open class Animal(val name: String) {
    open fun speak() = println("...")
}
class Dog(name: String) : Animal(name) {
    override fun speak() = println("woof")
}

Kotlin classes and functions are final by default. Mark open to allow inheritance. This is a deliberate design — favor composition over inheritance means you should rarely need open in app code.

Abstract classes vs interfaces

interface Repository<T> {
    fun fetch(): T
    fun cached(): T = fetch()   // default impl OK in interface
}

abstract class BaseVm : ViewModel() {
    abstract val state: StateFlow<UiState>
    protected fun handleError(e: Throwable) { /* shared */ }
}

Data class — structural equality for free

data class User(val id: String, val name: String, val email: String?)

val a = User("1", "Anu", "a@x.com")
val b = a.copy(name = "Anurag")        // generated copy()
val (id, name, email) = a              // destructuring via componentN()
println(a == b)                        // structural equality (false)
println(a === a.copy())                // referential (false)
Data class gotchas

(1) Must have at least one constructor parameter. (2) Cannot be abstract, open, sealed, or inner. (3) Non-constructor var properties are NOT included in copy/equals/hashCode — that's a sneaky bug source.

Sealed class / sealed interface

sealed interface UiState {
    data object Loading : UiState
    data class Success(val items: List<Item>) : UiState
    data class Error(val cause: Throwable) : UiState
}

fun render(state: UiState) = when (state) {        // exhaustive — compiler enforces
    UiState.Loading      -> showSpinner()
    is UiState.Success   -> showItems(state.items)
    is UiState.Error     -> showError(state.cause)
}
Why this pattern wins: each subclass can carry different data (Success has items, Error has a cause). The compiler knows the full set of subclasses and forces you to handle every case in when — so adding a fourth state (e.g., Empty) becomes a compile error in every consumer until you handle it. This is the foundation of MVI state modeling: one closed type for "what could the screen possibly be showing right now."

Sealed hierarchies for exhaustive when is the foundation of MVI state modeling.

Object, companion, @JvmStatic

object Singleton {                          // thread-safe lazy singleton
    fun init() { /* ... */ }
}

class User private constructor(val id: String) {
    companion object {                       // also a singleton, accessed via User.create()
        @JvmStatic fun create(id: String) = User(id)
    }
}

Inner vs nested class

class Outer {
    inner class Inner { fun ref() = this@Outer }   // implicit ref to Outer
    class Nested { /* no ref to Outer */ }
}
Inner class memory leaks

inner class holds an implicit reference to the outer instance. Common Android leak: an Activity's inner Handler/AsyncTask outliving the Activity. Prefer class Nested (default) and pass references explicitly.

4. Generics — variance and reified

Declaration-site variance

// out — covariant: producer of T
class Box<out T>(val item: T)
val animals: Box<Animal> = Box<Dog>(Dog())   // ✅ Box<Dog> is-a Box<Animal>

// in — contravariant: consumer of T
interface Comparator<in T> { fun compare(a: T, b: T): Int }
val animalCmp: Comparator<Dog> = AnimalComparator   // ✅

Reified generics — only inside inline functions

// without reified — you'd need to pass the Class<T>
inline fun <reified T> fromJson(json: String): T = gson.fromJson(json, T::class.java)

val user: User = fromJson(jsonString)   // T is known at runtime!

5. Functions — extensions, infix, operator, default args

// Extension function
fun String.isEmail(): Boolean = matches(Regex("..."))

// Extension property
val Int.dp: Float get() = this * Resources.getSystem().displayMetrics.density

// Infix
infix fun Int.times(action: () -> Unit) { repeat(this) { action() } }
3 times { println("hi") }

// Operator overload — invoke for use cases
class GetUserUseCase @Inject constructor(private val repo: UserRepo) {
    suspend operator fun invoke(id: String): User = repo.fetch(id)
}
// usage: getUser(id)   not getUser.execute(id)

// Default + named args
fun newRequest(
    url: String,
    method: String = "GET",
    timeout: Long = 30_000,
    retries: Int = 3
) = Request(url, method, timeout, retries)
newRequest("/users", retries = 5)
Senior phrase

"Extension functions are statically dispatched — resolved at compile time based on the static type. A member function with the same signature wins. This is why I avoid extending sealed type hierarchies — easy to write something that looks polymorphic but isn't."

6. Lambdas, higher-order, and inline

// Lambda
val add: (Int, Int) -> Int = { a, b -> a + b }

// Higher-order with single-lambda parameter — trailing lambda syntax
fun <T> Iterable<T>.eachSafely(action: (T) -> Unit) {
    forEach { try { action(it) } catch (_: Exception) {} }
}
items.eachSafely { send(it) }

// inline — avoids lambda object allocation; supports non-local returns
inline fun <T> measureMillis(block: () -> T): T {
    val start = System.nanoTime()
    return block().also { println((System.nanoTime() - start) / 1_000_000) }
}

// crossinline — when the lambda must NOT have a non-local return
inline fun launchWith(crossinline action: () -> Unit) {
    Thread { action() }.start()   // can't return from outer here
}

// noinline — opt out for one parameter
inline fun foo(crossinline a: () -> Unit, noinline b: () -> Unit) { /* ... */ }

8. Delegation — both flavors

Class delegation — composition over inheritance

interface UserRepo { suspend fun fetch(id: String): User }

class CachingUserRepo(
    private val backing: UserRepo
) : UserRepo by backing {
    private val cache = mutableMapOf<String, User>()
    override suspend fun fetch(id: String): User =
        cache[id] ?: backing.fetch(id).also { cache[id] = it }
}

Property delegation

class Settings(prefs: SharedPreferences) {
    var darkMode: Boolean by prefs.boolean("dark_mode", false)
    val deviceId: String by lazy { generateId() }
    var name: String by Delegates.observable("") { _, old, new ->
        log("name $old -> $new")
    }
}

9. Collections — Kotlin idioms

val nums = listOf(1, 2, 3, 4, 5)
nums.filter { it % 2 == 0 }.map { it * it }.sum()              // 20
nums.fold(0) { acc, n -> acc + n }                              // 15
nums.groupBy { it % 2 }                                         // {1=[1,3,5], 0=[2,4]}
nums.partition { it > 2 }                                       // ([3,4,5], [1,2])
nums.windowed(3, step = 1)                                      // [[1,2,3],[2,3,4],[3,4,5]]
nums.chunked(2)                                                 // [[1,2],[3,4],[5]]
nums.zipWithNext()                                              // [(1,2),(2,3),(3,4),(4,5)]
nums.associate { it to it * it }                                // {1=1, 2=4, 3=9, 4=16, 5=25}
nums.asSequence().filter { it > 2 }.map { it * 2 }.toList()    // lazy chain
Sequence vs Iterable

Iterable: each operator allocates a new list. Sequence: lazy, terminal-driven. Use Sequences for big collections with multiple intermediate ops; for small collections, plain Iterable is faster (no overhead).

10. Equality and identity

data class P(val x: Int)
val a = P(1); val b = P(1)
a == b      // true  (calls equals)  — structural
a === b     // false (different objects) — referential
a == a.copy()   // true
a === a.copy()  // false

11. Value classes (formerly inline classes)

@JvmInline
value class UserId(val raw: String)

@JvmInline
value class Email(val raw: String) {
    init { require(raw.contains("@")) { "invalid email" } }
}

fun fetch(id: UserId, email: Email) { /* type-safe; no boxing at runtime when erasable */ }

Value classes wrap a single value with no runtime overhead in most cases. Use them for type-safe primitives — preventing "fetch(email, userId)" mix-ups at compile time.

13. Functional patterns: Result and runCatching

val r: Result<User> = runCatching { repo.fetch(id) }
r.onSuccess { user -> bind(user) }
 .onFailure { e -> showError(e) }
val u = r.getOrElse { fallback }

Don't return Result<T> from public APIs — Kotlin's official guidance is to prefer sealed classes for domain results. runCatching is fine internally for catching-and-mapping.

14. Spot the bug — drills (try first, then expand)

Drill 1 — what's wrong? 
data class User(val id: String) {
    var lastSeen: Long = 0
}
val a = User("1"); a.lastSeen = 100
val b = User("1"); b.lastSeen = 200
println(a == b)  // ?

Answer: Prints true. lastSeen is a var body property — NOT in the primary constructor — so it's NOT included in generated equals/hashCode/copy. This is a notorious bug source. Fix by moving lastSeen into the constructor (data class User(val id: String, var lastSeen: Long = 0)) or making it val in the body if it's truly derived.

Drill 2 — type inference 
val list = listOf(1, "a", 3.14)
// what is list's type?

Answer: List<Any>. The least-common-supertype is Any (or Comparable<*> & java.io.Serializable in Kotlin's older inference). Avoid this — be explicit.

Drill 3 — null safety 
fun greet(name: String?) {
    if (name != null) {
        Thread { println(name.length) }.start()  // compile error?
    }
}

Answer: Compile error. Smart cast to non-null requires that the variable can't change between check and use. Inside a lambda passed to another thread, Kotlin can't prove that — name could be reassigned (it can't here, it's a parameter — but smart-cast is conservative across closure boundaries for var; for val/parameter it actually works in many cases). The safe form: name?.let { Thread { println(it.length) }.start() }.

Drill 4 — extension dispatch 
open class A { open fun hi() = "A" }
class B : A() { override fun hi() = "B" }
fun A.greet() = "Hi from A"
fun B.greet() = "Hi from B"
val a: A = B()
println(a.hi())     // ?
println(a.greet())  // ?

Answer: "B" then "Hi from A". Member functions are dynamically dispatched (virtual). Extension functions are statically dispatched based on the declared type — a is declared A, so A.greet() wins.

Drill 5 — equality of nullable 
val x: String? = null
val y: String? = null
println(x == y)
println(x === y)

Answer: Both true. == on nulls returns true (both are null). === on nulls returns true because there's only one null reference.

Drill 6 — lambda capture 
fun makeCounter(): () -> Int {
    var i = 0
    return { i++ }
}
val c = makeCounter()
println(c()); println(c()); println(c())

Answer: 0, 1, 2. The lambda captures the variable by reference, not the value. In Kotlin, the var is wrapped in a Ref.IntRef object so the lambda mutates the shared state. (Same as Java's effectively-final closure but with mutability via the Ref wrapper.)

Drill 7 — apply vs also 
val intent = Intent().apply {
    putExtra("k", "v")
    setAction("X")
}.also {
    Log.d("tag", "$it")   // it is the Intent
}

Answer: Compiles fine. Inside apply, this is the Intent; inside also, it is the Intent. Both return the receiver. Common pattern: apply for config, also for logging/side effects in a chain.

Drill 8 — when exhaustiveness 
sealed class Result<out T> {
    data class Ok<T>(val value: T) : Result<T>()
    data class Err(val msg: String) : Result<Nothing>()
}
fun <T> render(r: Result<T>) {
    when (r) {                  // is this exhaustive?
        is Result.Ok -> show(r.value)
    }
}

Answer: NOT exhaustive — missing Err. Compiler will warn (or error if you assign when to a value). Fixing: add is Result.Err -> showError(r.msg). Bonus: notice Err uses Result<Nothing> — Nothing is a subtype of every type, enabling the covariant out T.

Drill 9 — companion + @JvmStatic 
class Logger {
    companion object {
        fun create() = Logger()       // Java: Logger.Companion.create()
        @JvmStatic fun build() = Logger()   // Java: Logger.build()
    }
}

Answer: Without @JvmStatic, Java callers must go through Logger.Companion. With it, the function is also exposed as a true static method on the outer class — cleaner Java interop.

Drill 10 — string templates 
val name = "Anu"
println("Hello $name, ${name.length} chars")

Answer: Hello Anu, 3 chars. $x for simple identifiers; ${expr} for expressions. Note: this compiles to StringBuilder.append, not String concatenation — slightly more efficient than Java's "Hello " + name + ... for complex templates.

15. OOP recipe — the senior way

Senior framing for "design a class"

(1) Start with an interface describing intent. (2) Implement it in a class with constructor injection. (3) Make the class internal if it's an impl detail. (4) Keep state immutable where possible (data class + copy). (5) Use sealed for closed type hierarchies, open sparingly.

interface AnalyticsTracker {
    fun screen(name: String, props: Map<String, Any> = emptyMap())
    fun event(name: String, props: Map<String, Any> = emptyMap())
}

internal class NewRelicTracker @Inject constructor(
    private val nrApi: NewRelicApi,
    @IoDispatcher private val io: CoroutineDispatcher
) : AnalyticsTracker {
    override fun screen(name: String, props: Map<String, Any>) {
        // dispatched off-main; failures logged, never thrown
    }
    /* ... */
}

class AnalyticsHub @Inject constructor(
    private val trackers: Set<@JvmSuppressWildcards AnalyticsTracker>
) : AnalyticsTracker {
    override fun screen(name: String, props: Map<String, Any>) =
        trackers.forEach { runCatching { it.screen(name, props) } }
    /* ... */
}
