Glossary — definitions with context

Every term is described with the same shape: what it is → what it replaces / when to use → why. Skim this if a topic page uses a word that suddenly seems unfamiliar.

Reading guide

If you come from XML / LiveData / Java background, focus on the "replaces" and "why" parts — they map old concepts to new ones. The cards are designed for skimming; if you want depth, the topic pages are linked from the sidebar.

Kotlin & language

val / var

What: val is read-only after init; var is reassignable. Replaces: Java's final vs non-final fields. Why: immutability is the default; reassignment is opt-in. Reading the keyword tells you whether the variable can change after this line.

const val

What: compile-time constant inlined at the call site. Only at top-level / object / companion. Only primitives + String. Replaces: public static final in Java. Why: no getter call at runtime; the value is baked into the bytecode of every consumer.

nullable type (String?)

What: a type that may hold null. Replaces: "every Java reference might be null with no compiler warning." Why: compile-time enforcement of null checks before access — eliminates most NullPointerExceptions.

safe call (?.)

What: calls a method only if receiver is non-null; returns null otherwise. Replaces: if (x != null) x.foo() else null. Why: compact, chainable, and works inline in expressions.

Elvis operator (?:)

What: provides a fallback value when the left side is null. Replaces: ternary x != null ? x : default. Why: reads naturally with safe calls — name?.length ?: 0 means "length if non-null, else 0."

!!

What: asserts non-null; throws NullPointerException if null. Replaces: nothing; it's a code smell. Why use: only in tests, or when truly impossible to be null and you've justified that comment-wise. Production code should use checkNotNull(x) { "..." } instead so the failure has a message.

smart cast

What: after if (x is String), the compiler auto-treats x as String in that branch. Replaces: Java's manual cast after instanceof. Why: less ceremony, no risk of forgetting the cast.

data class

What: auto-generates equals/hashCode/toString/copy/componentN. Replaces: Java POJO with hand-written equals/hashCode (or Lombok). Why: eliminates ~50 lines of boilerplate per type. Use for value-bearing types — DTOs, domain models, UI state.

sealed class / sealed interface

What: a type whose subclasses are all known at compile time. Replaces: "enum but each value can have different fields" — Java doesn't have a clean equivalent. Why: exhaustive when at compile time. Adding a new subclass forces every consumer to handle it. Foundation of MVI state.

object (singleton)

What: a singleton declaration. Lazily initialized, thread-safe. Replaces: Java's hand-written private-constructor + getInstance() pattern. Why: 1 line vs 10. Use when you genuinely need exactly one instance app-wide (rare; prefer DI).

companion object

What: singleton tied to a class. Replaces: Java's static members. Why: namespace for class-level constants and factory functions. Add @JvmStatic for true Java-static interop.

extension function

What: a function added to a type from outside. Replaces: Java utility classes like StringUtils.isEmail(s). Why: calls become s.isEmail() — reads as if the type owned the method. Statically dispatched (resolved at compile time on declared type).

inline function

What: compiler copies the body to each call site. Replaces: nothing equivalent in Java. Why: avoids object allocation for lambda parameters; supports non-local return. Use for high-frequency higher-order functions (forEach, let).

reified

What: type parameter accessible at runtime as T::class. Only inside inline functions. Replaces: passing Class<T> param manually like gson.fromJson(json, User::class.java). Why: cleaner call sites — fromJson<User>(json).

scope function (let/run/with/apply/also)

What: helpers that scope a block to an object. Replaces: intermediate variables and chained method calls. Why: compact, readable composition. apply/also return the receiver (chain-friendly); let/run/with return the lambda result.

lateinit var

What: non-null reference whose value will be set later. Replaces: nullable field that's "really" non-null but set in onCreate. Why: avoids !! everywhere it's accessed. Use for DI fields, view binding, test setup.

by lazy

What: read-only property computed once on first access, thread-safe. Replaces: manual double-checked locking. Why: defer expensive init until actually needed.

delegation (by)

What: forward calls to another object — for class delegation (composition) or property delegation (custom get/set). Replaces: hand-written forwarding methods. Why: class Cached(impl: Repo) : Repo by impl = "Cached IS-A Repo, forwards everything to impl, override only what you customize." Powerful for decorators.

variance (out / in)

What: declares whether a generic type is covariant (out T — producer) or contravariant (in T — consumer). Replaces: Java's ? extends T / ? super T at use site. Why: Kotlin lets you declare variance once at the type, not at every use. List<Dog> IS-A List<Animal> because List is out.

value class

What: wraps a single value with no runtime overhead in most cases. Replaces: primitives passed everywhere with no type safety (String for both userId and email). Why: compile-time prevents fetch(email, userId) from being called with args swapped — but at runtime there's no boxing on direct calls.

typealias

What: an alias for an existing type. Replaces: nothing — pure readability aid. Why: typealias OnUserSelected = (User) -> Unit reads better than (User) -> Unit sprinkled everywhere. Doesn't create a distinct type — for that, use value class.

Result<T>

What: wrapper for "succeeded or failed" with a value or throwable. Replaces: try/catch when you want to convert to a typed result. Why use: runCatching { ... }.onSuccess { }.onFailure { } reads cleanly. Don't return Result from public APIs — Kotlin official guidance is to use a sealed class instead.

Nothing

What: a type that's a subtype of every type — used for "this never returns." Replaces: nothing equivalent in Java. Why use: throw, error(), TODO() all return Nothing. Lets you write val x = nullable ?: error("missing") — the compiler knows x is non-null after this line.

Coroutines

coroutine

What: a "lightweight thread" — a unit of async work that pauses/resumes without blocking a real thread. Replaces: AsyncTask (deprecated), Thread + Handler, RxJava chains. Why: sequential code style for async work; thousands of coroutines run on a few threads; structured concurrency prevents leaks.

suspend function

What: a function that can pause/resume; only callable from another suspend or a coroutine builder. Replaces: callbacks, RxJava Single<T>, futures. Why: async code reads top-to-bottom like sync code; exceptions propagate via try/catch; no callback hell.

coroutine builder (launch / async / runBlocking)

What: functions that start a coroutine. launch = fire-and-forget (Job); async = produces a result (Deferred); runBlocking = blocks the calling thread. Replaces: new Thread().start(), ExecutorService.submit(). Why use: these are the entry points from non-suspend code into the coroutine world.

CoroutineScope

What: a scope that ties coroutines to a lifetime. Cancelling the scope cancels its coroutines. Replaces: manual onDestroy { thread.interrupt() } bookkeeping. Why: structured concurrency — no orphan coroutines outliving their owner. Use viewModelScope, lifecycleScope, or build a custom one.

Job

What: handle for a coroutine's lifecycle — cancel(), join(), isActive. Replaces: Future in Java land. Why use: sometimes you need to cancel a specific coroutine without tearing down the whole scope.

Deferred<T>

What: a Job that produces a value; call await() for the result. Replaces: Java's Future<T> / RxJava Single. Why use: for parallel work where you need both calls in flight, then collect results.

Dispatcher

What: decides which thread(s) a coroutine runs on. Main / Default / IO / Unconfined. Replaces: hand-managed ExecutorService for different work types. Why: declarative — withContext(Dispatchers.IO) { ... } just works. Inject dispatchers for testability.

structured concurrency

What: coroutines have parent-child relationships; cancellation and failures propagate. A coroutine cannot outlive its scope. Replaces: the "fire a thread, hope it cleans up" model. Why: eliminates the leak class of "forgot to cancel that background work."

SupervisorJob

What: a Job where one child's failure does NOT cancel siblings or parent. Replaces: nothing — the regular Job is the default; SupervisorJob is the exception. Why use: when independent children should keep going if one fails — think "parallel partial-success fetch."

cooperative cancellation

What: cancellation is a flag that suspending functions check. Replaces: Thread.interrupt() + checked InterruptedException. Why: safer — your code only stops at well-defined points (suspend calls, ensureActive()). CPU loops must opt in by calling ensureActive() or yield().

CoroutineExceptionHandler

What: handler for uncaught exceptions in launch. Replaces: Thread.UncaughtExceptionHandler. Why use: last-resort safety net for telemetry. Note: does NOT catch async exceptions until await() is called.

viewModelScope

What: a built-in CoroutineScope tied to ViewModel lifecycle. Cancelled in onCleared(). Replaces: manually creating + cancelling scopes. Why use: the default for VM business logic. Survives configuration change but not process death.

lifecycleScope

What: a built-in scope tied to a Lifecycle (Activity / Fragment view). Cancelled at DESTROYED. Replaces: manual onDestroy cleanup. Why use: view-bound work. In Fragments use viewLifecycleOwner.lifecycleScope not this.lifecycleScope to avoid leaks across view recreate.

Mutex

What: a coroutine-friendly lock. mutex.withLock { } serializes access without blocking the thread. Replaces: synchronized blocks (which block the thread — bad for coroutines on Main). Why: when waiting, the coroutine yields the thread to others.

Channel

What: coroutine-safe queue with send/receive. Replaces: BlockingQueue. Why use: producer/consumer with fan-out / fan-in. For most state and event needs, prefer Flow over Channel.

Flows

Flow<T>

What: a cold async stream of values. The producer block runs per collector. Replaces: RxJava's Observable, callback APIs that emit multiple values. Why: sequential composition (map/filter/etc.), backpressure, integrates with coroutines and structured concurrency.

cold vs hot

What: Cold (Flow): producer runs per collector — no collector → no work. Hot (StateFlow / SharedFlow): emit independently of collectors. Replaces: RxJava's "Observable vs Subject" distinction. Why use: Cold for one-off computations (a network call); Hot for ambient state shared by many.

StateFlow

What: a hot, conflated Flow that always has a current value, replays exactly 1 to new collectors, dedupes by equality. Replaces: LiveData for new code. Why use: ViewModel exposes UI state via StateFlow<UiState>; the View collects and renders. Compose-friendly via collectAsStateWithLifecycle(); works in pure Kotlin (KMP-ready); composes with all Flow operators (combine, flatMapLatest, debounce). LiveData is Android-only and limited to map/switchMap.

SharedFlow

What: a hot Flow with configurable replay/buffer, no equality dedupe. Replaces: RxJava PublishSubject / ReplaySubject. Why use: for events (navigation, snackbar) — set replay = 0 so a screen rotation doesn't replay yesterday's events.

conflation

What: drop intermediate values; consumers always see only the latest. Replaces: manual debounce-like throttling. Why use: for fast-changing state where you only care about the current value (e.g., scroll position). StateFlow is conflated by default.

backpressure

What: what to do when the producer is faster than the consumer. buffer(n), conflate(), collectLatest { }. Replaces: RxJava's onBackpressureBuffer/Drop/Latest. Why: prevents OOM when a fast producer drowns a slow consumer.

flowOn(dispatcher)

What: sets the dispatcher for upstream operators. Doesn't affect downstream. Replaces: RxJava's subscribeOn. Why use: shift the producer's work off Main: flow.map { heavy() }.flowOn(Default).collect { renderOnMain(it) }.

stateIn / shareIn

What: convert a cold Flow to a hot StateFlow / SharedFlow. Replaces: RxJava's publish().refCount(). Why use: share the upstream computation across multiple collectors instead of running it per-collector.

SharingStarted.WhileSubscribed(5000)

What: a sharing strategy that keeps the upstream alive 5s after the last collector unsubscribes. Replaces: manually managing subscription refcounts. Why: handles configuration change (rotation) without re-fetching — the new collector arrives within the timeout and reuses the existing flow. The standard for VM exposure.

flatMapLatest

What: for each upstream emission, cancels the previous inner Flow and starts a new one. Replaces: RxJava's switchMap. Why use: search-as-you-type — each new query cancels the in-flight search.

combine(a, b) { x, y -> ... }

What: emits whenever any source emits, using the latest from each. Replaces: RxJava's combineLatest. Why use: derive screen state from multiple flows: query + filter + selection → ScreenState.

callbackFlow

What: wraps a callback API as a Flow. awaitClose { } unregisters when collection stops. Replaces: RxJava's Observable.create. Why use: bridge old listener APIs (LocationManager, ContentResolver) into the Flow world.

collectAsStateWithLifecycle()

What: Compose helper that collects a Flow only while the host is at least STARTED. Replaces: bare collectAsState which keeps collecting in the background. Why use: save battery + avoid stale-data risks when the screen is hidden.

repeatOnLifecycle(STARTED)

What: Activity/Fragment helper that re-collects when lifecycle reaches STARTED, cancels when below. Replaces: bare lifecycleScope.launch { collect { } } which doesn't stop off-screen. Why use: XML equivalent of collectAsStateWithLifecycle.

Compose

@Composable

What: a function that emits UI when called inside the composition. Replaces: XML layouts + custom View classes. Why: declarative, plain Kotlin, testable, previewable, less boilerplate than findViewById + adapters + ViewHolders.

composition

What: the tree of @Composable calls that describes UI for a given state. Replaces: the View hierarchy in XML. Why: Compose diffs and updates underlying nodes — you describe what should be on screen, not how to mutate the existing tree.

recomposition

What: re-running composable functions when state inputs change. Replaces: manual view.setText(...), adapter.notifyDataSetChanged(). Why: UI follows state automatically; eliminates "stale view" bugs.

slot table

What: the runtime data structure tracking where each composable lives. Replaces: nothing direct — internal mechanism. Why it matters: Compose uses positional memoization — remember values are keyed by call-site position, not by name. That's why two remember { 0 } calls produce two independent values.

stable (@Stable / @Immutable)

What: a type Compose can trust: equality means rendered-output equivalence. Replaces: nothing — Compose-specific. Why use: stable parameters enable recomposition skipping — unchanged children don't re-run. Mark data classes @Immutable when they have List/Map fields you guarantee won't mutate.

state hoisting

What: push state up to the lowest common parent; pass state + onEvent down. Replaces: stateful custom Views. Why: stateless leaves are easier to preview, test, and reuse — and the parent is where state should logically live (often the ViewModel).

remember

What: stores a value across recompositions, keyed by call-site location. Replaces: Activity field. Why use: survive recomposition without putting it in ViewModel. Lost on configuration change — use rememberSaveable for that.

rememberSaveable

What: like remember but also survives configuration changes and process death (Bundle-backed). Replaces: manual onSaveInstanceState / onRestoreInstanceState. Why use: small UI state you'd be sad to lose (text input, scroll position).

LaunchedEffect(key)

What: launches a coroutine when its key changes; cancels and restarts on key change. Replaces: onResume-tied API calls. Why use: trigger side effects from the composition lifecycle.

DisposableEffect

What: side effect with a required onDispose { } cleanup. Replaces: register/unregister patterns in Activity onCreate/onDestroy. Why use: resource that needs cleanup — listeners, observers, ExoPlayer.

derivedStateOf

What: wraps a derivation so dependents recompose only when the derived result changes. Replaces: nothing equivalent in XML. Why use: val showFab by remember { derivedStateOf { listState.firstVisibleItemIndex == 0 } } — the boolean only flips when scroll crosses the threshold, not on every pixel.

rememberUpdatedState

What: stores the latest value of a parameter; long-running effects can read the latest without restarting. Replaces: nothing equivalent. Why use: fix the stale-closure bug where a LaunchedEffect(Unit) captures an old version of a callback that the parent has since re-passed.

snapshotFlow

What: turns Compose State reads into a Flow. Replaces: manually wiring up callbacks + your own Flow producer. Why use: bridge Compose state to non-Compose code — analytics, coroutine pipelines.

CompositionLocal

What: implicitly passes values down the tree. Replaces: prop-drilling shared dependencies through every composable. Why use: cross-cutting concerns (theme, density, snackbar host). Don't overuse — params are clearer for normal data.

Modifier

What: decorates a composable: padding, background, click, size. Order matters (left-to-right, outside-to-inside). Replaces: XML attributes (android:padding, etc.). Why: composable, type-safe, plain Kotlin.

Architecture

MVVM

What: ViewModel exposes UI state via observable streams; View observes; events flow back via methods. Replaces: "fat Activity" MVC where Activity owns everything. Why use: separation of UI logic from view, lifecycle-safe state container, easier to test. The default for most Android apps.

MVI

What: stricter MVVM — single State object, sealed Intents, Effects on a separate channel. Replaces: MVVM's scattered "multiple flows" pattern when state grows complex. Why use: state-heavy screens (player, multi-step forms, search). Single source of truth for what the screen is showing means easy testing and reproduction.

Clean Architecture

What: a layering pattern (presentation / domain / data) where inner layers know nothing about outer. Replaces: "everything in one module" monoliths. Why use: testable domain layer (no Android = JVM unit tests), swappable data implementations, enforced boundaries.

UDF (Unidirectional Data Flow)

What: state flows one way (down to UI), events flow back (up to VM). Replaces: two-way data binding tangles. Why use: predictable — at any moment you know what flows where; debugging is following the wire.

UseCase

What: a class encapsulating one piece of business logic, often with operator fun invoke. Replaces: business logic inside ViewModel. Why use: reusable across VMs, composable (one use case calls another), easy to unit-test in isolation. Skip when it's a passthrough to the repo.

Repository

What: single source of truth for a data type. Coordinates network + DB + cache; returns domain models. Replaces: ad-hoc API calls scattered across VMs. Why use: centralizes caching and offline-first logic; UI never sees DTOs or entities.

SDUI (Server-Driven UI)

What: server returns layout descriptors; client renders generically. Replaces: hard-coded screen layouts that need an app release to change. Why use: product can iterate without ship cycles. Tradeoff: client must support all module types in advance.

multi-module

What: splitting the app into Gradle modules. Replaces: single-module monolith. Why use: parallel compile (faster builds), enforced boundaries (build fails if UI imports data layer directly), reusability across app variants.

api vs implementation

What: Gradle dependency scopes. api exposes; implementation hides. Replaces: the old compile scope (which always exposed). Why: default to implementation to keep transitive deps invisible — consumers don't accidentally couple to your library choices.

dependency inversion

What: inner layers define interfaces; outer layers implement. Replaces: "domain depends on data" backwards arrow. Why use: domain layer becomes pure (no Android, fast tests), and you can swap implementations (network → mock) without touching domain.

Hilt / DI

dependency injection (DI)

What: a class receives its collaborators from outside instead of constructing them itself. Replaces: val foo = FooImpl() hardcoded inside classes. Why: testability (swap real for fake), decoupling (depend on interfaces), single source for "how to build X."

constructor injection

What: dependencies passed via the class's constructor. Replaces: field injection or service-locator lookups. Why use: immutable (deps are val), explicit (reading the constructor tells you everything the class needs), easy to fake in tests without DI library.

field injection

What: @Inject lateinit var dep: Dep set after construction. Replaces: nothing — used when you can't control the constructor. Why use: Android components Hilt can't construct (Activity, Fragment, View, Service, Receiver). Everywhere else, prefer constructor injection.

Hilt

What: Google's DI library for Android, built on Dagger 2. Replaces: hand-written Dagger components, service locators. Why use: compile-time graph verification (missing deps fail the build), pre-built scopes for Android lifecycle owners (Application, Activity, ViewModel), official Google support.

@HiltAndroidApp

What: annotation on Application class. Replaces: manually creating Dagger components in onCreate. Why: triggers Hilt code generation; the entry point of the DI graph.

@AndroidEntryPoint

What: annotation enabling field injection in Activity / Fragment / Service / Receiver / View. Replaces: manual component.inject(this) calls. Why use: required for Hilt to inject into framework-constructed components.

@Inject constructor

What: annotation telling Hilt how to construct this class. Replaces: writing @Provides functions for every class. Why use: for any class you control — Hilt resolves the constructor params from the graph automatically.

@Module + @InstallIn

What: a class holding bindings via @Provides/@Binds; @InstallIn says which scope. Replaces: nothing — Dagger pattern. Why use: for types you can't @Inject (third-party, interfaces, configured instances).

@Provides

What: function that returns an instance — for any expression. Replaces: hand-rolled factories. Why use: when the binding requires construction logic — building a Retrofit, configuring an OkHttp.

@Binds

What: abstract function mapping interface → impl. Replaces: a @Provides function that just returns a constructor call. Why use: more efficient (Dagger doesn't generate a factory class) and clearer when there's no construction logic.

@Qualifier

What: custom annotation to disambiguate same-type bindings. Replaces: wrapper classes for type-safe primitives. Why use: when two bindings of the same type must coexist (@IoDispatcher vs @MainDispatcher for CoroutineDispatcher).

scope (@Singleton, @ViewModelScoped, etc.)

What: annotation declaring how long the binding lives within a component. Replaces: manual lifecycle bookkeeping. Why use: ensure the same instance is reused for the duration of the component (Application, Activity, ViewModel, etc.). Wider scope can be injected into narrower; not the reverse.

multibinding (@IntoSet / @IntoMap)

What: collect multiple impls into a Set or Map. Replaces: hand-rolled registry of plugins. Why use: plugin-style architecture — multiple analytics trackers, multiple deep-link handlers.

assisted injection

What: mix DI'd args + runtime args. Use @AssistedInject + @Assisted + @AssistedFactory. Replaces: manually writing factories that take both. Why use: when a class needs both injected deps and a value only known at runtime (a userId from a route).

Jetpack libraries

ViewModel

What: lifecycle-aware container of UI state that survives configuration changes. Replaces: Activity fields lost on rotation; onSaveInstanceState for non-trivial state. Why use: hold UI state and business logic separately from the view; tests don't need an emulator. Cleared in onCleared() when the owner is permanently destroyed.

SavedStateHandle

What: Bundle-backed map injected into a ViewModel. Replaces: onSaveInstanceState for VM-owned state. Why use: survives both configuration change AND process death — for state like search input or scroll position you'd be sad to lose.

LiveData

What: lifecycle-aware observable. Auto-pauses observers off-screen. Replaces: manual lifecycle observers + RxJava in early-2010s Android. Why use: in legacy XML codebases. New code should prefer StateFlow.

Room

What: SQLite ORM with compile-time SQL verification, Flow returns, type converters, migrations. Replaces: raw SQLiteOpenHelper + Cursor. Why use: SQL typos fail the build, not the app. Flow returns mean UI updates automatically when DB changes.

DAO

What: Data Access Object — interface declaring SQL queries that Room implements. Replaces: hand-written db.rawQuery + cursor parsing. Why use: type-safe API surface for the DB.

@Entity

What: annotates a class as a Room table. Replaces: CREATE TABLE SQL strings. Why: schema lives next to the model class.

@Relation / @Embedded

What: compose entities — @Relation for one-to-many joins; @Embedded for inline composition. Replaces: manual JOIN queries + result mapping. Why use: pull joined data as a typed object in one query.

migration

What: SQL transformation between Room schema versions. Replaces: manual onUpgrade code in SQLiteOpenHelper. Why use: upgrade users' on-device databases without losing their data. Always test with the schema export checked into Git.

DataStore

What: async, Flow-based replacement for SharedPreferences. Preferences (untyped KV) or Proto (typed). Replaces: SharedPreferences (synchronous, ANR risk on big writes). Why use: async by design, transactional, type-safe with proto schemas. New code should use DataStore.

WorkManager

What: deferrable, guaranteed background work. Survives reboot. Replaces: JobScheduler / AlarmManager / firebase-job-dispatcher. Why use: picks the right backend internally; constraints (network, charging, battery) handled for you. The default for sync, upload, indexing.

foreground service

What: Service with persistent notification — OS keeps it alive. Replaces: regular Service for user-visible long work (Android 8+ kills regular Services in background). Why use: music, location tracking, video processing. Must show a notification within ~5 seconds of starting.

expedited work

What: WorkManager request that runs ASAP, can promote to foreground service. Android 12+. Replaces: short-lived foreground services. Why use: "do this in seconds, not whenever the system feels like it" — like syncing a critical user action.

Paging 3

What: library for paginated data. PagingSource for network; RemoteMediator for offline-first paging. Replaces: hand-rolled pagination logic + boundary callbacks. Why use: handles loading windows, prefetch, error states, and deduplication for free.

Coil

What: modern Kotlin-first image loading. Memory + disk LRU caches. Replaces: Glide for new code. Why use: coroutine-native, Compose-friendly (AsyncImage), smaller dependency, Kotlin Multiplatform ready.

Glide

What: older image-loading library. Reflection-based. Replaces: nothing in modern code — used in legacy. Why exists: mature, well-tested, common in big codebases. New code should consider Coil.

Networking & Auth

REST

What: HTTP API style: resources via URLs, verbs (GET/POST/PUT/PATCH/DELETE), stateless. Replaces: SOAP / XML-RPC. Why: simple, cacheable via HTTP semantics, every backend supports it.

Retrofit

What: library that converts Kotlin interfaces into HTTP clients via runtime proxies. Replaces: hand-written HttpURLConnection code. Why use: type-safe API definitions, integrates with kotlinx.serialization and coroutines, swappable converters.

OkHttp

What: the HTTP client Retrofit sits on. Connection pooling, gzip, HTTP/2, redirects, caching. Replaces: Apache HttpClient, HttpURLConnection. Why: fast, modern, extensible via interceptors. The de facto standard.

Interceptor

What: OkHttp pipeline element that runs on every request. Replaces: filter chains. Why use: add headers (Authorization), logging, retry on 5xx, telemetry. Cross-cutting network concerns live here.

Authenticator

What: OkHttp pipeline element that runs on a 401 response. Replaces: manual "if 401, refresh token, retry" logic in every API call. Why use: refresh access token + retry the original request transparently. Must dedupe concurrent 401s with a Mutex.

JSON

What: JavaScript Object Notation — the default text format for REST APIs. Replaces: XML in most REST APIs. Why: compact, human-readable, every language has a parser.

Gson

What: Google's reflection-based JSON library. Replaces: hand-written JSONObject parsing. Why use: mostly legacy code. New Kotlin code should prefer kotlinx.serialization or Moshi (compile-time codegen, faster, Kotlin-aware).

Moshi

What: Square's modern JSON library with codegen. Replaces: Gson for new code. Why use: faster than Gson (no reflection), Kotlin-aware (handles default args, nullability).

kotlinx.serialization

What: Kotlin-first JSON (and other formats) library. KSP codegen, no reflection, KMP-ready. Replaces: Gson and Moshi for greenfield Kotlin code. Why use: handles default args, sealed classes, nullability natively; works on Kotlin Multiplatform; smaller APK (no reflection lib).

OAuth 2.0

What: standard for delegated authorization — letting an app access an API on a user's behalf. Replaces: shipping the user's password. Why use: the standard for "Login with Google/GitHub/etc." Mobile apps use the PKCE flavor.

PKCE

What: Proof Key for Code Exchange. Replaces a static client secret with a per-flow random verifier. Replaces: shipping a hardcoded client secret in your APK (which attackers can extract). Why use: the secure way to do OAuth 2.0 from a mobile app.

access token

What: short-lived credential proving auth. Sent as Authorization: Bearer .... Replaces: sending the user's password on every request. Why short-lived: if leaked, exposure is bounded — it expires in minutes.

refresh token

What: longer-lived credential used to get new access tokens. Replaces: forcing re-login when access expires. Why use: seamless UX — when access expires, the app silently refreshes. Stored encrypted; server should rotate it on each use.

EncryptedSharedPreferences

What: SharedPreferences variant backed by Android Keystore for encryption. Replaces: plain SharedPreferences for sensitive data. Why use: tokens, secrets — keys live in hardware-isolated Keystore on supported devices.

certificate pinning

What: hardcoding the expected server cert hash. Replaces: trusting whatever cert chain the OS validates. Why use: defends against rogue CAs / MITM attacks. Pin to multiple keys (current + backup) to avoid bricking the app on cert rotation.

idempotency key

What: a UUID sent with a POST request so the server can dedupe retries. Replaces: hoping the network doesn't drop between request-sent and response-received. Why use: safe POST retries; no duplicate orders / messages / payments.

cursor pagination

What: server returns nextCursor; client sends it back for the next page. Replaces: page-number pagination (which is unstable under concurrent inserts). Why use: stable — new items inserted between pages don't cause "skip" or "duplicate" rows.

Android core

Activity

What: a UI screen with its own lifecycle (onCreate / onStart / onResume / onPause / onStop / onDestroy). Replaces: nothing — fundamental Android primitive. Why: the entry point for user interaction with a screen.

Fragment

What: a UI piece embedded in an Activity. Has its own lifecycle plus a separate view lifecycle. Replaces: putting everything in one Activity. Why use: reusable UI chunks, multiple panes on tablets, modular navigation.

configuration change

What: rotation, font scale change, theme switch, multi-window. Replaces: nothing — OS-driven event. Why it matters: Activity is recreated by default; ViewModel survives, fields don't. onSaveInstanceState Bundle preserves UI state.

process death

What: OS kills your background process to reclaim memory. Replaces: nothing — OS reality. Why it matters: ViewModel does NOT survive. Only SavedStateHandle / onSaveInstanceState Bundle does. Test by enabling "Don't keep activities" in dev options.

launch mode

What: how an Activity is added to the task back stack — standard / singleTop / singleTask / singleInstance. Replaces: defaults. Why use: singleTop deduplicates an Activity already on top (notifications); singleTask makes there be only one instance task-wide (home / launcher screens).

onNewIntent

What: fired when an existing Activity instance is reused (singleTop / singleTask). Replaces: onCreate for the reuse case. Why use: handle the new Intent's extras when reusing an instance.

Service

What: background work without UI. Foreground (notification), bound (IPC), or restricted background. Replaces: raw threads + Notifications. Why use: long-running work that must keep going past Activity lifecycle. On Android 8+, prefer WorkManager unless you specifically need a foreground service.

BroadcastReceiver

What: receives system or app events. Manifest-declared (restricted on Android 8+) or dynamic. Replaces: polling for events. Why use: react to system events (boot, network change). For app-internal events, prefer SharedFlow / EventBus.

ContentProvider

What: Android component for cross-app data sharing via content:// URIs. Replaces: nothing for cross-app sharing. Why use: only when exposing data to other apps (rare in modern apps). Inside your app, use Room directly.

Intent

What: a messaging object describing an action. Explicit (you name the target) or implicit (system picks). Replaces: direct method calls between components (which would couple them). Why use: launch activities, start services, send broadcasts, with optional extras.

PendingIntent

What: an Intent another app can fire on your behalf. Used for notifications, alarms, widgets. Android 12+ requires FLAG_IMMUTABLE or FLAG_MUTABLE. Replaces: nothing — required for cross-app callbacks. Why immutable matters: prevents another app from modifying the wrapped Intent's extras or target.

Doze mode

What: battery optimization on idle stationary devices. Defers background CPU/network. Replaces: nothing — OS-driven. Why it matters: your background work can be delayed by minutes-to-hours. WorkManager handles this; foreground services + high-priority FCM bypass it.

App Standby

What: OS bucketing of apps by usage (active / working set / frequent / rare / restricted). Replaces: "all apps treated equally for background." Why it matters: rarely-used apps get tighter background restrictions — alarms, jobs, FCM all delayed.

ANR (Application Not Responding)

What: OS dialog when UI thread is blocked too long. UI input >5s, BroadcastReceiver >10s, Service >20s. Replaces: nothing — OS protection. Why it matters: ANRs annoy users and tank Play Store ratings. Avoid main-thread I/O at all costs.

StrictMode

What: debug enforcement of best practices — detects disk/network on main thread, leaks. Replaces: hoping bugs surface in QA. Why use: catch ANR-causing main-thread I/O during development, not in production. Enable only in debug builds.

Performance

cold / warm / hot start

What: Cold = process newly created. Warm = process exists, Activity recreated. Hot = both exist, brought to foreground. Replaces: nothing — terminology for talking about startup. Why it matters: different optimizations target different starts. Cold is the slowest and most worth optimizing.

Baseline Profile

What: AOT compile hints shipped in the APK. ART pre-compiles your hot startup paths. Replaces: hoping JIT catches up. Why use: 20-40% cold start improvement typical. Generated via Macrobenchmark recording the critical user journey.

Macrobenchmark

What: AndroidX library for stable, repeatable startup/scroll benchmarks. Replaces: ad-hoc System.nanoTime measurements. Why use: CI gates against perf regressions; the source of truth for "is this faster than yesterday."

R8

What: code shrinker + obfuscator + optimizer. Default in release. Replaces: ProGuard (which it absorbed). Why use: ~20-40% APK size reduction, faster class loading, harder reverse engineering.

ProGuard

What: predecessor to R8. The rules file (proguard-rules.pro) is still used. Replaces: nothing — R8 absorbed its role. Why mention: in interviews you'll say "R8 / ProGuard" interchangeably; the rules format and intent are the same.

resource shrinking

What: shrinkResources true strips unused images, layouts, strings. Replaces: manually deleting unused resources. Why use: works alongside R8 for additional APK size cuts.

App Bundle (AAB)

What: upload format for Play Store. Play splits per device (architecture, density, language). Replaces: universal APKs. Why use: download size cut by 30-50% typical — Play sends only the resources/code each device needs. Mandatory for new apps on Play since 2021.

jank

What: a frame missed its 16.6ms (60Hz) budget. User-perceptible stutter. Replaces: nothing — terminology. Why care: poor scrolling feels like a slow app, even when everything else is fast.

JankStats

What: AndroidX library for runtime jank detection in production-like builds. Replaces: hand-rolled Choreographer instrumentation. Why use: collect real-user jank metrics with the screen state attached, so you know which screens jank.

LeakCanary

What: debug-only library that automatically detects memory leaks. Replaces: hand-running heap dumps. Why use: just adding it as a debug dependency installs the detector — leaks pop up as notifications during development.

heap dump

What: snapshot of all live objects in memory. Profiler captures these. Replaces: guessing what's leaking. Why use: investigate leaks — group by class, find >1 instance of an Activity, follow the GC root chain.

TTID / TTFD

What: Time to Initial Display (first frame drawn) / Time to Full Display (content fully populated). Replaces: ambiguous "startup time." Why distinguish: a splash screen has fast TTID but slow TTFD until the data loads. Mark TTFD with reportFullyDrawn().

overdraw

What: a pixel drawn multiple times per frame. Wastes GPU. Replaces: nothing — phenomenon. Why care: deeply nested view hierarchies cause overdraw; visualize with Layout Inspector → "Show overdraw."

Player & streaming

ExoPlayer

What: open-source customizable Android media player. Replaces: the legacy MediaPlayer for serious streaming apps. Why use: consistent across OS versions, supports HLS / DASH / SmoothStreaming / DRM out of the box, fully customizable (track selection, buffering, ABR).

Media3

What: the modern umbrella unifying ExoPlayer + MediaSession + Cast + UI. Replaces: standalone ExoPlayer artifact for new code. Why use: single coherent API; future development happens in Media3.

HLS (HTTP Live Streaming)

What: Apple's adaptive streaming format. .m3u8 manifest, .ts or .mp4 segments. Replaces: progressive download / single-bitrate streams. Why use: native iOS support, works on Android via ExoPlayer.

DASH

What: MPEG's adaptive streaming format. .mpd manifest, .mp4 segments. Replaces: proprietary formats. Why use: open standard. No native iOS support — use HLS for iOS-friendly cross-platform.

CMAF

What: Common Media Application Format — .mp4 segments usable for both HLS and DASH. Replaces: packaging the same content twice (.ts for HLS, .mp4 for DASH). Why use: single packaging serves both formats — half the storage and CDN cost.

ABR (Adaptive Bitrate)

What: player measures bandwidth + buffer health, picks bitrate from manifest variants. Replaces: fixed-quality streams that buffer on slow networks. Why use: optimal quality on every connection — switches up if buffer healthy, down if draining.

Widevine

What: Google's DRM. Levels: L1 (hardware-backed), L2 (mixed), L3 (software, often no HD). Replaces: nothing — required by content owners for premium content. Why levels: Netflix / HBO etc. only allow HD on L1 devices because hardware-isolated keys are harder to extract.

DRM

What: Digital Rights Management. Encrypts content; license server issues decryption keys. Replaces: shipping unencrypted streams. Why use: required for premium content licensing. Common implementations: Widevine (Android), FairPlay (Apple), PlayReady (Microsoft).

CSAI

What: Client-Side Ad Insertion. Player pauses content during ad break and plays the ad separately (IMA SDK on Android). Replaces: "no ads." Why use: easy per-user targeting; downside is ad blockers can detect/skip.

SSAI

What: Server-Side Ad Insertion. Ads stitched into the master stream — single continuous playback. Replaces: CSAI when you want ad-block resistance and seamless playback. Why use: harder to block, no buffer at break. Per-user targeting requires per-user manifests.

quartile beacon

What: HTTP ping fired at ad lifecycle moments (start, 25%, 50%, 75%, complete) for billing/measurement. Replaces: nothing — ad industry standard. Why care: partners pay based on these. If beacons don't fire, you don't get paid.

MediaSession

What: exposes playback state to system UI (lock screen, BT, notification, Cast). Replaces: hand-built lock screen controls. Why use: single source of truth for playback; system controls just work.

TTFF (Time to First Frame)

What: the start-of-playback latency metric. Replaces: "is the player fast" hand-waving. Why care: users abandon playback if TTFF > 3 seconds; tracked in production via Conviva or Bitmovin Analytics.

rebuffer ratio

What: rebuffer time / total play time. Replaces: "buffering issues" anecdote. Why use: the player health KPI — <1% is excellent, >5% is a problem.

Build & tools

Gradle

What: build tool for Android. Build scripts in Kotlin DSL or Groovy. Replaces: Ant + Maven for Android. Why use: the only officially supported Android build system.

KSP (Kotlin Symbol Processing)

What: annotation processor framework for Kotlin. Replaces: kapt. Why use: 2x faster than kapt, doesn't generate Java stubs, official Kotlin support. Used by Room, Hilt, Moshi, kotlinx.serialization.

kapt

What: Kotlin annotation processor (older). Replaces: nothing — historical. Why mention: deprecated for new code; libraries have migrated to KSP. Old projects still use it.

build variant

What: combination of build type (debug/release) + flavor (free/paid). Each produces a distinct APK. Replaces: separate code branches per variant. Why use: different baseUrls, different Crashlytics keys, different feature sets — all from one codebase.

BuildConfig

What: generated class with build-time constants (BuildConfig.DEBUG, version, etc.). Replaces: manually defining DEBUG flags. Why use: conditional code paths between debug and release without polluting source.

AndroidManifest

What: XML declaring app components, permissions, intent filters. Replaces: nothing — fundamental. Why care: first thing the OS reads when installing your app; misdeclarations break runtime behavior.

Crashlytics

What: Firebase's crash reporting tool. Collects stack traces from production. Replaces: hand-rolled crash reporting. Why use: de facto standard; integrates with Play Console Vitals; supports Kotlin coroutine context preservation.

Misc & data structures

HashMap

What: hashtable map. O(1) average get/put. Keys must have stable hashCode/equals. Replaces: List<Pair> for key-based lookups. Why use: when you need value by key.

HashSet

What: hashtable set. O(1) average contains/add. Replaces: List.contains() for uniqueness checks. Why use: when you only need to know if X is present, no value to store. Backed by HashMap with dummy values.

LinkedHashMap

What: HashMap that preserves insertion (or access) order. Replaces: hand-tracking insertion order alongside a HashMap. Why use: ordered iteration. Foundation of LruCache.

LruCache

What: Least Recently Used cache with bounded size. Evicts the oldest when full. Replaces: unbounded HashMap that OOMs eventually. Why use: in-memory caching with a memory ceiling — bitmaps, parsed objects.

SparseArray

What: Map<Int, Object> without autoboxing. Replaces: HashMap<Int, T> on hot paths. Why use: avoids the boxing of Int → Integer for every key — saves memory and GC pressure on small int-keyed maps.

Bundle

What: map of basic types used to pass data between Activities and to/from process death restoration. Replaces: static fields (which don't survive process death). Why use: the only thing that survives Android killing your process; small data only (a few KB).

Parcelable

What: Android's serialization format for IPC and Bundle. Replaces: Java Serializable. Why use: 10x faster than Serializable, smaller wire size. Use @Parcelize annotation to auto-generate the boilerplate.

AAR vs JAR

What: AAR is Android Archive (includes resources, manifest, native libs); JAR is plain Java archive. Replaces: nothing — different output formats. Why use: AAR for any library that touches Android types; JAR for pure Kotlin/Java logic that should be consumable from non-Android JVM apps too.
