Glossary — definitions with context

Every term is described with the same shape: what it is → what it replaces / when to use → why. Skim this if a topic page uses a word that suddenly seems unfamiliar.

Reading guide

If you come from XML / LiveData / Java background, focus on the "replaces" and "why" parts — they map old concepts to new ones. The cards are designed for skimming; if you want depth, the topic pages are linked from the sidebar.

Kotlin & language

val / var

What: val creates a read-only reference — once assigned, it cannot point to a different object. var is reassignable. Replaces: Java's final vs non-final fields. Why: immutability is the default; reassignment is opt-in. Key gotcha: val does NOT mean the object is immutable — val list = mutableListOf<Int>() can still have items added to it. For deep immutability, use immutable collections.

const val

What: a compile-time constant inlined at every call site — no getter invocation, no class loading required. Only at top-level, inside object, or companion object; only primitives and String. Replaces: public static final in Java. Why: under the hood it becomes a JVM static final field. Right choice for string keys, tag names, route segments, and numeric thresholds. Don't use val for these — val still generates a getter.

nullable type (String?)

What: the ? suffix creates a distinct compile-time type. String and String? are different types; the compiler rejects calls on String? without null handling. Replaces: "every Java reference might be null with no compiler warning." Why: eliminates the entire class of NullPointerExceptions at compile time. Gotcha: platform types from Java interop (shown as String!) bypass this — the compiler gives no null guarantee, so always annotate Java APIs with @Nullable/@NonNull.

safe call (?.)

What: calls a method only if receiver is non-null; returns null otherwise. Replaces: if (x != null) x.foo() else null. Why: chains cleanly — user?.address?.city returns null at the first null in the chain without any nested ifs. Can appear on the left of an assignment: user?.name = "Alice" does nothing if user is null, which is often the right silent-skip behavior.

Elvis operator (?:)

What: a ?: b returns a if non-null, otherwise evaluates and returns b. Replaces: ternary x != null ? x : default. Why: the right-hand side can be any expression, including return or throw — making it perfect for guard clauses: val id = args["id"] ?: return. Chained with safe call: user?.email ?: "anonymous" is idiomatic Kotlin.

!!

What: forces the compiler to treat a nullable as non-null; throws KotlinNullPointerException if null at runtime. Replaces: nothing — it's a code smell. Why use: every !! in production is a latent crash. Prefer checkNotNull(value) { "descriptive message" } so failures carry diagnostic context. Acceptable in tests where you want loud failure, or in a few cases where you have external invariant knowledge the compiler can't verify.

smart cast

What: after a type check (is or null check), the compiler automatically narrows the type in that branch — no explicit cast needed. Replaces: Java's manual cast after instanceof. Why: works in if, when, and && chains. Gotcha: doesn't work on var properties that could be mutated from another thread — the compiler can't guarantee the type hasn't changed between check and use. Works fine on local vals.

data class

What: auto-generates equals (structural, on primary constructor params), hashCode, toString, copy (shallow), and componentN for destructuring. Replaces: Java POJO with hand-written equals/hashCode (or Lombok). Why: eliminates boilerplate for value types — DTOs, domain models, UI state. Key detail: copy() is shallow — nested mutable objects are shared between the original and the copy. Body properties (not in the primary constructor) are excluded from equals/ hashCode.

sealed class / sealed interface

What: a type whose direct subclasses are all known at compile time (same package, since Kotlin 1.5+). Replaces: "enum but each variant can carry different data" — Java has no clean equivalent. Why: enables exhaustive when — add a new subclass and every when without an else fails to compile. Foundation of MVI state, Result types, and navigation events. sealed interface allows a class to implement multiple sealed hierarchies simultaneously.

object (singleton)

What: a single-instance class, lazily initialized on first access using a thread-safe static holder. Replaces: Java's private-constructor + getInstance() pattern. Why: one line instead of ten. Use when you genuinely need one instance app-wide — but prefer DI for most cases. Gotcha: never store mutable Android context (Activity, Context) inside an object — it lives as long as the classloader and will leak the context.

companion object

What: a singleton tied to its enclosing class, holding members that belong to the class but don't require an instance. In Kotlin there are no static members — the companion is an actual object instance named Companion. Replaces: Java's static. Why: namespace for class-level constants and factory functions. Add @JvmStatic on methods and @JvmField on properties for proper Java interop — otherwise Java callers must write MyClass.Companion.create().

extension function

What: syntactically adds a function to a type from outside. Under the JVM, compiles to a static method with the receiver as the first parameter. Replaces: Java utility classes like StringUtils.isEmail(s). Why: calls read as s.isEmail() — as if the type owned the method. Key constraints: resolved at compile time on the declared type (not runtime type — no overriding), and can only access public/internal members, not private ones.

inline function

What: the compiler copies the entire function body (including lambda bodies) to each call site, eliminating object allocation for lambda parameters. Replaces: nothing equivalent in Java. Why: avoids GC pressure on hot paths like forEach; enables non-local return from within a lambda. Modifiers: noinline opts a specific lambda param out of inlining; crossinline allows inlining but forbids non-local returns. Don't mark everything inline — it increases bytecode size.

reified

What: preserves a generic type parameter at runtime, allowing T::class, is T checks, and T::class.java. Only usable inside inline functions. Replaces: passing Class<T> manually. Why: normally JVM type erasure removes generic info at runtime. Classic use: inline fun <reified T> fromJson(json: String): T = gson.fromJson(json, T::class.java) — call site becomes fromJson<User>(json) instead of fromJson(json, User::class.java).

scope functions (let/run/with/apply/also)

What: helpers that execute a block scoped to an object. Two axes: context object is this (run, with, apply) or it (let, also); return value is lambda result (let, run, with) or the receiver (apply, also). Replaces: intermediate variables and chained calls. Why: decision guide — apply: configure/build an object; also: side effects without changing the chain; let: null-safe block or transformation; run: execute a block with a receiver; with: operate on a non-null receiver. All five compile to the same bytecode — choose for readability.

lateinit var

What: a non-null reference whose value is set after construction — not in the constructor. Replaces: a nullable field that's "really always non-null but set in onCreate." Why: avoids !! at every access. Check safety: ::myVar.isInitialized. Accessing before initialization throws UninitializedPropertyAccessException with a clear message (not a silent NPE). Not available for primitives (Int, Boolean, etc.) — only reference types.

by lazy

What: a read-only property computed exactly once on first access, result cached indefinitely. Default thread-safety mode is double-checked locking (SYNCHRONIZED). Replaces: manual double-checked locking patterns. Why: defer expensive initialization until actually needed. For single-threaded contexts (e.g., inside a Composable), use lazy(LazyThreadSafetyMode.NONE) { } for a slight performance gain. Be careful what the lambda captures — it holds a reference to the enclosing scope until first access.

delegation (by)

What: two flavors — class delegation (class A(impl: B) : B by impl forwards all interface methods to impl) and property delegation (get/set route through a ReadOnlyProperty/ReadWriteProperty). Replaces: hand-written forwarding methods. Why: class delegation enables clean decorator/composition patterns — override only the methods you want to change, forward the rest. Built-in delegates: by lazy, by Delegates.observable { _, old, new -> }, by Delegates.vetoable, by savedStateHandle in ViewModel.

variance (out / in)

What: declaration-site variance annotates a type once rather than at every use site. out T (covariant): the class only produces T, never consumes it — List<Dog> IS-A List<Animal>. in T (contravariant): the class only consumes T — Comparator<Animal> can act as Comparator<Dog>. Replaces: Java's ? extends T / ? super T at every use site. Why: declare it once, benefit everywhere. Star projection List<*> means the type parameter is unknown — readable as Any?, not writable.

value class

What: a @JvmInline value class wraps a single property with zero runtime overhead in most cases — the JVM sees the underlying type. Replaces: using raw primitives or String for typed identifiers (e.g., both UserId and EmailAddress wrapping String). Why: compile-time type safety — swapping UserId and EmailAddress args is a compile error. Boxing occurs when the value class is used as nullable, cast to Any, or used as a generic type parameter.

typealias

What: a compile-time alias for an existing type — completely transparent to the compiler. Replaces: nothing — purely a readability aid. Why: excellent for function types (typealias Callback = (Result<User>) -> Unit) and complex generics. No type safety gained: a typealias UserId = String still accepts any String where UserId is expected. For real type safety, use value class instead.

Result<T>

What: stdlib wrapper for "succeeded with a value" or "failed with a Throwable." runCatching { } executes a block and catches any Throwable. Chain with mapCatching, onSuccess, onFailure, getOrElse. Replaces: try/catch when you want a typed result you can pass around. Key warning: runCatching catches CancellationException too — always rethrow it. Official Kotlin guidance: don't return Result from public APIs; use a sealed class for richer error modelling.

Nothing

What: the bottom type — a subtype of every other type. Any expression of type Nothing can appear where any type is expected, because it never returns. throw, error(), and TODO() all have return type Nothing. Replaces: nothing equivalent in Java. Why use: enables val x: String = nullable ?: error("x must be set") — the compiler infers x is non-null because the error() branch provably never returns. Also used as the return type of functions that always throw.

Coroutines

coroutine

What: a lightweight concurrent execution unit implemented as a compiled state machine. Each suspend point becomes a state; when suspended, the coroutine releases its thread for other work. ~1 KB memory per coroutine vs ~1 MB for a thread — thousands can run on a handful of threads. Replaces: AsyncTask (deprecated), Thread + Handler, RxJava chains. Why: sequential code style for async work, structured cancellation, and composable scope lifetimes — none of which raw threads provide.

suspend function

What: a function that can pause and resume without blocking a thread. Under the hood, the compiler adds a Continuation<T> parameter and transforms the body into a state machine. Replaces: callbacks, RxJava Single<T>, Future. Why: async code reads top-to-bottom like synchronous code; exceptions propagate via normal try/catch. Key: suspend is NOT about which thread runs the function — a suspend function can run entirely on the main thread.

coroutine builder (launch / async / runBlocking)

What: the bridge from regular code into the coroutine world. launch — fire-and-forget, returns Job, exceptions go to CoroutineExceptionHandler. async — produces a value, returns Deferred<T>, exceptions held until await(). runBlocking — blocks the calling thread; only for main(), tests, and bridge APIs, never on Android's main thread. Replaces: new Thread().start(), ExecutorService.submit().

CoroutineScope

What: a scope that ties coroutine lifetimes to a lifecycle — cancelling the scope cancels all children. Replaces: manual onDestroy { thread.interrupt() } bookkeeping. Why: structured concurrency — no orphan coroutines. Anti-pattern: GlobalScope — coroutines launched there run until completion regardless of any lifecycle and cannot be cancelled as a group. The lowercase coroutineScope { } suspend function creates a child scope that waits for all its children before returning — great for parallelizing work inside a single operation.

Job

What: a handle to a coroutine's lifecycle. States: New → Active → Completing → Completed (or Cancelling → Cancelled). cancel() requests cancellation; join() (suspend function) waits for completion. Replaces: Future in Java. Why use: Jobs form a tree — cancelling a parent cancels all children. A failed child cancels the parent and siblings (with a regular Job). Use job.invokeOnCompletion { } for cleanup without suspending.

Deferred<T>

What: a Job that produces a value — call await() to get it. Replaces: Future<T> / RxJava Single. Why use: for parallel work — val a = async { fetchA() }; val b = async { fetchB() }; a.await() + b.await() runs both in flight simultaneously. Critical gotcha: exceptions from a failed async are stored silently and only thrown when await() is called — easy to miss. Within supervisorScope, a failed sibling doesn't cancel others.

Dispatcher

What: determines which thread pool a coroutine runs on. Main — Android UI thread. Default — CPU-bound work, pool sized to CPU count. IO — blocking I/O, up to 64 threads. Unconfined — starts in caller's thread, resumes in whatever thread dispatched the continuation; avoid in production. Replaces: hand-managed ExecutorService. Why: withContext(Dispatchers.IO) { } is declarative and cheap (no new coroutine). Always inject dispatchers via constructor for testability — tests swap in UnconfinedTestDispatcher.

structured concurrency

What: three rules that eliminate "fire and forget" leaks: (1) every coroutine belongs to a scope, (2) the scope only completes when all children finish, (3) child failure propagates up and cancels siblings. A coroutine cannot outlive its scope. Replaces: the "launch a thread and hope it cleans up" model. Why: guarantees no orphaned background work, no coroutines running after their owner is destroyed, and deterministic cancellation.

SupervisorJob

What: changes the failure rule — one child's failure does NOT cancel siblings or the parent. Replaces: the default Job for independent-child scenarios. Why use: supervisorScope { } is the idiomatic block form. Used internally by viewModelScope. Use when children are independent — separate parallel API calls where one failing shouldn't abort the others. Without it, any child exception propagates and cancels all siblings, even if you catch it inside the child.

cooperative cancellation

What: cancellation is a request, not a kill — the coroutine checks for cancellation at each suspension point. All standard suspend functions already check. Replaces: Thread.interrupt(). Why: CPU-bound loops that never suspend are "uncooperative" — call ensureActive() or yield() periodically. Critical rule: never swallow CancellationException in a catch(e: Exception) block without rethrowing — it breaks the cancellation chain and causes the coroutine to keep running.

CoroutineExceptionHandler

What: installed as a context element on launch { }; the last-resort handler for uncaught exceptions after propagation. Replaces: Thread.UncaughtExceptionHandler. Why use: logging and telemetry in production. Three important limits: (1) only works with root launch coroutines, not async; (2) does NOT catch async exceptions (those surface at await()); (3) invoked after the exception has already cancelled the coroutine — you can't prevent the cancellation from here.

viewModelScope

What: a built-in CoroutineScope tied to the ViewModel, cancelled in onCleared(). Backed by SupervisorJob() + Dispatchers.Main.immediate. Replaces: manually creating and cancelling scopes. Why use: Dispatchers.Main.immediate means: if you're already on Main, updates are visible without waiting for the next frame — UI state updates appear instantly. Survives configuration changes because the ViewModel survives them. Does NOT survive process death.

lifecycleScope

What: a scope tied to a Lifecycle owner, cancelled at DESTROYED. Replaces: manual onDestroy cleanup. Why use: the canonical XML flow collection pattern: lifecycleScope.launch { repeatOnLifecycle(STARTED) { flow.collect { render(it) } } }. In Fragments, always use viewLifecycleOwner.lifecycleScope — the view's lifecycle is destroyed when the fragment's view is destroyed, while this.lifecycleScope persists until the fragment is destroyed, leaking the previous view.

Mutex

What: a coroutine-friendly mutual exclusion lock. mutex.withLock { } serializes access — while waiting, the coroutine suspends rather than blocking the thread. Replaces: synchronized blocks (which block the thread — dangerous on Dispatchers.Main). Why: safe for the main thread. Key gotcha: Mutexes are NOT reentrant — calling withLock from a coroutine already holding the lock will deadlock. For simple state updates, prefer MutableStateFlow.update { } which is atomic without a lock.

Channel

What: a coroutine-safe queue with send()/receive(). Types: RENDEZVOUS (no buffer, both sides sync), BUFFERED (fixed-size), CONFLATED (only latest), UNLIMITED (unbounded, OOM risk). Closing a channel with close() signals completion. Replaces: BlockingQueue. Why use: MVI event channels — Channel<Event>(BUFFERED) exposed as receiveAsFlow(). For most state and ambient event needs, prefer StateFlow/SharedFlow over raw Channel.

Flows

Flow<T>

What: a cold async stream of values — the producer block runs per collector, and no work happens until collect is called. Created with flow { emit(value) }, flowOf(...), list.asFlow(), or builder extensions. Replaces: RxJava Observable, callback APIs emitting multiple values. Why: sequential composition (map/filter/flatMap), backpressure, integrates with structured concurrency. Collecting the same cold Flow twice starts two independent executions — use stateIn/shareIn to share one upstream.

cold vs hot

What: Cold (plain Flow) — producer runs per collector; no collector → no work. Hot (StateFlow/SharedFlow) — run independently of collectors; multiple collectors share the same upstream. Replaces: RxJava's Observable vs Subject distinction. Why use cold: one-off computations like a single network call. Why use hot: ambient state or events shared across multiple UI observers. Converting cold → hot requires explicit stateIn/shareIn.

StateFlow

What: a hot, conflated flow always holding a current value, replaying exactly one value to new collectors, deduplicating by equals. stateFlow.value gives synchronous read access. MutableStateFlow.update { current -> ... } is atomic — safe for concurrent updates. Replaces: LiveData for all new code. Why use: KMP-ready, full Flow operator support (combine, flatMapLatest, debounce), Compose-friendly via collectAsStateWithLifecycle(). LiveData is Android-only and limited to map/switchMap.

SharedFlow

What: a hot flow with configurable replay (past values cached for new collectors) and extraBufferCapacity. No equality deduplication — every emission is delivered. Replaces: RxJava PublishSubject/ReplaySubject. Why use: set replay = 0 for one-time events (navigation, snackbar) — a new subscriber arriving after the event won't receive it. MutableSharedFlow.subscriptionCount lets you start/stop upstream work when collectors arrive/leave.

conflation

What: when the collector is slower than the producer, drops intermediate values and always delivers the latest. Replaces: manual debounce-like throttling. Why use: for state where only the latest value matters (scroll position, sensor readings). StateFlow is conflated by design. Distinction: conflate() drops emissions; collectLatest { } cancels the current collector invocation and restarts it with the newest emission — better when each emission triggers expensive work that should be abandoned for a fresher value.

backpressure

What: the strategy for when a producer emits faster than the consumer processes. buffer(n) — runs producer and consumer concurrently with a bounded buffer. conflate() — keeps only the latest. collectLatest { } — cancels and restarts the collector on each new emission. Replaces: RxJava's onBackpressureBuffer/Drop/Latest. Decision guide: buffer for bursty data where all values matter; conflate for state where only latest matters; collectLatest for expensive per-emission work.

flowOn(dispatcher)

What: shifts the context of everything upstream of the call. Everything below (including collect) continues in the caller's context. Can be chained — each section runs in its own context. Replaces: RxJava's subscribeOn. Why use: flow { }.map { parseHeavy(it) }.flowOn(Default).collect { updateUi(it) } keeps parsing off Main. Unlike RxJava there's no "downstream" equivalent — collect always inherits the caller's context.

stateIn / shareIn

What: converts a cold Flow into a hot StateFlow / SharedFlow, sharing one upstream subscription among all collectors. Replaces: RxJava's publish().refCount(). Why use: without this, each collect call starts a new upstream execution (separate network call per observer). The started param controls activation: Eagerly, Lazily (first collector), or WhileSubscribed(timeout). Canonical ViewModel pattern: stateIn(viewModelScope, WhileSubscribed(5000), UiState.Loading).

SharingStarted.WhileSubscribed(5000)

What: starts collecting the upstream when the first collector arrives; stops 5000 ms after the last collector leaves. Replaces: manually managing subscription refcounts. Why: the 5-second grace period covers configuration changes (rotation takes ~100–300 ms), so the new collector reuses the active upstream without refetching. Setting 0 would cancel and restart the upstream on every rotation. When the user backgrounds the app, the scope's collector eventually stops → upstream cancels after 5 s, saving battery and network.

flatMapLatest

What: for each upstream emission, cancels the previous inner Flow and starts a new one. Replaces: RxJava's switchMap. Why use: the canonical search-as-you-type pattern — each new query cancels the in-flight search for the previous one. Equivalent to transformLatest { value -> emitAll(innerFlow(value)) }. Don't use flatMapMerge for search — it accumulates results from every query concurrently, delivering stale results.

combine(a, b) { x, y -> ... }

What: waits for all source flows to emit at least once, then emits whenever ANY source emits, pairing the latest from each. Replaces: RxJava's combineLatest. Why use: derive screen state from multiple sources: combine(searchQuery, filters, tab) { q, f, t -> ScreenState(q, f, t) }. Distinct from zip which pairs by emission index — combine uses the latest, not corresponding emissions. If one source never emits, combine never emits.

callbackFlow

What: wraps a callback-based API into a Flow. Inside the block, use trySend(value) (non-suspending, safe from callbacks) or send(value) (suspending). awaitClose { unregister() } is mandatory — it suspends until the collector cancels and runs cleanup. Replaces: RxJava's Observable.create. Why use: bridge legacy listener APIs (LocationManager, ContentResolver, BroadcastReceiver) into the Flow world. If the callback can fire after awaitClose, check isClosedForSend before sending.

collectAsStateWithLifecycle()

What: Compose extension (requires lifecycle-runtime-compose) that collects a Flow only while the lifecycle is at least STARTED. Returns the current value as Compose State<T>. Replaces: bare collectAsState() which keeps collecting even when backgrounded. Why use: when the app backgrounds, collection pauses — no wasted CPU/battery. Always use this over collectAsState() in production Compose screens.

repeatOnLifecycle(STARTED)

What: the XML/Fragment equivalent of collectAsStateWithLifecycle. Cancels the inner coroutine block when the lifecycle drops below STARTED, restarts it when it rises back. Each background → foreground transition starts a fresh collection. Replaces: lifecycleScope.launch { flow.collect { } } which keeps collecting off-screen. Why use: canonical pattern — lifecycleScope.launch { repeatOnLifecycle(STARTED) { vm.state.collect { render(it) } } }. launchWhenStarted is deprecated in favor of this.

Compose

@Composable

What: a function that emits UI nodes when called inside the composition. Composables are pure functions of state — same inputs always produce the same UI description. They can only be called from other composables or within composition; the Compose compiler plugin validates call sites and instruments each function for recomposition tracking. Replaces: XML layouts + custom View classes. Why: declarative, testable, previewable, no findViewById, no adapters, no ViewHolders. Never trigger side effects directly in the function body — use effect handlers.

composition

What: the process of calling all @Composable functions to build the UI node tree. Compose works in three phases: Composition (what) → Layout (where) → Drawing (how). Initial composition runs everything; subsequent recompositions only re-run composables whose inputs changed. Replaces: the XML View hierarchy. Why: the runtime diffs and updates underlying nodes — you describe the desired state, not how to mutate the existing tree. This makes UI bugs where view state diverges from data state nearly impossible.

recomposition

What: re-running composable functions when their state inputs change. Compose is smart — only composables that read changed state are re-run, not the whole tree. Replaces: manual view.setText(...), adapter.notifyDataSetChanged(). Why: UI follows state automatically. Main performance trap: an unstable type (like a List<T> field) causes the composable to always recompose. Diagnose with Layout Inspector → Recomposition tab — a count climbing while idle is a bug.

slot table

What: Compose's internal data structure tracking the position and memoized state of every composable in the tree. remember values are keyed by call-site position in the slot table — not by variable name. Replaces: nothing direct — internal mechanism. Why it matters: in lists, use key(item.id) { ItemComposable(item) } to give Compose a stable identity for each item. Without key, items are identified by position — reordering the list associates the wrong remembered state with the wrong item (e.g., wrong text in an input field).

stable (@Stable / @Immutable)

What: a type Compose can trust: its equals correctly determines whether re-rendering is needed. Primitives and String are stable. Data classes with all stable fields are inferred stable — but a List<T> field breaks this because List is a mutable interface. Why use: stable parameters enable recomposition skipping — unchanged composables are not re-run. Fix: replace List<T> with ImmutableList<T> (kotlinx-collections-immutable) or annotate with @Immutable. Breaking the @Immutable contract causes silent stale UI.

state hoisting

What: move state to the lowest common ancestor of all composables that need it; pass value down and onValueChange lambda back up. Replaces: stateful custom Views. Why: stateless leaves are easier to preview, test, and reuse. In practice, ViewModel is the "lowest common ancestor" for screen-level state — it owns the state and composables just render it. Keep truly local UI state (dropdown open/closed, animation progress) inside the composable with remember.

remember

What: stores a value across recompositions for the lifetime of the composable's presence in composition, keyed by call-site position. remember(key) { compute(key) } recomputes when the key changes. Replaces: Activity field for transient UI state. Why use: survive recomposition without putting it in ViewModel. Lost when the composable leaves composition (navigates away). Use rememberSaveable for state that must survive configuration change or process death.

rememberSaveable

What: like remember but backed by SavedStateRegistry (the Activity Bundle). Survives both configuration changes and process death. Replaces: manual onSaveInstanceState/onRestoreInstanceState. Why use: text input, scroll position, selected tab — small UI state you'd be sad to lose. For custom types, provide a Saver. Keep values small — Bundles have a ~500 KB limit. Heavy state (fetched lists) belongs in SavedStateHandle in ViewModel.

LaunchedEffect(key)

What: launches a coroutine scoped to the composable. Runs on first composition; restarts (cancels previous coroutine, starts new) when any key changes. LaunchedEffect(Unit) runs exactly once. Replaces: onResume-tied API calls. Why use: trigger a one-time action when a value becomes non-null, load data when entering a screen, start an animation. Don't use for collecting flows — use collectAsStateWithLifecycle instead, which handles lifecycle automatically.

DisposableEffect

What: runs setup code and registers an onDispose { } cleanup block. Cleanup runs when the composable leaves composition or the key changes. The Compose compiler enforces that onDispose is always provided. Replaces: register/unregister patterns in Activity onCreate/onDestroy. Why use: ExoPlayer prepare/release, BroadcastReceiver register/unregister, LocationManager request/remove. DisposableEffect(key) re-runs both setup and cleanup when the key changes.

derivedStateOf

What: creates a derived state that recomputes only when its source states change, and only triggers recomposition in dependents when the derived result changes. Wrap in remember { derivedStateOf { } } — without remember, a new object is created each recomposition, defeating the purpose. Replaces: nothing equivalent in XML. Why use: val showFab by remember { derivedStateOf { listState.firstVisibleItemIndex > 0 } } — recomposition fires only when the boolean flips, not on every pixel of scroll.

rememberUpdatedState

What: stores the latest value of a parameter so that a long-running effect can always read the current value without restarting. Replaces: nothing equivalent. Why use: solves the stale-closure bug. If LaunchedEffect(Unit) captures a callback param, it captures the value at effect-start time — if the parent recomposes with a new callback, the effect still calls the old one. Fix: val currentCallback by rememberUpdatedState(callback); the effect stays running but always reads the latest callback.

snapshotFlow

What: converts Compose State reads into a Flow. The block is re-executed whenever any observed Compose state changes, and the flow emits the new result. Replaces: manually wiring callbacks + Flow producers. Why use: bridge Compose state (scroll position, text input) to non-Compose coroutine pipelines or analytics: snapshotFlow { listState.firstVisibleItemIndex }.debounce(300).collect { logScroll(it) }.

CompositionLocal

What: passes values implicitly down the composable tree without threading them through every function parameter. Accessed with .current: LocalContext.current, MaterialTheme.colorScheme. Two variants: compositionLocalOf (throws if no provider; changes re-render only direct consumers) vs staticCompositionLocalOf (recomposes the entire subtree on change, but faster when unchanged). Replaces: prop-drilling. Why use: theme, density, snackbar host — cross-cutting concerns. Don't use for regular data — explicit parameters are always clearer.

Modifier

What: decorates a composable with layout, drawing, and interaction behaviors. Order is significant — applied outside-in: Modifier.padding(8.dp).background(Red) gives red background with colorless padding outside it; reversed gives red extending into the padding area. Replaces: XML attributes. Why: composable, type-safe, plain Kotlin. Always accept modifier: Modifier = Modifier as a parameter in your own composables so callers control positioning and sizing — never hardcode layout decisions inside a composable.

Architecture

MVVM

What: ViewModel holds UI state and business logic; View observes and renders; user events flow back via method calls. ViewModel must never hold a reference to Activity, Fragment, or View — it outlives them. Replaces: "fat Activity" MVC. Why use: testable (inject fake repos, no emulator needed), lifecycle-safe state, separation of concerns. Pitfall: don't put one-time UI actions (navigation, dialogs) in StateFlow state — use a separate Channel(BUFFERED).receiveAsFlow() with replay = 0.

MVI

What: a stricter pattern — UI emits sealed Intent objects (all possible user actions), ViewModel processes each → produces a new immutable UiState, one-time side effects (navigation, snackbar) go on a separate Channel/SharedFlow(replay=0). Replaces: MVVM's scattered "multiple flows" approach for complex screens. Why use: every state is reproducible (replay the Intent log); exhaustive when on the sealed Intent catches unhandled cases at compile time. Libraries: Orbit, MVI Kotlin. Overkill for simple CRUD screens.

Clean Architecture

What: three rings with a strict dependency rule — inner rings know nothing about outer. Data (Room, Retrofit) → Domain (pure Kotlin — entities, use cases, repository interfaces) → Presentation (ViewModel, UI). Replaces: monolith with no layer separation. Why use: the Domain layer has zero Android imports — tested with pure JVM unit tests, no emulator. Repository interfaces in Domain; implementations in Data. Never import a Data class from Presentation. Multi-module projects often map each ring to a Gradle module.

UDF (Unidirectional Data Flow)

What: state flows down (ViewModel → View), events flow up (View → ViewModel). Never bidirectional between the same two layers. Replaces: two-way data binding tangles where A updates B which triggers A again. Why use: every state change has a single traceable source. Debug a bug by finding the Intent/event that caused the state. Compose's @Composable(state, onEvent) pattern is UDF at the function level.

UseCase

What: a class encapsulating exactly one piece of business logic, callable via operator fun invoke. Naming: VerbNounUseCase — GetUserFeedUseCase, SubmitPaymentUseCase. Replaces: business logic scattered inside ViewModel. Why use: reusable across multiple ViewModels; composable (one UseCase can call another); trivially unit-testable. Skip it when the class would be a direct passthrough to one repository method — that's noise. Add one when combining repos, adding validation, or reusing across VMs.

Repository

What: single source of truth for a data type. Coordinates network + DB + cache; returns domain models (never DTOs or Room entities). The ViewModel asks for data; the Repository decides whether to serve from cache, DB, or network. Replaces: ad-hoc API calls scattered in ViewModels. Why use: enables offline-first — Room is the source of truth; background sync updates Room; Room Flow updates the UI. Repository interfaces live in the Domain layer; implementations in Data.

SDUI (Server-Driven UI)

What: server returns a JSON tree of component descriptors; client renders each via a component registry (a map from type string to composable). Replaces: hard-coded layouts requiring an app release to change. Why use: update UI, run A/B tests, fix layouts without a new release. Challenges: client must handle unknown component types gracefully (skip or fallback, never crash); versioning and schema migration are complex; state for server-driven forms requires careful design. Used widely at Netflix, Airbnb.

multi-module

What: splitting the app into Gradle modules (typically: :app, :feature:X, :core:ui, :core:data, :core:domain). Replaces: single-module monolith. Why use: the build system compiles independent modules in parallel; changing :feature:feed doesn't force :feature:profile to recompile. Enforces architecture boundaries — build fails if the UI module imports the data layer directly. Module graph must be a DAG (no circular deps). NowInAndroid is the reference implementation.

api vs implementation

What: Gradle dependency scopes. api(dep) exposes dep's types to any module that depends on yours (transitive). implementation(dep) hides dep from consumers. Replaces: the old compile scope which always exposed. Why: default to implementation — leaking api deps couples consumers to your implementation choices and forces unnecessary recompiles across the module graph. Only use api when your module's public method signatures expose types from that dependency.

dependency inversion

What: inner rings define interfaces; outer rings implement them. The dependency arrow points inward: Data knows about Domain (implements its interfaces), Domain knows nothing about Data. Replaces: "domain depends on data" — the backwards, untestable design. Why use: Domain layer has no Android imports and can be tested with pure JVM unit tests. In DI terms: Hilt binds interface UserRepository → UserRepositoryImpl in a module; every ViewModel only ever sees the interface.

Hilt / DI

dependency injection (DI)

What: a class receives its collaborators from outside (constructor, field, or method) rather than constructing them internally. The DI framework owns the object graph. Replaces: val foo = FooImpl() hardcoded inside classes. Why: testability (swap real for fake), decoupling (depend on interfaces), single place to configure "how to build X." Compile-time DI (Dagger/Hilt) verifies the graph at build time — missing or mismatched bindings fail the build, not the app.

constructor injection

What: dependencies are passed as constructor parameters and stored as val. Replaces: field injection or service-locator lookups. Why use: the constructor signature is the complete contract — reading it tells you everything the class needs. Deps are immutable after construction. Easy to use in tests without any DI library — just pass fakes directly. The preferred form for any class you control.

field injection

What: @Inject lateinit var dep: Dep — Hilt sets the field after construction. Replaces: nothing — used only when you can't control the constructor. Why use: Android framework classes the OS instantiates (Activity, Fragment, Service, BroadcastReceiver, View). Fields must be var, not val, and not private. Everywhere else, prefer constructor injection — the dependencies are more explicit and the class is easier to test.

Hilt

What: Google's opinionated DI library for Android, built on Dagger 2. Generates Dagger components for standard Android lifecycle owners automatically. Replaces: hand-written Dagger components, service locators (Koin). Why use: the dependency graph is verified at compile time — missing bindings or scope violations are build errors. Predefined scopes: @Singleton, @ActivityRetainedScoped, @ViewModelScoped, @ActivityScoped, @FragmentScoped. Official Google recommendation.

@HiltAndroidApp

What: placed on the Application class. Triggers all of Hilt's code generation and creates the top-level SingletonComponent. Replaces: manually creating and wiring Dagger components in Application.onCreate(). Why: the root of the entire DI graph — without it, nothing is injected. Every injected Activity/Fragment must be in an app that has this annotation.

@AndroidEntryPoint

What: enables Hilt field injection in Activity, Fragment, Service, BroadcastReceiver, and View. Hilt generates a superclass wrapping your component; the framework instantiates the subclass transparently. Replaces: manual component.inject(this) calls. Why use: required for Hilt to inject into framework-constructed components. For Fragments, the host Activity must also be @AndroidEntryPoint. ViewModels use @HiltViewModel instead.

@Inject constructor

What: tells Hilt how to construct this class — Hilt inspects the constructor params and resolves each from the graph automatically. Replaces: writing a @Provides function for every class. Why use: for any class you own — repositories, use cases, data sources. Combine with a scope annotation (@Singleton) and Hilt ensures only one instance is ever created. No boilerplate needed beyond the annotation.

@Module + @InstallIn

What: @Module is a class with @Provides/@Binds functions. @InstallIn(SingletonComponent::class) installs the bindings into that component's scope. Replaces: manually wiring Dagger modules. Why use: for types you can't @Inject: third-party classes (Retrofit, OkHttp), interface bindings, or instances that need configuration. The component in @InstallIn constrains the max scope of bindings inside it.

@Provides

What: a function inside a @Module that constructs and returns an instance. The function's parameters are themselves resolved from the DI graph. Replaces: hand-rolled factories. Why use: when construction requires logic — building a Retrofit with a specific base URL and OkHttp client, or creating an instance with configuration that isn't deterministic from the type alone.

@Binds

What: an abstract function in an abstract @Module class that maps an interface to its implementation. Replaces: a @Provides function that just calls the constructor. Why use: more efficient — Dagger doesn't generate a factory class, just a type alias in the component. Cleaner when there's no construction logic: @Binds abstract fun bindRepo(impl: UserRepoImpl): UserRepository. Must be in an abstract class, not an object.

@Qualifier

What: a custom annotation that disambiguates multiple bindings of the same type in the graph. Replaces: wrapper classes for type-safe primitives. Why use: classic case — two CoroutineDispatcher bindings: @Qualifier @Retention(BINARY) annotation class IoDispatcher. Then @Provides @IoDispatcher fun provideIo() = Dispatchers.IO. The consumer: @Inject constructor(@IoDispatcher val dispatcher: CoroutineDispatcher).

scope (@Singleton, @ViewModelScoped, etc.)

What: an annotation declaring how long a binding lives within a component. Unscoped = new instance per injection. @Singleton = one instance per app process. @ViewModelScoped = one per ViewModel instance. Replaces: manual lifecycle bookkeeping. Why use: ensure the same Repository/service instance is reused within its lifetime. Rule: a wider-scoped binding can be injected into a narrower scope; a narrower scope cannot be injected into a wider one (compile error).

multibinding (@IntoSet / @IntoMap)

What: collects multiple contributions into a single Set<T> or Map<K, V>. Each contributing binding is annotated with @IntoSet or @IntoMap(key). The consumer declares Set<AnalyticsTracker> and receives all registered trackers. Replaces: hand-rolled plugin registries. Why use: plugin architecture — multiple analytics providers, deep-link handlers, interceptors. Adding a new plugin requires only a new @Binds @IntoSet binding.

assisted injection

What: for classes that need both graph-provided deps and runtime values. Annotate with @AssistedInject, mark runtime params with @Assisted, define an @AssistedFactory interface. Hilt generates the factory. Replaces: manually writing factories that accept both injected and runtime params. Why use: when a class needs an injected Repository plus a runtime itemId from nav args. Common with ViewModel: Hilt's HiltViewModelFactory supports assisted-injected ViewModels.

Jetpack libraries

ViewModel

What: a lifecycle-aware holder for UI state and business logic, stored in the ViewModelStore which is NOT destroyed on configuration change. Cleared in onCleared() when the user truly leaves the screen. Replaces: Activity fields lost on rotation; onSaveInstanceState for non-trivial state. Why use: test without an emulator (inject fake repos); state survives rotation. Critical: never hold a reference to Activity, Fragment, or View in a ViewModel — they have shorter lifetimes. Use AndroidViewModel or Hilt injection for Application context if needed.

SavedStateHandle

What: a Bundle-backed Map<String, Any?> injected into the ViewModel. Survives both configuration changes AND process death. getStateFlow("key", default) gives a Flow-based accessor. Replaces: onSaveInstanceState for VM-owned state. Why use: nav arguments (Hilt's @HiltViewModel auto-injects it), search queries, scroll positions — anything lost on process death would confuse the user. Values must be Bundle-compatible; large objects go in Room/DataStore.

LiveData

What: a lifecycle-aware observable holder. Automatically stops delivering updates to Observers when the lifecycle owner is STOPPED and resumes at STARTED. Replaces: manual lifecycle observers + RxJava in early Android. Why use: only in legacy XML codebases. New code uses StateFlow. LiveData limitations: Android-only (no KMP), limited operators (only map/switchMap), must use postValue from background threads, no backpressure.

Room

What: a compile-time-verified SQLite abstraction. Annotations on data classes (@Entity), interfaces (@Dao), and a @Database class generate all boilerplate. SQL errors are caught at build time. Replaces: raw SQLiteOpenHelper + Cursor. Why use: @Query returning Flow<T> re-emits automatically when the underlying table changes — UI stays in sync. Always export schema JSON to version control so you can audit migrations. Integrates with Paging 3 via PagingSource.

DAO

What: Data Access Object — an interface with @Insert, @Update, @Delete, and @Query methods that Room implements at compile time. Replaces: hand-written db.rawQuery() + Cursor parsing. Why use: type-safe API surface for the database. Return types: suspend fun for one-shot operations; Flow<T> for observable queries; PagingSource<K, V> for pagination. The DAO lives in the data layer and is injected into the Repository via Hilt.

@Entity

What: maps a data class to a Room table. Requires a @PrimaryKey (can be autoGenerate = true). Column names default to the property name; override with @ColumnInfo(name = "..."). Replaces: CREATE TABLE SQL strings. Why: schema lives next to the model; changes are visible in the diff. Add indices with @Entity(indices = [Index("email")]). Use @Ignore on fields you don't want persisted.

@Relation / @Embedded

What: @Embedded flattens a nested data class into the parent table's columns (no JOIN needed). @Relation(parentColumn = "userId", entityColumn = "ownerId") declares a one-to-many or one-to-one join Room executes automatically. Replaces: manual JOIN queries + result mapping. Why use: load a typed POJO with related data in one query. Always annotate the DAO method with @Transaction when using @Relation to ensure the parent and children are fetched atomically.

migration

What: a Migration(oldVersion, newVersion) object with a migrate(db) function executing raw SQL to transform the schema. Room checks schema hashes — if the schema changed without a matching migration, the app crashes on upgrade. Replaces: manual onUpgrade in SQLiteOpenHelper. Why use: upgrade users' on-device data without losing it. Provide fallbackToDestructiveMigration() in dev (wipes and recreates); real migrations in production. Test with MigrationTestHelper.

DataStore

What: async, coroutine-native, transactional key-value or proto storage. All writes are atomic and durable. Reading returns a Flow<Preferences> — no synchronous read. Replaces: SharedPreferences (synchronous, ANR risk on large writes, not transactional). Why use: async by design; proto variant is type-safe. Never access on the main thread. Migration from SharedPreferences: use SharedPreferencesMigration to move existing data automatically on first run.

WorkManager

What: guaranteed, deferrable background work. Survives app restarts and device reboots. Takes Constraints (network, charging, battery), supports chaining and periodic work. Replaces: JobScheduler, AlarmManager, firebase-job-dispatcher. Why use: picks the best system scheduler internally; constraints handled for you. For urgent user-visible work, use setExpedited(OutOfQuotaPolicy) to request fast execution (can promote to a short-lived foreground service). The default for sync, upload, and indexing jobs.

foreground service

What: a Service with a persistent user-visible notification — the OS gives it significantly higher priority and is unlikely to kill it. Replaces: regular background Services (Android 8+ kills those within minutes). Why use: audio playback, location tracking, video upload. Must call startForeground(id, notification) within ~5 seconds of starting. Since Android 9+ must declare foregroundServiceType in the manifest. Since Android 12+ cannot start from the background — use WorkManager expedited instead.

expedited work

What: a WorkRequest with setExpedited(OutOfQuotaPolicy) that runs ASAP. On Android 12+, WorkManager can promote it to a short-lived foreground service if system quota allows. Replaces: short-lived foreground services for urgent background work. Why use: syncing a critical user action within seconds, not when the OS feels like it. OutOfQuotaPolicy.RUN_AS_NON_EXPEDITED_WORK_REQUEST falls back gracefully when quota is exhausted.

Paging 3

What: loads pages of data on demand. PagingSource for network-only; RemoteMediator for offline-first (network fetches → Room → Paging reads from Room). PagingData<T> is collected into LazyPagingItems in Compose or PagingDataAdapter in RecyclerView. Replaces: hand-rolled pagination. Why use: loading windows, prefetch, error states, retry, and deduplication handled for free. LoadState exposes loading/error/idle state for top, bottom, and refresh indicators.

Coil

What: Kotlin-first image loading library. Coroutine-native, memory + disk LRU caching, OkHttp for networking. AsyncImage(model = url, contentDescription = "...") in Compose. Supports GIFs, SVG (via extension), video thumbnails. Replaces: Glide for new code. Why use: Compose-native API, smaller dependency footprint, KMP support in Coil 3, no reflection. The default for new projects.

Glide

What: mature, battle-tested image loading library with a generated API (@GlideExtension annotation processing) and excellent bitmap pool management for RecyclerView scroll performance. Replaces: nothing in modern projects — found in existing large codebases. Why exists: well-tested, extensive platform coverage, custom ModelLoader/ResourceDecoder for unusual data sources. Glide 5 adds Kotlin and Compose support. Consider Coil for new projects.

Networking & Auth

REST

What: HTTP API style — resources via URLs, standard verbs (GET read/cache, POST create, PUT full replace, PATCH partial update, DELETE remove), stateless. Status codes carry semantics: 200 OK, 201 Created, 204 No Content, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 429 Rate Limited, 500 Server Error. Replaces: SOAP/XML-RPC. Why: simple, cacheable via HTTP semantics, every backend and toolchain supports it. Correct verb and status code usage enables automatic retry and cache logic.

Retrofit

What: turns a Kotlin interface into a fully functional HTTP client via JVM dynamic proxy (or KSP codegen). Annotations on methods (@GET, @POST, @Path, @Query, @Body) map to HTTP calls. Return types: suspend fun for coroutines; Flow<T> via adapter; Call<T> for manual callback. Replaces: hand-written HttpURLConnection. Why use: type-safe API definitions, pluggable converter factories (kotlinx.serialization, Moshi, Gson), pluggable call adapters.

OkHttp

What: the HTTP engine underlying Retrofit. Handles connection pooling, transparent gzip, HTTP/2 multiplexing, response caching (respects Cache-Control), redirects, and timeouts. Replaces: Apache HttpClient, HttpURLConnection. Why: extensible via interceptors; the de facto standard. Application interceptors run first/last; network interceptors run closer to the wire and see redirects. All network calls block the calling thread — use on Dispatchers.IO.

Interceptor

What: an OkHttp chain element wrapping every request/response. Application interceptors (addInterceptor): always run once, ideal for adding auth headers, logging, and retrying on business errors. Network interceptors (addNetworkInterceptor): run closer to the wire, can see cache responses, may run multiple times on redirects. Replaces: ad-hoc header manipulation in each API call. Why use: authorization headers, logging (HttpLoggingInterceptor), telemetry, retry-on-5xx — cross-cutting network concerns in one place.

Authenticator

What: an OkHttp callback invoked specifically on 401 responses. Returns a new Request with refreshed credentials, or null to propagate the 401. Runs after the Interceptor chain. Replaces: "if 401, refresh, retry" logic duplicated in every call. Why use: refresh the access token and replay the original request transparently. Critical: if three requests all 401 simultaneously, all three will try to refresh — wrap the refresh in a Mutex so only one refresh happens and others wait for the result.

JSON

What: JavaScript Object Notation — the standard text format for REST APIs. Supports objects {}, arrays [], strings, numbers, booleans, null. Replaces: XML in most REST APIs. Why: compact, human-readable, every language has a parser. Performance note: JSON parsing is CPU-intensive; for high-frequency data, consider binary formats (Protocol Buffers, CBOR). Large JSON parsing blocks the thread — always parse in a background coroutine.

Gson

What: Google's reflection-based JSON library. Uses Reflection.newInstance to create objects and reflectively sets fields. Replaces: hand-written JSONObject parsing. Why avoid in new code: reflection doesn't understand Kotlin's type system — a non-null String field with a null JSON value is silently set to null, bypassing the null-safety system. Default parameter values in data classes are also ignored. Prefer Moshi or kotlinx.serialization for Kotlin.

Moshi

What: Square's modern JSON library with KSP/kapt codegen (@JsonClass(generateAdapter = true)). Unlike Gson, respects Kotlin nullability — a non-null field with a null JSON value throws. Replaces: Gson for new Android-only Kotlin code. Why use: faster than Gson (no reflection), Kotlin-aware (handles nullable types, default args). Compatible with Retrofit via MoshiConverterFactory. For KMP projects, use kotlinx.serialization instead.

kotlinx.serialization

What: JetBrains' official Kotlin-first serialization library. Supports JSON, CBOR, ProtoBuf, Properties. KSP-based codegen — no reflection at runtime. @Serializable on the data class; @SerialName("json_key") for field name mapping. Replaces: Gson and Moshi for all new Kotlin code. Why use: handles nullable types, default parameter values, sealed classes natively; KMP-ready (same code on Android, iOS K/N, JS); smaller APK. The default for new projects.

OAuth 2.0

What: delegated authorization — the app requests an auth code; the server redirects back with the code; the app exchanges it for access + refresh tokens. The app never sees the user's password. Replaces: shipping the user's password. Why use: the standard for "Login with Google/GitHub/etc." Mobile apps use the Authorization Code + PKCE flow. Token endpoint returns access_token, refresh_token, expires_in. Use the AppAuth library on Android for a correct implementation.

PKCE

What: Proof Key for Code Exchange — an OAuth 2.0 extension for public clients. The client generates a random code_verifier, computes code_challenge = Base64(SHA256(verifier)), sends the challenge with the auth request, then sends the original verifier when exchanging the code for tokens — the server verifies the match. Replaces: shipping a hardcoded client_secret in the APK (extractable by attackers). Why use: prevents authorization code interception attacks. Required for all mobile OAuth flows.

access token

What: a short-lived credential (typically 15 min–1 hour) sent as Authorization: Bearer <token>. JWT tokens encode claims (user ID, scopes, expiry) that can be decoded client-side. Replaces: sending the user's password on every request. Why short-lived: if leaked, the exposure window is bounded. Store in memory or EncryptedSharedPreferences — never plain SharedPreferences. Proactively refresh before expiry rather than waiting for a 401.

refresh token

What: a longer-lived credential (days to months) used to silently obtain new access tokens. Replaces: forcing re-login on every access token expiry. Why use: seamless UX. Server should rotate the refresh token on each use — a reused refresh token signals a replay attack. Store encrypted (EncryptedSharedPreferences or Keystore). If refresh fails with 401, the user re-authenticates. On force-logout, revoke the refresh token server-side.

EncryptedSharedPreferences

What: a SharedPreferences implementation from AndroidX Security that encrypts keys with AES-256-SIV and values with AES-256-GCM, backed by Android Keystore. On supported hardware, keys never leave the hardware security module. Replaces: plain SharedPreferences for sensitive data. Why use: tokens, API keys, session identifiers. Drop-in replacement for the SharedPreferences API. On devices without hardware Keystore, keys are stored in software (still encrypted but lower assurance).

certificate pinning

What: restricting which TLS certificates are trusted for your domain to specific pinned hashes, instead of trusting any CA in the OS trust store. OkHttp: CertificatePinner.Builder().add("api.example.com", "sha256/AAAA..."). Replaces: trusting any CA that can sign for your domain (vulnerable to rogue CAs). Why use: defends against MITM attacks using compromised or rogue CAs. Always pin 2–3 keys: current cert + backup cert + CA public key. Pinning only the leaf cert will brick all users on that app version when the cert rotates.

idempotency key

What: a client-generated UUID sent with state-changing requests (POST, PATCH). The server stores (idempotency_key → result); duplicate requests return the cached result without re-executing. Replaces: hoping the network doesn't drop between send and response. Why use: safe retries — no duplicate charges, bookings, or messages. Generate the UUID once per logical user action; reuse it for all network retries of that action. The server should expire idempotency keys after a reasonable window (e.g., 24 hours).

cursor pagination

What: the server returns nextCursor (opaque token encoding position); the client passes it back for the next page. Replaces: offset/page-number pagination (unstable under concurrent inserts). Why use: inserting a new row between requests doesn't shift page boundaries — no items skipped, no items duplicated. Common cursor encodings: opaque base64 of last-item ID, timestamp, or composite key. Works well with Paging 3's PagingSource<String, T> where the key is the cursor.

Android core

Activity

What: a UI screen with its own window and lifecycle: onCreate → onStart → onResume (visible + interactive) → onPause → onStop → onDestroy. Between onStop and onStart the Activity is off-screen. Replaces: nothing — fundamental Android primitive. Why it matters: always release camera, sensors, and heavy resources in onPause/onStop, not just onDestroy. Keep Activities thin — delegate logic to ViewModel, UI to Composables or ViewBinding. Each Activity is an entry point the system can independently restart.

Fragment

What: a reusable UI component hosted by an Activity. Has two lifecycles: the Fragment's own (onCreate → onDestroy) and the View lifecycle (onCreateView → onDestroyView — destroyed when pushed to the back stack). Replaces: putting all UI in one Activity. Why use: the key rule: use viewLifecycleOwner.lifecycleScope for observing LiveData or collecting flows — using this.lifecycleScope (Fragment lifecycle) collects state even when the view is gone, leaking the old view.

configuration change

What: any of — screen rotation, font scale change, locale change, multi-window resize, night mode toggle. By default, Android destroys and recreates the Activity. Replaces: nothing — OS-driven event. Why it matters: ViewModel survives (stored in ViewModelStore); Activity fields don't. onSaveInstanceState(Bundle) saves small state (selected tab, text input) that persists across recreation. Large state (fetched data, bitmaps) stays in the ViewModel.

process death

What: the OS silently kills your background process to reclaim memory. When the user returns, Android creates a fresh process and restores the back stack from the saved instance state. Replaces: nothing — OS reality. Why it matters: ViewModel is NOT recreated — a new one starts fresh. Only SavedStateHandle and onSaveInstanceState Bundle survive. Test it: go home, run adb shell am kill <package>, return to the app. "Don't keep activities" in Developer Options simulates this on every background.

launch mode

What: controls how an Activity is created and stacked. standard: new instance every time. singleTop: reuses existing instance if already on top (delivers via onNewIntent). singleTask: one instance per task; existing instance brought to front, back stack above it cleared. singleInstance: one instance globally in its own task. Why use: singleTop for notification taps on an already-open screen; singleTask for the app's home screen to prevent duplicate instances.

onNewIntent

What: called when an Activity with singleTop/singleTask launch mode receives a new Intent while already in the stack. Replaces: onCreate for the reuse case. Why use: extract new Intent extras and update UI. Always call setIntent(intent) inside onNewIntent so that subsequent getIntent() calls return the latest intent — otherwise getIntent() still returns the original launch intent.

Service

What: a component for long-running background work with no UI. Three types: Started / Foreground (user-visible, notification required), Bound (IPC, clients call methods), Background (severely restricted on Android 8+). Replaces: raw threads for persistent background work. Why use: Services run on the main thread by default — offload work to a coroutine. On Android 8+, prefer WorkManager for deferrable work; use a foreground service only for ongoing user-visible operations (audio, navigation).

BroadcastReceiver

What: receives system or app-level Intent broadcasts. Manifest-declared receivers are heavily restricted on Android 8+ — most implicit system broadcasts no longer wake them. Dynamic receivers (registered in code) still work but must be unregistered on onStop/onDestroy to avoid leaks. Replaces: polling for system events. Why use: react to ACTION_BOOT_COMPLETED, BATTERY_CHANGED. For in-app events, always prefer SharedFlow — it's lifecycle-aware and doesn't need manifest permissions.

ContentProvider

What: exposes structured data to other apps via content://authority/path URIs, over Binder IPC. ContentResolver is the client-side API. Replaces: nothing for cross-app sharing. Why use: only when genuinely sharing data with other apps (Contacts, MediaStore, FileProvider for sharing files via Intent). Inside your own app, access Room or DataStore directly — no ContentProvider needed. On Android 11+, combine with package visibility rules.

Intent

What: a messaging object describing an action and optional data. Explicit (you name the target component) or implicit (the system picks a matching handler via intent filters). Replaces: direct method calls between components (which would create tight coupling). Why use: launch Activities, start Services, send broadcasts. Security: never put sensitive data in implicit Intents — any installed app can receive them. Pass large objects via ViewModel, Room ID, or in-memory cache; put only small identifiers in extras.

PendingIntent

What: wraps an Intent and grants another process permission to fire it on your behalf at a future time. Used for notification tap actions, AlarmManager callbacks, and widget interactions. Since Android 12+, must specify FLAG_IMMUTABLE (extras cannot be modified) or FLAG_MUTABLE (needed for RemoteViews fill-in). Replaces: nothing — required for cross-process callbacks. Why immutable: prevents a malicious app from modifying the wrapped Intent's target or extras before it fires.

Doze mode

What: battery optimization activated when the device is stationary, screen off, and unplugged. During Doze, Android defers network access, wake locks, AlarmManager, and JobScheduler. Apps get periodic "maintenance windows" for deferred work. Replaces: nothing — OS-driven. Why it matters: background work can be delayed by minutes to hours. WorkManager is Doze-aware. FCM high-priority messages bypass Doze. Foreground services are not affected. Use setExactAndAllowWhileIdle() for one-time exact alarms during Doze.

App Standby

What: the OS buckets apps by usage frequency — Active → Working Set → Frequent → Rare → Restricted. Each bucket has increasingly tight limits on jobs, alarms, FCM, and network. A rarely used app may have its daily job deferred by hours. Replaces: "all apps treated equally for background." Why it matters: don't rely on precise background scheduling for user-facing features in rarely-used apps. WorkManager's constraints are bucket-aware; high-priority FCM partially bypasses buckets.

ANR (Application Not Responding)

What: the OS shows a dialog when the main thread doesn't process an event within: 5 seconds for user input, 10 seconds for BroadcastReceiver, 20 seconds for Service. Replaces: nothing — OS protection. Why it matters: ANRs appear in Play Console Vitals and tank store ratings. ANR traces log to /data/anr/traces.txt. Common causes: network/disk I/O on main thread, Thread.sleep(), deadlocks, slow onDraw(). StrictMode catches these in development.

StrictMode

What: a development tool that detects accidental violations — disk/network on main thread, leaked SQLite cursors, leaked closeable objects. Set ThreadPolicy and VmPolicy in Application.onCreate(). Penalties: penaltyLog() (logcat), penaltyDeath() (crash in debug). Replaces: hoping bugs surface in QA. Why use: catch ANR-causing main-thread I/O during development. Enable only in debug builds — never ship StrictMode in release. Pair with LeakCanary for comprehensive quality gates.

Performance

cold / warm / hot start

What: Cold — process not running; OS forks process, loads classes, initializes Application, creates Activity. Warm — process alive but Activity was destroyed; restores state without class loading. Hot — process and Activity both alive; just needs to come to the foreground. Cold is the slowest and most worth optimizing. Why it matters: TTID is measured from process fork to first frame drawn. Optimize cold start by deferring work in Application.onCreate(), using Baseline Profiles, and lazy-initializing heavy objects.

Baseline Profile

What: a text file (baseline-prof.txt) listing hot methods and classes to AOT-compile. Shipped inside the APK/AAB; Play Store uses it for cloud-side compilation before the user even installs. Replaces: relying on JIT to warm up during first use. Why use: typical improvement is 20–40% reduction in cold start. Generated by BaselineProfileRule in a Macrobenchmark test. From AGP 8+, generation integrates into the build with the Macrobenchmark Gradle plugin.

Macrobenchmark

What: an AndroidX library running stable, repeatable performance benchmarks on a real device or emulator. Measures startup time (StartupMode.COLD/WARM/HOT), scroll jank (FrameTimingMetric), custom traces (TraceSectionMetric). Results are statistically stable across iterations. Replaces: ad-hoc System.nanoTime(). Why use: run in CI to catch regressions; also the source for generating Baseline Profiles. Uses a separate test APK targeting the app via UiAutomator.

R8

What: the release build pipeline tool that shrinks (removes unused code), optimizes (inlines methods, rewrites bytecode), and obfuscates (renames classes/methods/fields). Default since AGP 3.4. Replaces: ProGuard (which R8 absorbed). Why use: 20–40% APK size reduction, faster class loading, harder to reverse-engineer. Common pitfalls: forgetting to keep serialization model class names (Gson/Moshi need them), forgetting JNI class signatures. Always test release builds with R8 before shipping.

ProGuard

What: the predecessor to R8. The proguard-rules.pro rules file format is identical and still consumed by R8. The @Keep annotation is equivalent to a -keep rule. Replaces: nothing — R8 absorbed its role. Why mention: you'll see "R8 / ProGuard" used interchangeably in interviews and documentation. The build/outputs/mapping/release/mapping.txt file maps obfuscated names back to originals for deobfuscating crash stack traces.

resource shrinking

What: shrinkResources true removes unused resource files (drawables, layouts, strings) from the final APK. Requires minifyEnabled true — R8 must first remove unused code to determine which resources are actually referenced. Replaces: manually deleting unused resources. Why use: additional APK size reduction beyond R8 code shrinking. Use tools:keep/tools:discard in res/raw/keep.xml for resources referenced dynamically via getIdentifier().

App Bundle (AAB)

What: the upload format for Google Play. The Play Store generates per-device APKs (splits) containing only the resources and code that specific device needs — screen density, CPU architecture, language. Replaces: universal APKs containing everything for all devices. Why use: download size typically drops 30–50%. Dynamic Feature Modules are also packaged in the AAB and downloaded on demand. Mandatory for new apps on Play since August 2021.

jank

What: a frame that misses its display refresh budget — 16.6 ms at 60 Hz, 11.1 ms at 90 Hz, 8.3 ms at 120 Hz. User-perceptible stutter during scroll or animation. Replaces: nothing — industry terminology. Why care: clusters of jank degrade perceived app quality even if all other metrics are fine. Target: <1% of frames janked (Play Console Vitals). Causes: slow Compose recomposition, main-thread work during draw, bitmap decoding on main thread. Use adb shell dumpsys gfxinfo <package> for per-frame data.

JankStats

What: JankStats.createAndTrack(window, listener) records per-frame timing and calls back for janky frames. Attach UI state context: jankStats.perFrameMetrics["screen"] = screenName — reports include which screen janked. Replaces: hand-rolled Choreographer instrumentation. Why use: collect real user jank metrics in production and upload to your analytics backend. Part of AndroidX metrics-performance.

LeakCanary

What: a debug-only library that automatically detects memory leaks by watching objects that should be GC'd (Activities, Fragments, ViewModels, Views after their lifecycle ends). Uses weak references + ObjectWatcher; triggers a heap dump if watched objects are still reachable after GC. Replaces: hand-capturing heap dumps. Why use: just add as debugImplementation — leaks surface as notifications during development. The heap dump is analyzed in-process and shows the shortest GC root chain. Never ship in release builds.

heap dump

What: a snapshot of all live objects on the JVM heap at a point in time. Captured via Android Studio Profiler, LeakCanary, or Debug.dumpHprofData(). Replaces: guessing what's leaking. Why use: analyze in Android Studio's heap dump viewer — group by class, look for >1 instance of an Activity (should be 1), follow the GC root chain to the retaining reference. Common findings: static fields holding Context, uncancelled Flows, unclosed Cursors/database connections.

TTID / TTFD

What: TTID (Time to Initial Display) — system-measured time from process start to the first frame drawn. TTFD (Time to Full Display) — user-declared time when all meaningful content is visible; call activity.reportFullyDrawn() (or ReportDrawnWhen/ReportDrawnAfter in Compose). Replaces: vague "startup time." Why distinguish: a splash screen has fast TTID but slow TTFD. Both are logged to logcat and tracked in Play Console. Baseline Profiles reduce TTID; lazy/async data loading reduces TTFD.

overdraw

What: a pixel rendered more than once per frame — wasted GPU work. Visualized in Developer Options → "Show overdraw areas" with colors from white (no overdraw) to dark red (4x+). Replaces: nothing — a rendering phenomenon. Why care: caused by stacked backgrounds in XML view hierarchies (every layer has its own background). Fixes: set android:background="@null" on intermediate Views whose parent already paints the background; flatten the hierarchy with ConstraintLayout. Compose is generally smarter about this but can still overdraw with misplaced background modifiers.

Player & streaming

ExoPlayer

What: Google's open-source customizable Android media player, now part of Media3. Supports HLS, DASH, SmoothStreaming, progressive MP4/WebM/Ogg, and DRM (Widevine, ClearKey, PlayReady). Fully customizable: pluggable TrackSelector, LoadControl (buffer sizes), MediaSource (custom data). Replaces: legacy MediaPlayer for serious streaming. Why use: consistent behavior across OS versions; extensibility for ABR, ad insertion, and analytics. Lifecycle: create in onStart, release in onStop to free codec/DRM resources.

Media3

What: the AndroidX umbrella unifying ExoPlayer + MediaSession + Cast + UI into a single coherent API with consistent versioning. Migrate by changing imports from com.google.android.exoplayer2 to androidx.media3. Replaces: standalone ExoPlayer artifact for all new code. Why use: all future development happens in Media3; MediaSessionService enables background playback; MediaController connects UI to the player session without holding a direct player reference.

HLS (HTTP Live Streaming)

What: Apple's adaptive streaming format. A master .m3u8 manifest references rendition playlists (different bitrates/resolutions); each playlist lists segment files (.ts or .fmp4). ExoPlayer parses HLS natively. Replaces: progressive download / single-bitrate streams. Why use: native iOS support — the only format that works on iOS without a player library. Segment duration (2–10 s) trades off ABR agility vs server request rate. Supports AES-128 encryption and Widevine DRM via EXT-X-KEY.

DASH

What: MPEG's open adaptive streaming standard. An .mpd XML manifest describes representations (bitrate/codec variants), periods, and adaptation sets. Segments are .mp4 fragments. SegmentTemplate with $Number$ enables efficient manifests without listing every URL. Replaces: proprietary formats. Why use: open standard; richer manifest model than HLS; supports multiple DRM systems in one manifest. Limitation: no native iOS support — use HLS or CMAF for cross-platform delivery.

CMAF

What: Common Media Application Format — a packaging convention for fragmented .mp4 segments compatible with both HLS and DASH manifests. Replaces: packaging the same content twice (.ts for HLS, .mp4 for DASH). Why use: one set of segments on the CDN, served via both HLS .m3u8 and DASH .mpd manifests pointing to the same files — roughly halves storage and CDN cost. Requires HLS version 7+ and fmp4 segments (not .ts).

ABR (Adaptive Bitrate)

What: the player monitors bandwidth and buffer health, then selects bitrate among the manifest's available variants. ExoPlayer uses AdaptiveTrackSelection by default. Switches down proactively when bandwidth drops; switches up conservatively when buffer is healthy. Replaces: fixed-quality streams that stall on slow connections. Why use: optimal quality on every connection. Key tuning params: minDurationForQualityIncreaseMs (buffer threshold before upgrading), maxInitialBitrate (cap for first segment on slow networks). Rebuffer ratio is the primary ABR health signal.

Widevine

What: Google's DRM system, pre-installed on most Android devices. Three security levels: L1 — hardware-backed TEE, content keys never leave secure hardware (required for HD on most platforms); L2 — intermediate; L3 — software-only, keys in JVM heap (extractable, typically limited to SD). Check device level: MediaDrm(WIDEVINE_UUID).getPropertyString("securityLevel"). Replaces: nothing — required by content owners for premium content. ExoPlayer handles key fetching via DefaultDrmSessionManager.

DRM

What: encrypts media segments; a license server issues decryption keys after verifying the user's entitlement. Flow: player fetches manifest → sees DRM info → sends license request (signed with device key) to license server → server verifies subscription → returns encrypted key → player decrypts and plays. Replaces: unencrypted streams. Why use: required for premium content licensing. Common systems: Widevine (Android), FairPlay (Apple), PlayReady (Microsoft). Key rotation per period limits exposure if a key is compromised. Pre-fetch the license to avoid startup latency.

CSAI

What: Client-Side Ad Insertion. The player pauses content, plays the ad stream separately (via IMA SDK), then resumes. Ad URL is resolved at playback time — supports per-user targeting. Replaces: "no ads." Why use: simpler to implement; IMA SDK handles VAST parsing, quartile beacons, and skip detection. Downside: ad blockers can intercept the ad URL. ImaAdsLoader integrates IMA with ExoPlayer's AdsMediaSource. Content player and ad player are separate ExoPlayer instances; sync their lifecycles carefully.

SSAI

What: Server-Side Ad Insertion. Ads are stitched into the content stream server-side — a single continuous HLS/DASH playlist, no player-side break. Replaces: CSAI when you need ad-block resistance and seamless playback. Why use: harder to block, no buffering pause at break. Challenges: per-user manifests (each user gets a unique stream URL for targeting), DRM key synchronization across content and ad segments, and accurate quartile beacon timing from the server-stitched timeline. Used by AWS MediaTailor, Google DAI.

quartile beacon

What: HTTP GET requests fired at standardized ad lifecycle milestones: impression (started), firstQuartile (25%), midpoint (50%), thirdQuartile (75%), complete (100%). URLs come from VAST <Tracking> elements. Replaces: nothing — ad industry billing standard. Why care: advertisers pay based on these signals; if beacons don't fire, revenue is lost. Implement retry logic for transient network errors; never double-fire. IMA SDK and ExoPlayer's AdsLoader handle this automatically.

MediaSession

What: exposes playback state and controls to system consumers — lock screen, notification shade, Android Auto, Google Assistant, Bluetooth headsets, Cast. The new media3.session.MediaSession wraps ExoPlayer. MediaController (in the UI layer) connects to the session and controls playback without holding a direct player reference. Replaces: hand-built lock screen controls. Why use: single source of truth for playback state; system UI controls work automatically; enables background playback via MediaSessionService.

TTFF (Time to First Frame)

What: the latency from user pressing "play" to the first decoded video frame rendered. The primary start-of-playback quality metric tracked in production via Conviva, Mux, or Bitmovin. Replaces: subjective "is the player fast." Why care: users abandon playback if TTFF > ~3 seconds. Optimization: pre-load the manifest during browse, warm DRM sessions (pre-fetch the license), call setMediaItem() + prepare() early in the user flow before play is tapped.

rebuffer ratio

What: total_rebuffering_time / total_watch_time. The primary ongoing streaming quality KPI. Targets: <0.5% excellent, <1% good, >3% problem, >5% critical. Replaces: anecdotal "users complain about buffering." Why use: tracked per session and aggregated by CDN, ISP, device, and bitrate. When investigating a spike: cross-reference with bandwidth measurements, CDN cache-hit rates, ABR starting bitrate, and server error rates to isolate the root cause.

Build & tools

Gradle

What: Android's build system. Scripts are build.gradle.kts (Kotlin DSL, recommended) or build.gradle (Groovy DSL). A build is a DAG of tasks; incremental builds only re-run tasks whose inputs changed. Replaces: Ant + Maven for Android. Why use: the only officially supported system. Optimization levers: build cache (task output reuse), configuration cache (skip config phase on re-run), parallel module compilation. Profile slow builds with ./gradlew --profile or Gradle Build Scan.

KSP (Kotlin Symbol Processing)

What: a Kotlin-first annotation processing API that operates directly on Kotlin code without generating Java stubs. ~2x faster than kapt. Replaces: kapt. Why use: understands Kotlin-specific constructs (data classes, suspend functions, extension functions) that kapt doesn't. Used by Room 2.4+, Hilt 2.51+, Moshi 1.14+, and kotlinx.serialization. Switch: replace kapt("...") with ksp("...") in your build files.

kapt

What: Kotlin Annotation Processing Tool — bridges Kotlin code to Java annotation processors by generating Java stub files first, then running the processor. The stub generation step is slow and doesn't understand Kotlin-specific features (default args, nullability). Replaces: nothing — historical. Why mention: deprecated in favor of KSP; libraries have been migrating since 2022. Old projects still use it; the two cannot be mixed for the same processor. Check library migration status before replacing.

build variant

What: the Cartesian product of build types (debug/release) × product flavors (free/paid, staging/production). Each combination is a variant with its own APK and source set. Variant-specific sources in src/stagingDebug/ override defaults. Replaces: separate code branches per environment. Why use: one codebase delivers different API base URLs, signing configs, feature flags, analytics keys, and app IDs — multiple variants can even be installed simultaneously when they have distinct app IDs.

BuildConfig

What: a class generated at build time containing constants from the build configuration. BuildConfig.DEBUG, VERSION_NAME, VERSION_CODE. Add custom fields: buildConfigField "String", "BASE_URL", '"https://staging.example.com"'. Replaces: manually defining debug flags. Why use: conditional code paths without polluting source. Note: since AGP 8.0, BuildConfig generation is opt-in — add buildFeatures { buildConfig = true } in your module's android block, or it won't generate.

AndroidManifest

What: the XML file declaring app identity, all four component types (Activity, Service, BroadcastReceiver, ContentProvider), required permissions, and intent filters. Multiple manifests (app + libraries) are merged at build time. Replaces: nothing — fundamental OS contract. Why care: the OS reads this on install; a missing permission causes SecurityException at runtime; a wrong intent filter prevents your app from handling a deep link. Use tools:remove/tools:replace to control manifest merge behavior.

Crashlytics

What: Firebase's real-time crash reporting. Captures full stack traces, non-fatal exceptions (recordException), and custom key-value context (setCustomKey("screen", "Feed")). Groups crashes by signature, showing affected user count and version breakdown. Replaces: hand-rolled crash reporting. Why use: de facto standard; integrates with Play Console Android Vitals; supports coroutine context preservation so stack traces show the originating coroutine, not just the internal coroutine machinery. Set non-PII user identifiers (hashed IDs) to track affected user counts.

Misc & data structures

HashMap

What: an unordered map backed by a hash table. O(1) average for get/put/remove. Replaces: List<Pair> for key-based lookups. Why use: when you need O(1) value lookup by key. Gotchas: not thread-safe — use ConcurrentHashMap or protect with a Mutex in coroutines. Keys must have stable, consistent hashCode/ equals — mutating a key after insertion corrupts the map. Worst-case O(n) with many hash collisions (Java 8+ treeifies long chains to O(log n)).

HashSet

What: a hash table set — O(1) average add, contains, remove. Backed internally by a HashMap<T, PRESENT>. Replaces: List.contains() which is O(n). Why use: deduplication, membership tests, and set operations (union via addAll, intersection via retainAll). Use LinkedHashSet when insertion order matters. On Android, consider ArraySet from android.util for small sets (<100 elements) — uses a sorted array, more memory-efficient at small sizes.

LinkedHashMap

What: a HashMap with a doubly-linked list maintaining insertion order (or access order when constructed with LinkedHashMap(capacity, loadFactor, true)). O(1) operations like HashMap. Replaces: hand-tracking order alongside a HashMap. Why use: ordered iteration, history/MRU lists, and LRU caches. Access-order mode: every get() moves the entry to the tail. Override removeEldestEntry { size > maxSize } for simple LRU eviction. This is exactly how Android's LruCache works internally.

LruCache

What: a thread-safe, size-bounded cache backed by LinkedHashMap in access-order mode. Evicts the least recently used entry when the size limit is exceeded. Override sizeOf(key, value) to measure size in bytes (for bitmaps). Override create(key) to auto-populate on cache miss. Replaces: unbounded HashMap that eventually causes OOM. Why use: in-memory bitmap or object caching. Size guideline: Runtime.getRuntime().maxMemory() / 8 gives 1/8th of the app's memory budget.

SparseArray

What: an Android-specific Map<Int, T> that avoids autoboxing of integer keys. Stores keys in a sorted int[] and values in a parallel Object[]. Binary search for lookups — O(log n). Replaces: HashMap<Int, T> for small, int-keyed maps. Why use: avoids boxing Int → Integer for every key — saves memory and GC pressure for maps keyed by resource IDs, view IDs, or small integer sequences. Also: SparseIntArray, SparseLongArray, SparseBooleanArray for primitive values too.

Bundle

What: an Android Map<String, Any?> restricted to Bundle-serializable types — primitives, String, Parcelable, Serializable, arrays, and ArrayLists of those types. Used for: onSaveInstanceState, Fragment arguments, Intent extras, IPC. Replaces: static fields (which don't survive process death). Why use: the only mechanism that survives the OS killing your process. Hard limit: ~500 KB total (Binder transaction limit). TransactionTooLargeException crashes the app silently in some cases. Pass only IDs; keep large objects in Room or ViewModel.

Parcelable

What: Android's binary serialization format optimized for IPC and Bundle transfer. The @Parcelize Kotlin annotation (via kotlin-parcelize plugin) generates the writeToParcel / CREATOR boilerplate automatically. Replaces: Java Serializable. Why use: much faster than Serializable (no reflection); smaller binary representation; required for IPC across process boundaries. Don't use for disk persistence — the binary format can change between Android versions. For cross-process stable storage, use Room, DataStore, or proto.

AAR vs JAR

What: an AAR (Android Archive) bundles compiled code (.jar), Android resources (res/), assets, AndroidManifest.xml, JNI libraries, ProGuard rules, and Lint rules. A JAR contains only compiled bytecode. Replaces: nothing — different output formats. Why use: AAR for any library touching Android types (resources, Context, views); JAR for pure Kotlin/Java logic that should be reusable from non-Android JVM projects or Kotlin Multiplatform. If your library has zero Android imports, publish as JAR for maximum portability.
