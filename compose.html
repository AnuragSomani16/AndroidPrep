Jetpack Compose — internals, side effects, performance

From recomposition mechanics to stability annotations to side effect APIs. Plus Compose vs XML, theming, navigation, and pagination.

1. Why Compose over XML — the senior answer

Locked-in answer

Compose is a declarative UI toolkit: you describe what UI should look like for a given state, and the runtime diffs and updates the underlying nodes. Compared to XML + View hierarchy:

  • State-driven — UI follows state automatically; no findViewById, no manual setText. Eliminates a whole class of "stale view" bugs.
  • Composable functions are testable, previewable, reusable as plain Kotlin functions — not XML resources.
  • Less boilerplate — RecyclerView's adapter / ViewHolder dance becomes a few lines with LazyColumn.
  • Animations are first-class via animate*AsState, updateTransition; no AnimatorSet XML files.
  • Type-safe theming via Material 3 + CompositionLocal — versus AppCompat-themes-as-strings.
  • Interop — drop ComposeView in XML or AndroidView in Compose; you can migrate incrementally.

Caveats to mention — initial learning curve for stability/recomposition; tooling for very complex custom layouts is still maturing; large lists need explicit keys to perform.

2. The mental model — composition, recomposition, slot table

Composition = the tree of @Composable calls that describes the UI for a given state. Recomposition = re-running parts of that tree when state changes. Slot table = the runtime data structure storing the position of each composable so Compose knows where to put the new output (and what remember values belong to that position).

Three pillars:

  1. Composition — the tree of @Composable calls that produces the UI description.
  2. Recomposition — re-running parts of the composition when state changes.
  3. Slot table — the runtime data structure that tracks where each composable lives, enabling positional memoization across re-runs.
Senior phrase

"Compose uses positional memoization in the slot table — remember values are keyed by their call-site location in the composition tree, not by name. That's why two remember { 0 } calls in the same composable body produce two independent values."

3. Recomposition rules

  1. A composable recomposes when its inputs change OR it reads a State whose value changed.
  2. A composable is skipped if all parameters are stable AND none have changed.
  3. If a parameter changes, only that composable (and any descendant whose own inputs change) re-runs.

4. Stability — the perf gotcha

A type is stable if Compose can trust equals to mean "rendered output should be the same." Stable types let Compose skip recomposition.

TypeStable?
Primitives, StringYes
data class with all-stable val fieldsYes
data class with var fieldsNo
Lambda references (function references)Yes
List<T>, Map<K,V>, Set<T>No (interfaces — implementation could be mutable)
Types annotated @Immutable or @StableYes (you take responsibility)
kotlinx.collections.immutable.ImmutableList<T>Yes
MutableState, StateYes
 
// UNSTABLE — List interface; Compose can't skip
data class Filter(val tags: List<String>)

// STABLE — annotation tells Compose "trust me"
@Immutable
data class Filter(val tags: List<String>)

// Or use ImmutableList for true immutability
data class Filter(val tags: ImmutableList<String>)
Senior diagnostic phrase

"I run the Compose Compiler reports in CI — they output a stability-and-skippability report per composable. When a screen is sluggish, that report is my first stop. Then Layout Inspector for recomposition counts."

LaunchedEffect — coroutine on enter / key change

@Composable fun ScreenA(id: String, vm: MyVm = hiltViewModel()) {
    LaunchedEffect(id) {        // restarts when id changes
        vm.load(id)
    }
}

DisposableEffect — resource cleanup

@Composable fun PlayerScreen(player: ExoPlayer) {
    DisposableEffect(player) {
        val listener = object : Player.Listener { /* ... */ }
        player.addListener(listener)
        onDispose { player.removeListener(listener) }
    }
}

derivedStateOf — gate recomposition

@Composable fun Feed(listState: LazyListState) {
    val showFab by remember {
        derivedStateOf { listState.firstVisibleItemIndex == 0 }   // recomposes downstream only when boolean flips
    }
    if (showFab) ScrollFab()
}
Why this isn't just val showFab = listState.firstVisibleItemIndex == 0: firstVisibleItemIndex changes every time the user scrolls (every pixel). Without derivedStateOf, the surrounding composable would recompose hundreds of times per second. With it, Compose tracks the derived boolean and only triggers downstream recomposition when it actually flips (true→false or false→true). Massive performance win on scroll-driven booleans.

rememberUpdatedState — stale closure fix

@Composable fun Timer(onFinish: () -> Unit) {
    val current by rememberUpdatedState(onFinish)
    LaunchedEffect(Unit) {
        delay(5000)
        current()                // always latest, even if parent re-passed onFinish
    }
}
The bug this prevents: LaunchedEffect(Unit) only runs once when the composable enters the composition. Inside that coroutine, the onFinish lambda is captured by closure. If the parent passes a new onFinish on a later recomposition (e.g., a fresh lambda each time), the LaunchedEffect's coroutine is still using the old one. rememberUpdatedState stores the lambda in a State<T> wrapper that's updated on every composition. Reading current inside the coroutine always sees the latest version.

snapshotFlow — Compose State → Flow

LaunchedEffect(listState) {
    snapshotFlow { listState.firstVisibleItemIndex }
        .filter { it > 10 }
        .distinctUntilChanged()
        .collect { analytics.scrolledPast(it) }
}

7. State hoisting — the pattern

Push state up to the lowest common parent. Pass state + onEvent down. Stateless composables → easier preview, test, reuse.

// Stateless (preferred for reuse)
@Composable fun Counter(count: Int, onInc: () -> Unit, modifier: Modifier = Modifier) {
    Row(modifier) {
        Text("$count")
        Button(onClick = onInc) { Text("+") }
    }
}

// Stateful wrapper at the leaf level
@Composable fun StatefulCounter() {
    var count by remember { mutableIntStateOf(0) }
    Counter(count, { count++ })
}

8. LazyColumn / LazyRow — the hot path

LazyColumn {
    items(notes, key = { it.id }, contentType = { "note" }) { note ->
        NoteRow(note)
    }
}
Always provide key

Without keys, Compose treats item N at position 5 as "the same" across reorderings — leading to wrong recomposition, scroll jumps, animations targeting the wrong items. Provide key = { it.id } always. contentType helps view-pool reuse for heterogeneous lists.

9. Modifiers — order matters

Box(Modifier.padding(16.dp).background(Color.Red))   // padding around red bg
Box(Modifier.background(Color.Red).padding(16.dp))   // red bg, padding inside

Read modifiers left-to-right, outside-to-inside. The leftmost modifier wraps everything to its right.

// Type-safe (Nav 2.8+)
@Serializable data object HomeRoute
@Serializable data class NoteDetailRoute(val id: String)

NavHost(navController, startDestination = HomeRoute) {
    composable<HomeRoute> { HomeScreen(onOpen = { id -> navController.navigate(NoteDetailRoute(id)) }) }
    composable<NoteDetailRoute> { entry ->
        val args = entry.toRoute<NoteDetailRoute>()
        NoteDetailScreen(args.id)
    }

}

Nav features

  • Deep links: navDeepLink { uriPattern = "myapp://note/{id}" } in the composable.
  • Single top: navOptions { launchSingleTop = true }.
  • Pop up to: popUpTo("home") { inclusive = false }.
  • Result back: previous entry's SavedStateHandle for returning data.

13. Pagination in Compose — Paging 3

// In ViewModel
val pagedItems: Flow<PagingData<Note>> = Pager(
    config = PagingConfig(pageSize = 20, prefetchDistance = 5),
    pagingSourceFactory = { NotesPagingSource(api) }
).flow.cachedIn(viewModelScope)

// In Compose
@Composable fun NotesList(vm: NotesVm = hiltViewModel()) {
    val items = vm.pagedItems.collectAsLazyPagingItems()
    LazyColumn {
        items(count = items.itemCount, key = items.itemKey { it.id }) { i ->
            val note = items[i] ?: return@items   // null while loading
            NoteRow(note)
        }
        when (val s = items.loadState.append) {
            is LoadState.Loading -> item { Spinner() }
            is LoadState.Error   -> item { ErrorRow(s.error) { items.retry() } }
            else -> Unit
        }
    }
}

14. Compose performance — the toolkit

  • Layout Inspector → enable "Show recomposition counts" — anything > 1 per scroll is suspect.
  • Compose Compiler reports → per-function stability/skippability.
  • Macrobenchmark + Baseline Profiles → cold start + first-frame.
  • Modifier.composed → use sparingly, allocates per composition.
  • Avoid creating lambdas in body when used as parameters to stable composables — extract with remember { { ... } } if needed.
  • rememberSaveable only for things you'd be sad to lose; everything else remember.
  • For heavy custom drawing: Modifier.drawWithCache { onDrawWithContent { } }.

15. Compose vs XML — a practical compare

XML + View binding

// activity_main.xml — 30 lines
class MainActivity : AppCompatActivity() {
    private lateinit var binding: ActivityMainBinding
    override fun onCreate(s: Bundle?) {
        super.onCreate(s)
        binding = ActivityMainBinding.inflate(layoutInflater)
        setContentView(binding.root)
        binding.tvName.text = "Hello"
        binding.btnGo.setOnClickListener {
            binding.tvName.text = "Clicked"
        }
    }
}

Compose

class MainActivity : ComponentActivity() {
    override fun onCreate(s: Bundle?) {
        super.onCreate(s)
        setContent {
            var name by remember { mutableStateOf("Hello") }
            Column {
                Text(name)
                Button({ name = "Clicked" }) { Text("Go") }
            }
        }
    }
}

When XML is still right

  • Massive existing codebase mid-migration.
  • Specific custom View with hardware-accelerated paths that matter.
  • RecyclerView with extreme custom ViewHolder reuse for performance edge cases (rare in 2024+).

17. Common bugs & fixes

Q. My composable recomposes every time anything changes — why?

Likely an unstable parameter — a List, a Map, or a data class with a var field. Run Compose Compiler reports; mark the type @Immutable or use ImmutableList. Also check: are you passing a fresh lambda each composition? Either ignore (function refs are stable) or extract with remember.

Q. My LaunchedEffect runs forever / not at all.

Forever: probably a key that changes on every recomposition (e.g., new lambda each time). Use Unit or stable keys. Not at all: key never changes after first comp. Use key = Unit for "run once on enter."

Q. My snackbar fires twice.

You're collecting a Flow inside a LaunchedEffect that restarts on recomposition. Fix: collect with LaunchedEffect(Unit) and use a SharedFlow with replay = 0 for events.

Q. My LazyColumn jumps to top when items update.

Missing key. Without keys, Compose can't identify items across updates and treats them as new at the same position.

Q. State change in StateFlow doesn't trigger Compose.

Are you using collectAsState or collectAsStateWithLifecycle? Bare flow.collect in a LaunchedEffect won't subscribe Compose to the value. Also: is StateFlow actually emitting? It dedupes by equality.

18. Senior signal phrases for Compose

  • "Recomposition skipping requires stability — without it, every parent change cascades."
  • "derivedStateOf to gate recomposition when reading scroll position."
  • "I lift state, leave leaves stateless — easier to preview and test."
  • "collectAsStateWithLifecycle, never bare collectAsState."
  • "CompositionLocal for cross-cutting concerns; props for everything else."
  • "@Immutable on stable data models is part of our linting now."
